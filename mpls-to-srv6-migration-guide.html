Why Migrate to SRv6?

A clear-eyed look at what SRv6 gives you, what carries over from SR-MPLS, and what it costs — for engineers who have real networks to run.

Business Drivers

The industry push toward SRv6 is not just vendor marketing. There are genuine operational and architectural motivations — though they matter differently depending on your network scale and services mix.

IPv6-Native
No more service-layer dual-stack. SRv6 SIDs are IPv6 addresses. The outer transport header is IPv6. One address family to manage end-to-end.
One Data Plane
Eliminate the MPLS forwarding plane from your core. Transit routers need only IPv6 LPM. No MPLS-capable ASICs required in the core (with uSID).
Network Programming
SID functions encode rich behaviors: End.DT4 (VRF IPv4 decap), End.DX4 (per-CE), End.DT2M (EVPN BUM). Behaviors composable in the SRH.
Unified WAN + DC
SRv6 spans SP core and data center fabrics using the same SID space and control plane. Single policy framework across domains.
Larger SID Space
128-bit SIDs vs 20-bit labels. No SRGB sizing concerns. With /64 locators you have 264 possible functions per node — VPN label exhaustion is gone.
SR-Policy Evolution
SR-Policy (color, ODN, BSID, Flex-Algo) carries over almost unchanged. Your TE investment is not discarded — just re-expressed in SRv6 SIDs.
Bottom line for SP networks: The most compelling driver is eliminating MPLS from the core forwarding plane. Edges (PEs) still do SRv6 encap/decap, but transit (P) routers only need IPv6 forwarding. This simplifies your hardware refresh roadmap significantly.

What SRv6 Adds Over SR-MPLS

Side-by-side architectural comparison. The key shift: SRv6 SIDs encode both topology location (the locator, routable via IPv6) and network function (the function field, executed locally).

SR-MPLS

  • 20-bit label space (e.g., SRGB 16000–23999)
  • Requires MPLS-capable forwarding on every hop
  • Label per-hop swap (ILM/NHLFE tables)
  • Adj-SIDs are local — need Node-SID + Adj-SID (2 labels minimum)
  • VPN labels from dynamic pool (risk of exhaustion at scale)
  • No inline behavior encoding — VPN label just means "look up VRF"
  • ECMP via load-balance on label + IP header fields

SRv6

  • 128-bit SID space — effectively unlimited
  • Native IPv6 forwarding on all transit nodes
  • SID encodes locator (where to route) + function (what to do)
  • Adj-SIDs (uA) are globally routable within locator prefix
  • VPN SIDs per-VRF per-function — no pool exhaustion
  • Rich function encoding: End.DT4, End.DT6, End.DX4, End.DT2M, etc.
  • ECMP via IPv6 flow label + standard hashing
The programming model: In SR-MPLS, a packet with label stack [16001, 16004, 24001] means "go to PE1, go to PE2, deliver to VPN 24001." In SRv6, a packet with DA=2001:db8:cafe:4:dt4:: and no SRH means "IPv6 routing delivers to PE2, PE2 executes End.DT4 (decap + VRF lookup)." The behavior is encoded in the SID itself.

SRv6 Readiness Assessment

Before touching production, systematically validate hardware, software, control plane, and operational readiness. A failed migration is worse than staying on SR-MPLS.

IPv4 to IPv6 Underlay Migration

SRv6 needs IPv6 reachability between all PEs. If your core runs IPv4-only today, this is the first and most disruptive step — plan it carefully.

The Underlay Problem

SRv6 encapsulation uses IPv6 as the outer header. The source address is the ingress PE's IPv6 loopback. The destination address is the SRv6 SID at the egress PE. For this to work, every router in the path must have IPv6 forwarding enabled and reachability to the egress PE's locator prefix.

The hard truth: Most SP networks running SR-MPLS today use an IPv4-only IGP underlay. P-routers may have no IPv6 configuration at all. Adding IPv6 to every core link and loopback, enabling IPv6 in ISIS, and verifying end-to-end IPv6 reachability is a multi-week project on a large network — and it happens before you touch a single SRv6 feature.

Three Options for IPv6 Underlay Deployment

Option A: Dual-Stack Underlay

Add IPv6 to every link alongside existing IPv4. Both address families run in ISIS simultaneously (MT-ISIS).

  • Safest approach — IPv4/MPLS services unaffected during rollout
  • Independent IPv4 and IPv6 SPF (MT-ISIS)
  • Can roll back IPv6 per-link without service impact
  • Increases IGP topology size (2× prefixes)
  • Most common migration path in production networks
Recommended for most networks

Option B: IPv6-Only Core

Remove IPv4 from core links entirely. Only IPv6 runs in the core. PE-CE links remain IPv4.

  • Simplest long-term steady state
  • High risk — requires all services to work over IPv6 before cutover
  • Hard to roll back once IPv4 is removed
  • Requires IPv4-over-IPv6 or 6PE for IPv4 VPN transport
  • Appropriate only for greenfield or complete forklift migrations
Not recommended for brownfield migrations

Option C: IPv6 Islands (Phased Regional)

Upgrade network regions to SRv6 incrementally. Use SR-MPLS/SRv6 stitching at domain boundaries (GW node).

  • Migrate one region or PoP at a time
  • Stitching node (GW) bridges SRv6 and SR-MPLS domains
  • Requires SR-MPLS to SRv6 stitching support at boundary
  • Good for large networks that cannot do big-bang migration
  • Topology: CE1 — PE1(SRv6) — P1(SRv6) — GW(stitching) — P2(MPLS) — PE2(MPLS)
Best for phased large-scale rollout

Stitching at GW Node

The GW node (10.0.0.5 / 2001:db8::5) performs SRv6-to-MPLS handoff:

  • Inbound: SRv6 SID 2001:db8:cafe:5:: — decap SRv6, push MPLS label for onward MPLS segment
  • Outbound: Receive MPLS packet, swap/pop label, encap in SRv6 toward PE2's SRv6 locator
  • GW must support both SRv6 and SR-MPLS forwarding simultaneously
  • Requires BGP re-advertisement of VPN routes across the boundary with SID/label translation

IGP Considerations — MT-ISIS vs Single Topology

Single Topology ISIS

  • IPv4 and IPv6 share the same SPF
  • Simpler configuration — one topology, one set of metrics
  • Coupled failure: IPv6 route flap can affect IPv4 SPF and vice versa
  • An IPv6-only router in the topology breaks IPv4 SPF (it must carry both)
  • Not suitable for phased migration where some routers are IPv6-only
Risk during phased deployment

Multi-Topology ISIS (MT-ISIS)

  • IPv4 (MT-0) and IPv6 (MT-2) run independent SPFs
  • IPv6 failure does not affect IPv4 forwarding (and vice versa)
  • Routers can participate in IPv4 topology only, IPv6 only, or both
  • Slightly larger LSPs (two topology-specific TLVs per prefix)
  • Enables incremental IPv6 deployment without IPv4 disruption
Recommended for migration
Migration recommendation: Start with MT-ISIS. After full IPv6 deployment and several months of stable operation, you can optionally migrate to single-topology if your network design calls for it. The operational risk of single-topology during migration far outweighs the configuration simplicity benefit.

OSPFv3 Note

OSPFv3 runs natively over IPv6 but is rarely used as the primary IGP in SP core networks. If your network uses OSPF today for the IPv4 underlay, you would need either OSPFv3 (for IPv6) alongside OSPFv2 (for IPv4), or migrate the entire IGP to ISIS which handles both natively. For networks already on ISIS, the path is straightforward.

At this point the SRv6 underlay is ready. Services can now be migrated to SRv6 one by one.

SID design is the most consequential architectural decision in an SRv6 deployment. Get this right before configuration begins — changing SID structure mid-deployment requires re-advertisement of all routes.

Locator advertisement: Unlike SR-MPLS Prefix-SIDs (which are sub-TLVs attached to an existing prefix advertisement), SRv6 locators get their own TLV (ISIS SRv6 Locator TLV-27). Design your locator block to accommodate this: e.g., reserve 2001:db8:cafe:XX00::/56 per node, with :XX01::/64 for algo-0 and :XX02::/64 for algo-128.

uSID vs Full-Length SIDs

This is the single most important design decision in your SRv6 deployment. uSID (micro-SID) packs multiple 16-bit node/function identifiers into a single 128-bit SID container. This dramatically reduces encapsulation overhead and eliminates the SRH in most cases.

Full-Length SIDs (RFC 8986 baseline)

  • Each SID is a full 128-bit IPv6 address
  • 3-segment path needs 3 × 16-byte SRH entries + SRH fixed header
  • Overhead for 3-SID path (H.Encaps, non-reduced): 40 (IPv6) + 8 (SRH) + 3×16 = 96 bytes
  • All transit nodes must process SRH (Segments Left decrement + DA update)
  • Requires SRH-capable ASICs on all intermediate routers
  • SID space: 128 bits — vast, but one per SRH slot
Not recommended for new deployments — use uSID instead

ASIC Forwarding Considerations

SRv6 forwarding places different demands on the data plane ASIC compared to MPLS label switching. Before migrating, verify your hardware handles SRv6 at line rate.

What SRv6 Requires from the ASIC

  • IPv6 LPM lookup: SRv6 forwarding is done via IPv6 longest-prefix-match on the locator prefix. Older ASICs may have smaller IPv6 LPM tables than IPv4 tables.
  • SRH parsing: The ASIC must parse the IPv6 extension header chain to find and process the SRH — specifically reading Segments Left and the segment list.
  • SID decrement & copy: At each waypoint, the ASIC decrements SL and copies the next SID into the IPv6 DA field. This is a read-modify-write operation on the packet header.
  • uSID shift-and-forward: Micro-SID processing requires the ASIC to bit-shift the DA field left by 16 bits on each hop — a specific operation not all ASICs support natively.

Modern ASICs (Capable)

  • Native SRH parsing at line rate
  • Hardware uSID shift-and-forward
  • No recirculation needed
  • Full 100/400Gbps throughput maintained

Older ASICs (Risk)

  • SRH parsing may require recirculation
  • Each recirculation pass halves effective throughput
  • uSID may not be supported in hardware
  • Software fallback = severe performance degradation
Action before migration: Run a hardware capability check. Send SRv6-encapsulated traffic at line rate in a lab environment and measure throughput, latency, and CPU utilization. If you see CPU spikes or throughput degradation, your ASIC is recirculating. Do not migrate production traffic to SRv6 on that hardware without a platform upgrade.

Rollback Planning

Ensuring MPLS remains your safety net throughout the SRv6 migration

Why You Need a Rollback Plan

No migration to a new data plane technology goes perfectly. You will encounter at least one of the following: a feature gap you did not anticipate, a vendor bug in SRv6 implementation, an MTU issue that was missed in lab testing, or a performance regression on a specific hardware platform.

The cardinal rule: MPLS must remain fully operational throughout the entire migration until you explicitly and deliberately decommission it. Every step of the migration must be reversible. If you cannot roll back in under 30 minutes, your migration plan is incomplete.

What Can Go Wrong

  • Feature gap: SRv6 equivalent of an MPLS feature (e.g., RSVP-TE bypass, specific QoS marking) not yet implemented
  • Hardware bug: SRH parsing issue on specific ASIC revision causing drops
  • MTU silent drops: Specific traffic flows exceeding MTU not caught in testing
  • BGP convergence regression: SRv6 route reflector signaling slower than MPLS path
  • OAM gap: Existing monitoring tools cannot process SRv6 packets — loss of observability
Rollback is not failure — it is professional engineering. Build it in from day one.

Rollback Architecture

During the coexistence phase, both MPLS and SRv6 control planes and data planes are active simultaneously. Rollback means steering all traffic back to MPLS paths without removing any SRv6 configuration.

Three Rollback Mechanisms

1
BGP Preference — Global Rollback

Modify BGP route selection so MPLS paths are always preferred over SRv6 paths. On all PEs: set local-preference higher for MPLS-originated VPN routes, or use a route-map to set a lower local-preference on SRv6 next-hops. Traffic immediately shifts back to MPLS across all VRFs. Reversible by reverting the route-map.

2
SR-Policy Candidate Path — Per-Policy Rollback

If SRv6 traffic is steered via SR-Policy, lower the preference of the SRv6 candidate path below the MPLS candidate path. The MPLS candidate path with higher preference becomes active. No configuration removal needed — change one preference value.

3
Per-VRF Import Policy — Selective Rollback

For per-service migration (see Section 7), rollback individual VRFs by modifying the BGP import policy to prefer the MPLS-labeled next-hop over the SRv6 SID next-hop. VRF-A rolls back while VRF-B continues on SRv6. Finest granularity — recommended approach.

Rollback Procedure per Phase

Each migration phase has a distinct rollback procedure. Document and test each one before executing the forward migration step.

R1
Phase 1 Rollback — Remove IPv6 from Core Links

Trigger: IPv6 adjacency instability, ISIS SRv6 TLV issues, or IPv6 forwarding errors on core interfaces.

Action: Remove ipv6 address and ipv6 enable from P-router interfaces. This immediately takes down ISIS IPv6 adjacencies and withdraws all SRv6 locator routes from the IGP.

Risk: Low. MPLS data plane unaffected. ISIS IPv4 adjacencies remain up. All MPLS VPN traffic continues normally.

Recovery time: < 5 minutes. ISIS reconverges on IPv4 topology only.

R2
Phase 2 Rollback — Remove SRv6 Locators

Trigger: SRv6 SIDs not properly distributed, locator prefix reachability issues, or FIB programming failures.

Action: Remove the segment-routing srv6 locator configuration from PE routers. ISIS stops advertising SRv6 Locator TLVs. BGP SRv6 SIDs are withdrawn.

Risk: Low. Ensure BGP SRv6 advertisements are also suppressed (see R3) to prevent BGP from referencing unreachable SIDs.

Recovery time: < 10 minutes. ISIS locator withdrawal + BGP update propagation.

R3
Phase 3 Rollback — Revert BGP to MPLS-Only Advertisements

Trigger: BGP SRv6 Prefix-SID attribute causing interop issues, RR not propagating SRv6 attributes correctly, or VPN traffic taking SRv6 path with packet loss.

Action: On PE routers, modify the BGP VPNv4 address family to suppress SRv6 SID advertisement. Re-enable MPLS label advertisement if it was deprioritized. Modify local-preference to make MPLS paths preferred (LP=200 for MPLS, LP=150 for SRv6).

Risk: Medium. Brief traffic interruption (sub-second with pre-positioned MPLS paths) during BGP re-advertisement. MPLS paths must already exist in the RIB.

Recovery time: < 30 seconds for pre-positioned MPLS paths. Up to 3 minutes for full BGP reconvergence.

R4
Phase 4 Rollback — Re-Enable MPLS (If Partially Decommissioned)

Trigger: MPLS was partially or fully decommissioned and SRv6 is not stable. This is the most complex rollback scenario.

Action: Re-enable IS-IS SR-MPLS extensions (segment-routing mpls). Re-advertise Prefix-SIDs in ISIS TLV 135. Re-enable LDP if it was also running. Restore BGP VPNv4 MPLS label advertisements on all PEs.

Risk: High. MPLS label re-allocation may differ from original labels if SRGB was modified. Verify label bindings before declaring rollback complete. Traffic disruption likely during MPLS re-convergence.

Recovery time: 5–30 minutes depending on network scale. MPLS labels must re-program in LFIB on all nodes.

If you are at Phase 4 rollback, something went seriously wrong. This scenario should be prevented by maintaining MPLS through all earlier phases.

Advantages and Risks

Advantages

  • Gradual migration: Move one VRF at a time; no big-bang cutover
  • Per-service control: Different services can migrate at different rates based on readiness
  • Easy rollback: MPLS always available as fallback; preference change reverts instantly
  • No gateway needed: Same PE runs both — no stitching node required
  • Validation opportunity: Compare SRv6 and MPLS performance side-by-side on the same network
  • Zero-risk learning: Operators gain SRv6 operational experience without stakes

Risks

  • Double control plane state: ISIS and BGP carry twice the information — router memory and CPU increase
  • Operational complexity: NOC must understand two data planes simultaneously — higher training burden
  • ECMP imbalance: MPLS and SRv6 paths may load-balance differently, causing asymmetric traffic distribution
  • Extended exposure window: Longer dual-stack period = longer period of higher complexity
  • Monitoring gap: Some tools may not correctly attribute drops on SRv6 paths in a mixed environment
  • Configuration drift: Two parallel configs are harder to audit and keep consistent
Recommendation: Dual-stack is the right approach for most production migrations. Accept the increased control plane state as a temporary cost. Set a hard deadline for completing the migration — open-ended dual-stack without a deadline tends to become permanent dual-stack, which is the worst of both worlds.
Gateway / Stitching Nodes

Bridging MPLS and SRv6 domains when full dual-stack is not possible on every node

When You Need a Gateway

Not every router in your network can or will be upgraded to support SRv6. Hardware may be end-of-life, vendors may differ, or budget may constrain the upgrade timeline. A gateway (stitching) node sits at the boundary between the MPLS domain and the SRv6 domain and translates between the two data planes.

Gateway Use Cases

Partial Migration
Core upgraded to SRv6; edge (PE) routers still MPLS-only. Gateway translates at the core/edge boundary.
Multi-Vendor
Vendor A supports SRv6; Vendor B does not. Gateway sits at the inter-vendor boundary.
Acquired Networks
Merged network has MPLS; parent network runs SRv6. Gateway bridges the two until full integration.
Legacy Hardware
Older ASICs cannot forward SRv6 at line rate. Keep them on MPLS; gateway handles translation.
Inter-AS
AS boundary between SRv6 domain and MPLS domain in a different AS or provider.
Phased Rollout
Migrate regions one at a time. Gateway at regional boundary enables independent migration schedules.

Gateway Considerations

Performance

The GW performs encapsulation and decapsulation on every transiting packet. Unlike transit MPLS or SRv6 nodes (which only swap/update a header field), the GW must:

  • Pop or parse a full MPLS stack or SRH
  • Perform a stitching table lookup (VPN label → SRv6 SID mapping, or vice versa)
  • Push a new encapsulation header

This is significantly more work per packet than pure transit forwarding. GW throughput may be 30–50% lower than its rated line rate if encap/decap is done in software. Verify ASIC support for GW operations at line rate.

A GW that cannot forward at line rate becomes a network bottleneck and a single point of congestion. Capacity plan the GW separately from the rest of the core.

Redundancy Architecture

Active/Active GW Pair

  • Both GWs process traffic simultaneously
  • ECMP across both GWs from P1 and P2
  • Full capacity when both up; 50% capacity if one fails
  • Requires symmetric stitching tables on both

Active/Standby GW Pair

  • One GW active, one in hot standby
  • BFD detects failure; IGP metric shifts traffic to standby
  • Failover in < 1 second with BFD
  • Simpler stitching table management

Asymmetric Path Handling

In most deployments, MPLS→SRv6 translation goes through GW-A while SRv6→MPLS translation goes through GW-B (or the same GW in the other direction). This asymmetry is normal and expected. Ensure that:

  • Both GWs have complete stitching tables for all VPN label ↔ SRv6 SID mappings
  • Stateless forwarding only — GW must not require per-flow state (that would break asymmetric paths)
  • ECMP between GWs does not cause flow reordering — use flow-based (5-tuple) ECMP hashing

Preference Control

Transport selection is controlled entirely in the BGP policy layer — no data plane changes needed. Three mechanisms give you fine-grained control from per-domain down to per-VRF granularity.

A
BGP Local Preference — Coarse-Grained Global Control

Apply a route-map on ingress at the Route Reflector or on PE import that sets local-preference 200 for paths carrying a BGP Prefix-SID attribute (SRv6 paths) and local-preference 100 for paths with only the MPLS label. This makes SRv6 preferred globally across all VRFs on all PEs that apply the policy.

Rollback: Change SRv6 local-preference to 50 in the same route-map. Immediate effect — all traffic reverts to MPLS without any data plane configuration changes.

B
Route-Map with Next-Hop Type Match — Per-VRF Control

Match on the BGP Prefix-SID attribute presence using match srv6-sid (vendor-specific syntax). Set different local-preference per VRF: VRF-A import policy prefers SRv6 (LP=200), VRF-B import policy prefers MPLS (LP=200). VRF-C has no policy yet — keeps default MPLS behavior. Provides individual VRF migration control without affecting others.

C
SR-Policy Color — Traffic-Engineered Path Control

Attach a BGP color extended community to VPN routes at PE2 during advertisement. Define SR-Policies at PE1: color 100 → SRv6 segment list, color 200 → MPLS segment list. ODN auto-creates SR-Policies per color. Color assignment per-VRF at PE2 controls which PE1 SR-Policy the traffic uses. Enables simultaneous per-VRF TE path selection and transport migration.

SRv6 Functions for L2 Forwarding

L2 forwarding requires different endpoint behaviors than L3 forwarding. SRv6 defines three key functions for EVPN. Understanding which function to use for which traffic type is essential for correct EVPN behavior.

End.DT2M
Decapsulate outer IPv6 + SRH. Flood frame to all CE interfaces in the EVPN instance. Used for BUM (Broadcast, Unknown unicast, Multicast) traffic. Mapped from EVPN Type-3 (IMET) routes. Every remote PE advertising Type-3 for an EVI gets this SID.
End.DT2U
Decapsulate outer IPv6 + SRH. Perform MAC table lookup in the EVPN instance. Forward to the appropriate CE interface. Used for known unicast MAC forwarding within a VLAN/EVI. Provides MAC-based forwarding without specifying the exact interface.
End.DX2
Decapsulate outer IPv6 + SRH. Forward Ethernet frame directly to a specific CE interface (attachment circuit). Used for P2P Ethernet circuits (VPWS) or for EVPN E-LINE services where the MAC is pre-known. No MAC lookup needed.

EVPN Migration Steps

Migrate EVPN services one EVI (EVPN Instance) at a time. Each EVI is an independent migration unit — migrating one EVI does not affect others on the same PE.

1
Enable SRv6 on EVPN PEs

Configure the SRv6 locator on PE2 (and PE1) if not already done during L3VPN migration. Allocate SID functions for EVPN: End.DT2M, End.DT2U, and End.DX2. Verify SIDs are programmed in the local SID table: show segment-routing srv6 sid. Confirm SRv6 locator is reachable from all remote PEs via ISIS.

2
Advertise EVPN Routes with SRv6 SID

Under the EVPN address family on PE2, configure SRv6 encapsulation and specify the SID for each EVPN instance. BGP will now advertise EVPN Type-2 and Type-3 routes carrying the BGP Prefix-SID attribute (SRv6-L2-Service TLV) alongside (or replacing) the MPLS label. During dual-advertisement, both MPLS label and SRv6 SID are advertised.

3
Verify EVPN Routes Have SRv6 SID

On the remote PE (PE1), check the BGP EVPN table for Type-2 and Type-3 routes from PE2:

show bgp l2vpn evpn rd 65001:1 route-type 3 detail

4
Migrate Individual EVIs One at a Time

For EVI-1: set preference for SRv6 SID over MPLS label in the BGP import policy. Verify PE1 switches BUM flooding to use End.DT2M SID. Verify unicast traffic uses End.DX2 or End.DT2U SID. Monitor per-EVI traffic counters. Only proceed to EVI-2 after EVI-1 is validated. This is your blast radius: one EVI per change window.

5

From CE1, send a broadcast (ARP request) to CE2. On PE2, verify the End.DT2M SID counter increments. Verify CE2 receives the ARP. From CE1, send a unicast frame to CE2's MAC. Verify End.DX2 counter increments. Verify MAC table on PE1 shows CE2's MAC learned via SRv6 (not MPLS) path.

6

After all EVIs on a PE pair are validated on SRv6: suppress MPLS label advertisement in the EVPN BGP address family on PE2. Withdraw MPLS-based EVPN routes. Remote PEs will only have SRv6 paths in their BGP table for PE2's EVIs. Monitor for any traffic interruption. Confirm MPLS BUM flooding counters are zero. This completes the EVPN migration for this PE pair.

After Step 6 completes for all PE pairs: EVPN is fully on SRv6. MPLS can be decommissioned from the EVPN address family. The ISIS SR-MPLS configuration may still be needed for other services until all L3VPN services are also migrated.

Phase 1: IPv6 Underlay

Establish the IPv6 routing mesh that SRv6 will run on. All core links must be dual-stacked and ISIS must converge a complete IPv6 topology. This phase has no impact on MPLS services.

1
Plan IPv6 addressing
Assign IPv6 loopback addresses to all PEs and P-routers (e.g., PE1 = 2001:db8::1/128, PE2 = 2001:db8::4/128). Assign link-local or global /64s to core links. Define the SRv6 locator block: 2001:db8:cafe::/48 with per-node /64 locators (PE1 = 2001:db8:cafe:1::/64, PE2 = 2001:db8:cafe:4::/64). Document all assignments before touching devices.
2
Enable dual-stack on all core links
Add IPv6 addresses to PE-P and P-P interfaces. Enable IPv6 unicast routing globally. Verify IPv6 link-local adjacencies form on each interface. This does not affect IPv4/MPLS forwarding — dual-stack coexists cleanly.
3
Enable ISIS multi-topology for IPv6
In ISIS configuration: add address-family ipv6 unicast with multi-topology. Redistribute IPv6 loopbacks into ISIS. ISIS MT-IPv6 (topology ID 2) carries IPv6 reachability independently of IPv4 topology. Verify ISIS adjacencies remain up throughout — MT extension is backward-compatible.
4
Verify full IPv6 reachability mesh
From each PE, ping6 every other PE's IPv6 loopback. Run show isis database detail to verify IPv6 prefixes are in the LSDB. Run show route ipv6 to verify FIB. Every PE must reach every other PE's /128 loopback via ISIS.
5
Enable jumbo MTU on all core links
SRv6 adds 40 bytes (outer IPv6 header) plus up to 40 bytes (SRH) per encapsulation. Core links should support at least 9000-byte MTU (jumbo). Configure mtu 9000 on all PE-P and P-P interfaces. Verify with ping6 size 8900 df-bit between PE loopbacks.
Phase 1 Gate: All PEs can ping6 each other's IPv6 loopbacks successfully. ISIS LSDB on all nodes shows IPv6 prefixes for all routers. MTU test with 8900-byte ping passes on all core links. Zero impact on MPLS services — MPLS traffic counters unchanged throughout.

Phase 2: SRv6 Control Plane

Configure SRv6 on all PEs and enable ISIS and BGP SRv6 extensions. At the end of this phase, all PEs have every other PE's SRv6 SIDs programmed in their FIB. No service traffic moves to SRv6 yet.

1
Configure SRv6 locators on all PEs
On each PE: configure the SRv6 locator block and node-specific locator. Example on PE1: segment-routing srv6 locator LOC1 prefix 2001:db8:cafe:1::/64. The locator /64 is automatically advertised into ISIS as a reachability prefix. Verify with show segment-routing srv6 locator.
2
Enable ISIS SRv6 extensions (advertise locators)
Under ISIS: address-family ipv6 unicast segment-routing srv6. ISIS will advertise SRv6 Locator TLVs (TLV 27) for each configured locator. Verify with show isis database detail — look for "SRv6 Locator" entries for each router's locator prefix.
3
Enable BGP SRv6 capability on PE sessions
On each PE under BGP VPNv4/VPNv6 address-family: segment-routing srv6 capability. This enables BGP to encode SRv6 SID (Prefix-SID attribute, type 5) alongside MPLS labels in UPDATE messages. Both MPLS label and SRv6 SID are advertised simultaneously during dual-stack coexistence.
4
Verify SRv6 SIDs appear in ISIS LSDB
Run show isis database detail on each router. Verify SRv6 Locator TLV (TLV 27) is present for every PE in the LSDB. Verify the locator prefix matches the configured /64 (e.g., PE2's locator 2001:db8:cafe:4::/64 appears in PE1's LSDB).
5
Verify SRv6 SIDs in FIB on all PEs
Run show segment-routing srv6 sid — local SIDs should be listed (End, End.DT4, etc.). Run show route ipv6 2001:db8:cafe:4::/64 on PE1 to verify PE2's locator is reachable. Run show bgp vpnv4 unicast detail — VPN routes from PE2 should now show both MPLS label 24001 AND SRv6 SID 2001:db8:cafe:4:dt4::.
Phase 2 Gate: Every PE has every other PE's SRv6 locator /64 in its IPv6 FIB. Every PE has every other PE's End.DT4 SIDs programmed. BGP VPN routes carry both MPLS label and SRv6 SID. Service traffic is still entirely on MPLS — SRv6 is in standby. This can be confirmed by checking MPLS traffic counters (should still be incrementing) and SRv6 encap counters (should be zero).

Phase 3: Service Migration

Flip services from MPLS to SRv6, starting with a pilot VRF. The dual-advertisement from Phase 2 means each PE has both an MPLS path and an SRv6 path — BGP preference determines which is used. Migration is reversible until Phase 4.

1
Enable dual-advertisement for L3VPN (MPLS label + SRv6 SID)
Verify BGP is advertising both MPLS label 24001 and SRv6 SID 2001:db8:cafe:4:dt4:: for VRF-A prefixes. This should already be active from Phase 2. Confirm with show bgp vpnv4 unicast 172.16.2.0/24 detail — both attributes must be present before proceeding.
2
Migrate pilot VRF to SRv6 (set BGP preference)
Select a low-risk pilot VRF (minimal traffic, non-critical customer). Set BGP route preference to prefer the SRv6 path: configure bgp bestpath prefer-srv6 or equivalent policy for the pilot VRF. Alternatively, use a route-map to set local-preference higher for SRv6-encapsulated paths. Verify the pilot VRF's traffic switches to SRv6 encapsulation.
3
Verify pilot VRF traffic on SRv6 path
Check SRv6 encap counters on PE1 — should be incrementing for pilot VRF. Check IPv6 traffic counters on core links — should increase. Check MPLS counters for pilot VRF — should drop to zero. Verify end-to-end latency and packet loss are unchanged. Run extended ping CE1 → CE2 with large packets to verify MTU.
4
Migrate remaining L3VPNs
After pilot soak period (minimum 48–72 hours), migrate remaining L3VPNs in batches. Recommended batch size: 5–10 VRFs per change window. Verify each batch before proceeding. Monitor for customer complaints, latency spikes, or packet loss during each migration window.
5
Migrate L2VPN/EVPN
EVPN SRv6 uses the same BGP dual-advertisement mechanism. Migrate EVPN EVI instances to SRv6 after all L3VPNs are stable. EVPN SRv6 requires End.DX2 (L2 cross-connect) or End.DT2M (multicast) SIDs — verify these are configured on all PEs before migrating EVPN.
6
Migrate SR-TE policies
SR-TE policy framework (candidate paths, preference, BSID) is identical in MPLS and SRv6. Difference: segment list entries change from MPLS label SIDs to SRv6 SIDs (128-bit). Update policy segment lists to reference SRv6 node SIDs. Verify traffic engineering constraints (latency, affinity) are honored on SRv6 paths.
Phase 3 Gate: All services verified on SRv6 paths. MPLS traffic counters approaching zero (only control-plane MPLS traffic remaining). SRv6 encap/decap counters stable. No customer-reported issues. MPLS path is still available as backup — do not remove until Phase 4 soak period complete.

Phase 4: MPLS Cleanup

Remove all MPLS SR configuration once services have been running on SRv6 for a minimum soak period. This phase is irreversible — once label space is reclaimed, MPLS paths cannot be quickly restored.

Point of No Return: Execute Phase 4 only after a minimum 2-week soak period with all services on SRv6 and zero MPLS forwarding traffic. Verify with operations and management before proceeding. There is no automated rollback after label space is reclaimed.
IRREVERSIBLE — label reclaim cannot be undone without full re-migration
1
Stop BGP MPLS label advertisements
Remove send-label / allocate-label all from BGP VPNv4/VPNv6 address-family configuration. BGP will withdraw MPLS label attributes from UPDATE messages. Remote PEs will lose their MPLS VPN label for local prefixes and must use SRv6 SID only. Verify remote PE BGP tables show SRv6 SID only (no MPLS label) for advertised prefixes.
2
Remove SR-MPLS Prefix-SID configs
Under ISIS: remove segment-routing prefix-sid configuration from all loopback interfaces. This withdraws the Prefix-SID sub-TLV from ISIS LSPs. ISIS will no longer advertise SRGB-indexed labels for node SIDs. Verify with show isis database detail — Prefix-SID sub-TLVs should be absent.
3
Remove SRGB/SRLB configs
Remove segment-routing global-block 16000 23999 and any SRLB configuration. The SRGB label range 16000–23999 is released back to the label manager. MPLS forwarding entries for Node-SIDs (16001, 16004, etc.) are removed from the LFIB.
4
Remove LDP if still running
If LDP was running alongside SR-MPLS (for legacy services), disable it now: remove mpls ldp configuration globally and per-interface. LDP adjacencies will drop, LDP label bindings will be withdrawn. Verify no services depended on LDP labels.
5
Reclaim label space
Remove MPLS configuration from interfaces (no mpls ip / no mpls traffic-eng). The SRGB label range 16000–23999 is reclaimed and available for other uses (e.g., service labels if needed). MPLS forwarding is disabled. Core links carry only IPv6 traffic.
Phase 4 Complete: Network is now SRv6-only. No MPLS forwarding tables exist. All services run on IPv6/SRv6 transport. Begin 30-day monitoring period before declaring migration complete. Document the final SRv6 architecture for operations.
VERIFY MPLS traffic counters are zero on all interfaces before final sign-off.

OAM Differences: MPLS vs SRv6

SRv6 OAM is architecturally simpler than MPLS OAM because SRv6 is built on standard IPv6. Existing IPv6 tools (ping6, traceroute6) work without modification. MPLS OAM required specialized protocol extensions (RFC 4379/8029) because standard IP tools could not traverse the MPLS data plane.

ICMPv6 Echo Request to SRv6 End SID or locator (e.g., ping6 2001:db8:cafe:4::). For VPN verification, use SRv6 OAM (RFC 9259) with SRH targeting the service SID — standard ping6 to End.DT4 will not work as expected. YES

BFD over SRv6

Bidirectional Forwarding Detection (BFD) provides sub-50ms failure detection over SRv6 paths, equivalent to MPLS BFD. BFD sessions are established between PE loopbacks, with BFD packets SRv6-encapsulated by the data plane transparently.

MPLS BFD (existing)
  • BFD session: PE1 ↔ PE2 over LSP
  • BFD PDU encapsulated in MPLS label stack
  • Outer transport label: 16004 (PE2 Node-SID)
  • BFD uses UDP port 3784
  • Discriminator identifies LSP being monitored
  • Detection: ≤ 50ms with 10ms intervals
SRv6 BFD (target state)
  • BFD session: PE1 ↔ PE2 over SRv6 path
  • BFD PDU in UDP over IPv6 outer header
  • Outer IPv6 DA: 2001:db8:cafe:4:: (PE2 locator)
  • BFD uses same UDP port 3784
  • No SRH needed (single SID, H.Encaps.Red)
  • Detection: ≤ 50ms — same as MPLS BFD

1
Establish SRv6 BFD sessions alongside existing MPLS BFD
Configure BFD over SRv6 path on each PE: associate BFD with the SRv6 locator next-hop. Both MPLS BFD and SRv6 BFD sessions run simultaneously during Phase 3. This verifies SRv6 path liveness before MPLS BFD is removed.
2
Verify SRv6 BFD sessions reach Up state
show bfd session detail — SRv6 sessions should show state=Up, local discriminator assigned, detection time ≤ 50ms. Both MPLS and SRv6 BFD sessions should be Up simultaneously (verifying bidirectional path health).
3
Test SRv6 BFD failure detection
In a maintenance window: temporarily take down a core link and verify SRv6 BFD detects failure within configured detection time. Verify BGP session held (BFD should trigger IGP reroute, not BGP session drop). Restore link and verify BFD recovers.
4

Once all services are on SRv6 and SRv6 BFD sessions are stable, remove MPLS BFD configuration. MPLS BFD sessions will go down (expected). SRv6 BFD remains as the sole fast-failure detection mechanism.
SRv6 BFD advantage: SRv6 BFD sessions use standard UDP-over-IPv6, making them visible to standard IPv6 monitoring tools. MPLS BFD required MPLS-aware monitoring. SRv6 BFD detection time is identical to MPLS — no degradation in failure detection capability.

Troubleshooting Migration Issues

Diagnostic playbooks for the five most common failure modes during MPLS-to-SRv6 migration, with symptoms, root causes, and remediation steps

Issue 1: SRv6 SID Not in FIB

Severity: HIGH — Traffic is black-holed. VRF routes resolved via SRv6 have no forwarding entry. Packets sent to SRv6 DA are dropped at the first router with no route.

Symptoms

  • Traffic black-holed for VRFs that were migrated to SRv6 preference
  • ping6 2001:db8:cafe:4:: from PE1 fails (no route to locator)
  • SRv6 encap counter incrementing on PE1, but decap counter zero on PE2
  • show route ipv6 2001:db8:cafe:4::/64 shows no entry or unreachable

Root Causes & Diagnostics

Cause A: Locator not advertised in ISIS
Check: show isis database detail level-2 on any router — look for "SRv6 Locator" TLV (TLV 27) for the affected PE. If absent: the PE's ISIS SRv6 extension is not enabled or locator is not bound to ISIS.
Fix: Under ISIS on the affected PE: address-family ipv6 unicast segment-routing srv6 locator LOC1. Verify locator appears in ISIS LSP after config.
Cause B: ISIS SRv6 extensions not enabled
Check: show isis segment-routing srv6 detail — SRv6 should show "enabled" status. If not: ISIS is running without SRv6 capability enabled.
Fix: Enable segment-routing srv6 under ISIS address-family. This triggers LSP refresh and locator advertisement.
Cause C: SRv6 SID not configured on PE
Check: show segment-routing srv6 sid on the egress PE — End.DT4 SID for the target VRF should appear. If absent: SRv6 SID allocation is not configured for the VRF.
Fix: Under VRF configuration: segment-routing srv6 locator LOC1 alloc mode per-vrf. Verify SID appears in show segment-routing srv6 sid.
Cause D: FIB programming error
Check: show route ipv6 2001:db8:cafe:4::/64 shows route present in RIB, but show cef ipv6 2001:db8:cafe:4:dt4:: shows no FIB entry or drop action. This indicates a RIB→FIB programming failure.
Fix: Clear and reprogram FIB: clear ipv6 route 2001:db8:cafe:4::/64 and verify ISIS re-installs. If persistent: contact vendor support — this is a platform bug.

Issue 2: MTU Black-Holes

Severity: MEDIUM-HIGH — Small packets work, large packets silently dropped. Especially insidious because control-plane sessions (BGP, BFD) use small packets and remain up, masking the data-plane problem.

Symptoms

  • Ping works, but TCP sessions hang or transfer very slowly
  • Small pings (e.g., 64 bytes) succeed, large pings (1400+ bytes) fail

Transit routers that do not support ICMPv6 "Packet Too Big" (or have ICMP rate-limiting too aggressive) silently drop oversized packets, making the problem hard to diagnose.

Diagnostics

1
Ping with size sweep
ping6 2001:db8::4 size 1000 → success. ping6 2001:db8::4 size 1460 → success. ping6 2001:db8::4 size 1461 → fail. This identifies the MTU boundary. The exact failure point tells you where the bottleneck interface is (1461 customer bytes + 40B SRv6 header = 1501 bytes — link MTU is 1500).
2
Check interface MTU on all core links
show interfaces detail | include MTU on all PE-P and P-P links. All core links must support at least mtu 9000 (or minimum 1600 if jumbo not available). Identify any 1500-byte MTU links in the core path.
3
Check MTU exceeded counters
show interfaces GigE0/1 | include MTU-exceeded on each transit hop. A non-zero MTU exceeded counter on a specific interface pinpoints the bottleneck. Zero counters everywhere with packet loss indicates silent drop without counter increment (ASIC behavior — check vendor documentation).

Fixes

Preferred Fix: Enable Jumbo MTU
Configure mtu 9000 on all core PE-P and P-P interfaces. This accommodates SRv6 overhead plus customer frames up to ~8960 bytes. Zero service disruption — MTU increase is transparent to transit traffic.
Alternative: Reduce CE-facing MTU
If core links cannot be upgraded to jumbo: reduce advertised MTU on CE-facing interfaces to 1460 (SRv6 H.Encaps.Red) or 1420 (SRv6 with SRH). This reduces customer throughput but eliminates MTU drops. Last resort only.

Issue 3: BGP Not Advertising SRv6 SID

Severity: MEDIUM — Remote PE doesn't see the SRv6 SID for a VPN prefix. The remote PE can only use the MPLS label path — SRv6 cutover for affected VRFs is blocked.

Symptoms

  • On remote PE: show bgp vpnv4 unicast 172.16.2.0/24 detail shows MPLS label 24001 but NO SRv6 SID attribute
  • SRv6 preference cannot take effect — no SRv6 path available
  • Remote PE FIB shows MPLS next-hop only for VRF prefix

Diagnostic: show bgp vpnv4 unicast detail

BGP routing table entry for 100:1:172.16.2.0/24
  Paths: (1 available)
    Path: 10.0.0.4 (PE2)
    MPLS label: 24001                ← Present
    Prefix-SID attribute: absent     ← PROBLEM — SRv6 SID not advertised
    BGP next-hop: 10.0.0.4
    Best: yes (only path)
Compare to expected state: 
    MPLS label: 24001
    Prefix-SID (SRv6): 2001:db8:cafe:4:dt4::  ← Should be present
Cause A: BGP SRv6 capability not enabled
Check on advertising PE (PE2): show bgp neighbors 10.0.0.1 detail | include SRv6 — "SRv6 Capability: advertised, received" should appear. If only "advertised" or absent: capability negotiation failed.
Fix: Under BGP neighbor or address-family: capability extended-nexthop and segment-routing srv6. Reset BGP session to renegotiate capabilities.
Cause B: VRF not configured for SRv6 export
Check on PE2: show segment-routing srv6 sid | include VRF-A — End.DT4 SID for VRF-A should appear. If absent: VRF is not bound to an SRv6 SID.
Fix: Under VRF-A on PE2: segment-routing srv6 locator LOC1. This allocates an End.DT4 SID for the VRF and instructs BGP to include the SRv6 Prefix-SID attribute in UPDATE messages.
Cause C: SRv6 locator not assigned to VRF
Check on PE2: show segment-routing srv6 locator LOC1 — locator should be in "active" state with SID allocation enabled. If locator is configured but inactive: ISIS may not have converged the locator prefix, or there is a config dependency order issue.
Fix: Verify locator prefix 2001:db8:cafe:4::/64 is reachable in IPv6 RIB, then rebind VRF to locator.
Cause D: Route-map filtering SRv6 attributes
Check: show route-map — any route-map applied to BGP VPNv4 neighbors that strips extended communities or Prefix-SID attributes will silently drop the SRv6 SID.
Fix: Audit all route-maps applied to PE-to-PE BGP sessions. Ensure no set community none or similar action removes the Prefix-SID attribute (BGP attribute type 40).

Issue 4: Gateway Stitching Issues

Severity: HIGH — Traffic reaches the Gateway but does not cross to the other domain. The GW is the single translation point between MPLS and SRv6 — any misconfiguration causes a complete service outage for affected VRFs.

Symptoms

  • Traceroute from CE1 reaches GW but stops there (no further hops)
  • GW interface counters show input packets but no output on the egress side
  • Packet capture at GW: packets arrive, but no outgoing packets visible on egress interface
  • Asymmetric behavior: one direction works, other direction doesn't
Cause A: GW missing cross-connect mapping
The GW must maintain a bidirectional mapping: MPLS VPN label 24001 ↔ SRv6 SID 2001:db8:cafe:4:dt4::.
Check: show segment-routing srv6 sid on GW — should show a "stitching" SID that maps to the MPLS domain VPN label. Also verify MPLS LFIB has an entry that maps to SRv6 DA.
Fix: Configure the cross-connect explicitly, or verify BGP is distributing the mapping via BGP-LU or SR-Policy cross-domain stitching. GW must participate in both BGP domains (MPLS VPN and SRv6).
Cause B: GW doesn't have both MPLS and SRv6 forwarding tables populated
Check on GW: show mpls forwarding-table vrf VRF-A (for MPLS→SRv6 direction) and show route ipv6 2001:db8:cafe:4::/64 (for SRv6→MPLS direction). Both must have valid entries.
Fix: Verify GW has IGP/BGP peering in both MPLS and SRv6 domains. GW must run ISIS in both domains or have static routes covering both locator prefixes and MPLS label bindings.
Cause C: Asymmetric MTU (MPLS side smaller)
GW receives an SRv6 packet (40B overhead) and re-encapsulates as MPLS (8B overhead). If the GW's egress MTU toward the MPLS domain is less than the original payload + 8B, packets are dropped. The SRv6 payload is typically larger than expected by MPLS-domain routers.
Fix: Ensure GW's MPLS-domain egress interfaces support same jumbo MTU as SRv6 domain. GW should perform ICMP fragmentation or set DF bit correctly during translation.
Cause D: Routing loop at GW
GW's route for the VPN prefix may resolve back through itself if cross-connect is misconfigured. Symptom: TTL=1 packets arrive at GW from its own interfaces.
Check: show route vrf VRF-A 172.16.2.0/24 on GW — next-hop must not be a GW interface or the GW's own loopback.
GW diagnostic checklist: (1) Capture packets on both MPLS-side and SRv6-side GW interfaces simultaneously, (2) Verify cross-connect mapping in both directions, (3) Check both MPLS LFIB and IPv6 FIB on GW, (4) Verify BGP sessions on GW are up in both domains.

Issue 5: ECMP Imbalance During Coexistence

Severity: LOW-MEDIUM — Traffic engineering issue, not a complete outage. Some links carry more traffic than expected, potentially causing congestion while others are underutilized. Can cause latency spikes or drops under load.

Symptoms

  • Some core links near saturation, others at low utilization despite ECMP configuration
  • Telemetry shows uneven traffic distribution across equal-cost paths
  • Problem appears only after partial migration (some VRFs on MPLS, some on SRv6)
  • Traffic from the same customer appears on different paths after SRv6 migration

Root Cause

MPLS ECMP and SRv6 ECMP use different hash inputs, causing different load distribution:

Attribute SR-MPLS ECMP Hash SRv6 ECMP Hash
Primary hash inputs Top MPLS label, IP 5-tuple (if label-IP aware) IPv6 SA, IPv6 DA, IPv6 Flow Label, Next Header
Flow label Not applicable IPv6 Flow Label (20-bit) — may be zero if not set by PE
Inner IP visibility Depends on MPLS entropy label (EL) Inner IP not visible at transit without deep inspection
Hash polarization risk Medium — label space provides entropy High if Flow Label not set (all traffic has same SA/DA)

Fixes

1
Enable IPv6 Flow Label entropy on ingress PEs
Configure PE1 (and all ingress PEs) to set the IPv6 Flow Label field in the SRv6 outer header based on the inner flow 5-tuple. This provides equivalent entropy to MPLS entropy labels. Command (vendor-specific): segment-routing srv6 encapsulation source-address flow-label-entropy. Verify with packet capture that Flow Label field is non-zero and varies per flow.
2
Ensure consistent ECMP hashing on transit nodes
Verify all transit P-routers (P1, P2) use the same ECMP hash fields for IPv6 (SA + DA + Flow Label + protocol). Inconsistent hashing on different nodes can cause packet reordering within a flow. Check vendor documentation for IPv6 ECMP hash configuration and ensure all nodes use the same algorithm.
3
Use SR-Policy to control path selection during migration
If ECMP imbalance is causing congestion on specific links: deploy SR-TE policies to explicitly steer specific VRFs or customers to specific paths. This provides deterministic load control independent of ECMP hash behavior. Remove SR-TE constraints after migration is complete and ECMP hashing is verified to be balanced.
ECMP monitoring: Track per-interface utilization with 30-second granularity during Phase 3. An ECMP imbalance above 20% (some links at >80% while equal-cost links are at <40%) warrants investigation. After full SRv6 migration (Phase 4), re-validate ECMP hash behavior with only SRv6 traffic — the dual-stack coexistence period often masks the true SRv6-only hash distribution.

Decision Matrix: Dual-Stack vs Gateway Migration

Choose your migration model based on these criteria. The decision is primarily driven by hardware capabilities and operational tolerance for complexity.

Criteria Answer Recommendation Rationale
All PEs support SRv6 data plane natively? YES Dual-Stack No GW needed. Each PE handles both data planes simultaneously. Simpler operationally.
All PEs support SRv6 data plane natively? NO Gateway Legacy PEs cannot run SRv6. GW stitches MPLS and SRv6 domains. Mandatory choice.
Mixed hardware — some SRv6, some MPLS-only? MIXED Gateway GW isolates the two populations. SRv6 PEs migrate immediately; MPLS-only PEs stay behind the GW.
Need per-service (per-VRF) migration control? YES Dual-Stack Dual-stack + BGP preference gives per-VRF migration granularity. Flip one VRF at a time.
Acceptable to maintain double state (MPLS + SRv6 FIB)? YES Dual-Stack Dual-stack requires double FIB entries during migration. If memory allows, preferred approach.
Acceptable to maintain double state? NO Gateway GW keeps each domain clean — MPLS domain has only MPLS state; SRv6 domain has only SRv6 state.
Acquiring/merging with a new SRv6-only network? YES Gateway GW interconnects two independently operated domains without requiring re-architecture of either.
Operator has strong SRv6 expertise? YES Dual-Stack Dual-stack is easier to understand end-to-end. No GW cross-connect mapping to manage.
Operator has limited SRv6 expertise / phased learning? NO Gateway GW limits SRv6 blast radius. SRv6 expertise can be built incrementally on the SRv6 side of the GW.
Target: complete MPLS removal? YES Dual-Stack Dual-stack path ends cleanly with MPLS removal. GW migration requires additional GW decommission step.
Target: long-term coexistence (5+ years)? YES Gateway GW provides clean domain separation for indefinite coexistence with independent lifecycle management.
Summary rule: If all PE hardware supports SRv6 → use dual-stack for simplicity and per-VRF migration control. If any PE is MPLS-only, or you're interconnecting separate domains → use gateway for clean domain isolation.
