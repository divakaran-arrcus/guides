SRv6: Segment Routing over IPv6

Everything you need to bridge from SR-MPLS to SRv6

The Core Idea

SRv6 replaces the MPLS data plane with native IPv6. Instead of pushing MPLS labels onto packets, you encode routing instructions as IPv6 addresses in an extension header. The network forwards using standard IPv6 longest-prefix-match — no MPLS infrastructure needed.

One-line summary: SRv6 uses IPv6 addresses as segment identifiers. The forwarding plane is IPv6. The instruction set is encoded in the destination address and a Segment Routing Header (SRH).

The Biggest Paradigm Shift: Adjacency SIDs Are Routable

In SR-MPLS, an Adj-SID is a local label — to use ZR-11's Adj-SID toward MP-11, you must first push ZR-11's Node-SID to get there. Two labels minimum.

In SRv6, a uA (adjacency SID) is an IPv6 address within ZR-11's locator prefix. Any router can forward toward it using standard IPv6 routing. No separate Node SID is needed.

Locator Prefix

Each SRv6 node advertises one or more locator prefixes in IS-IS or OSPF. The locator is a normal IPv6 routable prefix that all routers install in their FIB. Every SID allocated by that node falls within its locator prefix.

Example: Node ZR-11 advertises locator fcbb:bb01:0011::/48. All of ZR-11's SIDs (uN, uA, End.DT4, etc.) are IPv6 addresses within this /48. Transit routers just do longest-prefix-match toward fcbb:bb01:0011::/48 — they don't need to know about individual SIDs.

Anycast Locators (A-flag)

A locator can be flagged as Anycast (A-flag) in IS-IS. Multiple nodes advertise the same locator prefix, and traffic is routed to the nearest one by IGP metric — exactly like an Anycast loopback in traditional IP or an Anycast prefix-SID in SR-MPLS. This is used for redundant service endpoints (e.g., dual PE nodes for VPN).

H.Encaps vs H.Encaps.Red — watch the Segments Left value: With standard H.Encaps, the first SID appears both in the DA and in the segment list (SL[n]), so SL = n. This off-by-one is the most common source of confusion when reading SRv6 packet captures.

When Is the SRH Present?

The SRH is not always required. If there is only one SID (e.g., a simple L3VPN with no SR-Policy), the ingress PE can set the IPv6 DA to that single SID and omit the SRH entirely. The egress PE matches the DA to its local SID table and executes the behavior. No SRH processing needed.

Rule of thumb: SRH is only needed when there are 2 or more SIDs in the segment list. For single-SID encapsulation (common in basic L3VPN without TE), the packet is just an outer IPv6 header (DA = VPN SID) + inner customer packet. This is the lowest-overhead SRv6 mode — only 40 bytes of outer IPv6 header.

How to design your SRv6 address space — locator allocation, uSID block planning, and migration from SR-MPLS

SRv6 Addressing Architecture

In SR-MPLS, your addressing plan involves SRGB ranges and per-node Prefix-SID indices. In SRv6, the address plan is about IPv6 prefix allocation — locator blocks, per-node prefixes, and function numbering. Getting this right is critical because unlike MPLS labels (which are local and re-assignable), SRv6 SIDs are globally routable IPv6 addresses that appear in IGP, BGP, and packet headers.

Migration Addressing Checklist

When migrating from SR-MPLS to SRv6, plan these addressing elements:

1
Choose the uSID block prefix and size

Recommend: fcbb:bbXX::/32 with 16-bit uSIDs. XX = your unique identifier (to avoid conflicts in multi-operator environments).

2
Design the node ID numbering scheme

Map your existing SR-MPLS Prefix-SID indices to SRv6 node IDs. Example: if PE-1 has Prefix-SID index 1, assign node ID 0001. Preserve the mapping for operational familiarity.

3
Allocate per-algorithm locator ranges

If using Flex-Algo, reserve node ID ranges per algorithm (e.g., 0xxx for algo-0, 1xxx for algo-128). Each node gets one locator per participating algorithm.

4
Reserve Anycast locator IDs

Identify all Anycast PE pairs and assign shared locator IDs from a dedicated range (e.g., 0Axx).

5
Define function numbering policy

Standardize which function ranges map to which behaviors (uN, End.DT4, End.DX2, uA, etc.). Configure this policy consistently across all nodes.

6
Plan IPv6 infrastructure addressing

SRv6 requires IPv6 on all links (for the outer IPv6 header). Plan link /127s or /64s, loopbacks, and management addressing. The SRv6 locator is separate from the loopback — both are needed.

7
Update ACLs and security policies

Permit the uSID block prefix at network edges. Block external traffic destined to your locator space. Update uRPF policies if deployed.

uN — Micro-Node SID

The SRv6 equivalent of an SR-MPLS Node-SID

What uN Does

uN is the micro-SID variant of the End function. It means: "I am a transit waypoint — advance the SID pointer and forward."

uN behavior step by step:
1. Left-shift the uSID container by 16 bits (consume current uSID)
2. If new active position ≠ 0000 → update DA, forward via longest-prefix-match
3. If new active position = 0000 → decrement Segments Left, copy next SRH entry to DA
4. Forward the packet with SRH intact

Key Property: uN Does NOT Remove the SRH

The packet still has the full SRH attached after uN processing. uN only advances the pointer. This is analogous to an MPLS label swap — the label stack stays, only the top label changes.

ECMP Behavior

Since uN forwards via the IGP FIB (longest-prefix-match on the updated DA's locator), it naturally supports ECMP across all parallel paths to the next node. This is identical to how a Node-SID in SR-MPLS allows ECMP.

uN with SID Flavors

A uN SID can carry PSP, USP, and USD flavors. Most commonly, uN is advertised with PSP+USP. When the uN is at the penultimate segment position (SL becomes 0 after processing), PSP flavor causes the node to remove the SRH before forwarding — so the egress node receives a clean IPv6 packet with no SRH to parse. This is critical for multi-vendor interop. See the PSP section for full details.

uA — Micro-Adjacency SID

The SRv6 equivalent of an SR-MPLS Adjacency SID

What uA Does

uA is the micro-SID variant of End.X. It means: "forward out a specific interface/adjacency, not via IGP shortest path."

uA behavior step by step:
1. Left-shift the uSID container by 16 bits (same as uN)
2. Update DA (same as uN)
3. Forward the packet out the specific adjacency bound to this uA (NOT via routing table lookup)

Why uA Doesn't Need uN (Unlike SR-MPLS)

A uA SID is an IPv6 address within the advertising node's locator prefix. Since the locator is in every router's IPv6 FIB, transit routers forward toward it using standard IPv6 routing. No separate Node SID needed — one SID encodes both "get to the node" and "use this adjacency."

ECMP Behavior — The Important Limitation

A standard uA is bound to a single adjacency (link). If there are multiple parallel links between two nodes, uA pins traffic to one. This is the same as per-link Adj-SIDs in SR-MPLS.

Parallel/Bundle uA: Some implementations support a uA that represents an entire link-bundle or LAG, allowing ECMP across parallel links to the same neighbor. This is analogous to an SR-MPLS Adj-SID with B-flag unset (parallel adjacency set). Whether this is available depends on the platform and IS-IS TLV encoding.

Decap Behaviors and SID Flavors

The decap behaviors (End.DT4, End.DT6, End.DX4, etc.) interact with SID flavors:

  • With USP (default): The egress node first removes the SRH (if present with SL=0), then processes the decap function — removing the outer IPv6 header and performing the VRF lookup.
  • With USD: The egress node removes both the SRH and the outer IPv6 header in one step, then does the VRF lookup. This is the most common mode for VPN endpoints.
  • With PSP on the penultimate node: The SRH is already gone when the packet arrives at the egress. The egress just sees DA = its End.DT4 SID with no SRH, decaps the outer IPv6, and does the VRF lookup. This simplifies egress processing and avoids interop issues with SRH handling.
PSP — Penultimate Segment Pop

Not PHP. The penultimate segment node removes the SRH when it's no longer needed.

What PSP Does

PSP is a flavor (optional behavior modifier) that can be applied to any SRv6 SID. When a node processes a SID with PSP flavor and Segments Left becomes 0 (meaning this node is the penultimate segment), it removes the entire SRH extension header before forwarding.

PSP trigger condition: After processing its own SID, if Segments Left = 0 AND the SID has PSP flavor → remove the SRH. The IPv6 header stays. The DA already contains the final SID. Only the SRH extension header is stripped.

The Fundamental Conceptual Difference

PHP: "Pop someone else's label"

The penultimate pops the egress's label because the egress asked for it (implicit-null). It's cooperative — the egress signals "remove my label before sending to me."

PSP: "Remove a now-useless header"

The penultimate segment node removes the SRH — a routing header that has served its purpose (all segments traversed). The final SID is still in DA. The egress doesn't ask for this; it's the penultimate node's own SID configuration.

Key implication: In PHP, fixing a problem at the egress requires the egress to change its signaling. In PSP, the penultimate node acts independently — even if the egress is broken or misconfigured, the penultimate can still strip the SRH. This makes PSP useful for interop workarounds in multi-vendor networks.
TI-LFA in SRv6

Fast-reroute with segment routing — same concept as SR-MPLS TI-LFA, different encoding

How TI-LFA Works in SRv6

TI-LFA (Topology Independent Loop-Free Alternate) in SRv6 uses the same algorithm as SR-MPLS TI-LFA — compute a post-convergence backup path and encode it as an explicit SID list. The difference is in the encapsulation.

SR-MPLS TI-LFA: Push extra labels

The PLR (Point of Local Repair) pushes additional MPLS labels on top of the existing label stack to steer traffic through the backup path.

SRv6 TI-LFA: Wrap in outer IPv6/SRH (H.Encaps)

The PLR encapsulates the entire original packet inside a new outer IPv6 header + SRH that carries the backup path SIDs. This creates a nested encapsulation — an outer IPv6/SRH wrapping the inner IPv6/SRH.

The Nested Encapsulation Challenge

In SR-MPLS, pushing extra labels is straightforward — you just add more labels to the stack. In SRv6, the TI-LFA repair creates an entirely new IPv6 header + SRH around the original packet. This means the endpoint of the backup path must:

  • Decapsulate the outer TI-LFA IPv6/SRH
  • Then process the original inner IPv6/SRH

This nested SRH or nested IPv6 decapsulation can be problematic if the endpoint's implementation doesn't handle it correctly.

Multi-vendor concern: Not all implementations handle nested IPv6/SRH encapsulations cleanly, especially for edge cases like native IPv6 prefixes (non-SRv6 locators). This is a known interop challenge in production SRv6 deployments.

Multihomed Prefix Originator Change

A special case arises with Anycast or multihomed prefixes — when the prefix is originated by multiple nodes and the best originator changes during convergence. TI-LFA must steer traffic to the new originator, which may require adding an extra SID for the new originator's node to prevent microloops.

This extra SID adds another layer to the nested encapsulation, making the decapsulation even more complex at the endpoint. Optimizations exist (Q-space checks, distance comparisons) to suppress the extra SID when the backup path is inherently loop-free, but they don't work for all topologies.

Flex-Algo in SRv6

IGP Flexible Algorithms (RFC 9350) — topology-aware SIDs for constraint-based routing

What Is Flex-Algo?

Flex-Algo lets you define multiple routing topologies within a single IGP instance, each with its own metric type or constraints. Each algorithm computes a separate SPF and produces a separate set of SIDs. Traffic using Algorithm 128's SIDs takes the low-latency path; traffic using Algorithm 129's SIDs takes the path avoiding certain SRLGs — all within the same IGP, same routers, same links.

The SR-MPLS version you know: In SR-MPLS, each node advertises a separate Prefix-SID per algorithm (e.g., Node-SID 16001 for algo 0, Node-SID 16101 for algo 128). The label stack uses the algorithm-specific SID. The FIB has per-algorithm entries. The concept in SRv6 is identical — only the SID encoding changes.

Flex-Algo + SR-Policy

Flex-Algo SIDs can be used directly inside SR-Policy segment lists. When an SR-Policy's segment list references SIDs from an algo-128 locator, the traffic follows the algo-128 constrained topology at each hop.

1
Simple case: All SIDs from the same algo

2
Mixed case: Different algos in the same policy

You can mix SIDs from different algorithms within a single SR-Policy. For example: use algo-128 (low-latency) for the first portion, then algo-0 (default) for the last hop. Each SID determines the forwarding topology for its segment.

3

In many deployments, SR-Policies are created automatically using ODN templates. A Color Extended Community (e.g., color 100 = low-latency) triggers the headend to auto-create a policy using algo-128 SIDs. The mapping is: Color → ODN template → Flex-Algo ID → locator selection → SID list.

TI-LFA + Flex-Algo: TI-LFA computes backup paths within the Flex-Algo constrained topology. If algo-128 excludes certain links, TI-LFA's backup path also excludes them — the repair stays within the constrained topology. This is the same behavior as SR-MPLS Flex-Algo TI-LFA.

Key Advantage: Native IPv6 Tooling

Unlike MPLS where you need specialized LSP ping/traceroute tools, SRv6 OAM uses standard IPv6 tools. A ping6 to an End.DT4 SID verifies that the SRv6 underlay path and SID processing are working — the packet reaches the egress PE and the SID is recognized. To verify true overlay VPN reachability (CE-to-CE forwarding), you still need to ping the customer prefix within the VRF context, just like in MPLS.

What each ping tests:
ping6 fcbb:bb01:0002:0040:: → Tests: SRv6 transport + SID processing at PE-2. Does NOT test VRF forwarding.
ping vrf CUST-A 10.2.2.2 → Tests: Full end-to-end VPN path including VRF lookup and CE reachability.
Practical benefit: Transit routers (which just do IPv6 forwarding) are visible in traceroute output — they decrement Hop Limit and return ICMPv6 Time Exceeded with their source address. In MPLS, transit P routers are often invisible unless you use MPLS-specific traceroute. SRv6 gives you IP-level visibility for free.
Caveat: The outer IPv6 Source Address of SRv6 packets matters for OAM. When a transit node sends ICMPv6 back (e.g., Time Exceeded), the reply goes to the SA of the outer IPv6 header. If the SA isn't routable (e.g., a non-globally-reachable address), OAM replies will be lost. Most implementations use the ingress PE's loopback or locator address as the SA.

SRv6 OAM for SR-Policy Paths

To verify a specific SR-Policy path, you construct an ICMPv6 Echo Request with the policy's SRH (full segment list). The packet follows the exact path the policy specifies, and each hop's response confirms forwarding correctness. This is the SRv6 equivalent of MPLS LSP traceroute with a specific label stack.

For Flex-Algo paths, use the algo-specific locator SIDs in the OAM probe — the probe will follow the constrained topology of that algorithm.

Decoding uSID Containers in Captures

When micro-SIDs are in use, a single 128-bit SRH entry contains multiple packed uSIDs. Here's how to read them:

1
Identify the block prefix

The first 32 bits (4 bytes) of each segment list entry is the uSID block. It should match your domain's block (e.g., fcbb:bb01). If it doesn't, it's either a full-length SID or from a different domain.

2
Read uSIDs left-to-right after the block

Each 16-bit (2-byte) chunk after the block is a uSID. Read them left-to-right — the leftmost is the first to be processed (or the next to be processed if this is the DA).

Example: fcbb:bb01:0001:0100:0042:0000:0000:0000

→ Block: fcbb:bb01, uSID 1: 0001 (PE-1 uN), uSID 2: 0100 (P-1 uN), uSID 3: 0042 (PE-2 End.DT4), rest is zeros (end markers).

3
Zeros mean "end of container"

When you hit 0000, the remaining uSID slots in this container are empty. The node processing the last non-zero uSID will decrement Segments Left and move to the next SRH entry (if any).

What Is an SR-Policy?

An SR-Policy is a named tunnel that steers traffic through a specific path in the network using a segment list. It's the SRv6 equivalent of an RSVP-TE LSP or an SR-MPLS policy — but expressed as a list of SRv6 SIDs.

An SR-Policy is uniquely identified by a 3-tuple:

FieldDescriptionSR-MPLS Equivalent

Color is the key innovation: BGP routes carry an extended community called the Color Extended Community. When a PE receives a VPN route with a color, it automatically maps the traffic to the SR-Policy with that color + endpoint. This is how SRv6 achieves intent-based routing — you tag traffic with a color (intent) and the policy delivers it.

Each candidate path has a preference (higher = more preferred) and an origin (configured locally, from PCE, or from BGP). Only the highest-preference valid candidate path is active. Others are standby.

3
Segment Lists (weighted)

Each candidate path contains one or more segment lists (SID lists). Multiple segment lists can be used for weighted ECMP — traffic is distributed across them based on configurable weights.

4
Segments (SIDs)

Each segment list is an ordered list of SRv6 SIDs that define the explicit path. These get encoded into the SRH when traffic is steered into the policy.

Binding SID (BSID)

Every SR-Policy has an associated Binding SID (BSID) — a local SID that, when matched in the DA, causes the node to steer the packet into the policy's active segment list. This is how policies are stitched together across domains.

SR-MPLS BSID

A local MPLS label. When the headend receives a packet with this label, it pops it and pushes the policy's SID list. Used for Inter-AS stitching and nested policies.

SRv6 BSID (End.B6.Encaps)

An SRv6 SID (IPv6 address). When matched in the DA, the node encapsulates the packet with the policy's SRH. Same purpose: policy stitching and abstraction.

BSID as abstraction: A remote node only needs to know the BSID — it doesn't need to know the full SID list inside the policy. If the policy path changes (re-routed by PCE, new candidate path), the BSID stays the same. This is the same abstraction as MPLS Binding SIDs.

How Traffic Gets Steered into an SR-Policy

The "glue" that connects VPN routes to SR-Policies is the Color Extended Community + BGP next-hop resolution. Here's the chain:

1
Egress PE attaches Color Extended Community to VPN route

Example: VPN route 10.2.2.0/24 with Color = 100 (meaning "low-latency"), next-hop = PE-2.

2
Ingress PE receives the route and resolves the next-hop

PE-1 looks for an SR-Policy matching (endpoint = PE-2, color = 100). If found, the VPN route is steered into that policy.

3
SR-Policy provides the SRH encapsulation

The policy's active segment list (e.g., [P1-uN, PE-2-End.DT4]) becomes the SRH. Traffic matching the VPN route is encapsulated with outer IPv6 + this SRH and forwarded.

Fallback behavior: If no SR-Policy matches the (endpoint, color) tuple, the behavior depends on configuration. Common options: fall back to a policy with no color (best-effort IGP path), fall back to the default BGP next-hop resolution (just use the End.DT4 SID with no TE path), or drop the route (strict mode).
SR-MPLS comparison: In SR-MPLS, the same mechanism exists — Color Extended Community on VPN routes maps to SR-TE policies. The steering logic is identical. Only the encapsulation changes (MPLS label stack vs IPv6 SRH).

BGP Signaling for SRv6

The Problem: MPLS Labels Don't Fit in SRv6

In MPLS L3VPN, BGP carries a 3-byte VPN label in the MP_REACH_NLRI (RFC 4364). In SRv6, the "VPN label" is replaced by a 128-bit SRv6 SID. A 3-byte label field can't hold a 16-byte IPv6 address. So BGP needed a new mechanism.

Common Gotchas

Mistakes and surprises that catch SR-MPLS engineers moving to SRv6

1. SRH Segment List Is in Reverse Order

SL[0] is the last segment, SL[n] is the first. When reading packet captures, the bottom of the segment list is the first SID processed, and the top is the last. This is the opposite of MPLS where the top label is the first processed. Virtually every engineer gets confused by this at first.

2. Segments Left Off-by-One (H.Encaps vs H.Encaps.Red)

With H.Encaps, the first SID is duplicated in both the DA and SL[n], so SL = n. With H.Encaps.Red (the common variant), the first SID is only in the DA, so SL = n-1. If you see SL = 1 with 2 SIDs in the segment list, you're looking at H.Encaps. If SL = 1 with 1 SID in the list, it's H.Encaps.Red. Getting this wrong leads to incorrect path analysis.

3. "End" and "uN" Are Different IANA Behavior Codes

This guide (and most documentation) treats End and uN as synonymous. Functionally they're similar — both are node-level transit SIDs. But they have different IANA behavior codes:

BehaviorIANA CodeDefined In
End0x0001RFC 8986 — full-length SID (128-bit)
End with PSP0x0002RFC 8986
End with PSP, USP0x0004RFC 8986
uN (micro-SID Node)0x003Edraft-ietf-spring-srv6-srh-compression
uA (micro-SID Adjacency)0x003Fdraft-ietf-spring-srv6-srh-compression

In packet captures and BGP Prefix-SID attributes, the behavior code tells you whether you're looking at a full-length SID or a micro-SID. Don't assume they're interchangeable in debugging.

4. PSP ≠ PHP — Don't Apply MPLS Mental Models

PSP removes the SRH extension header; the outer IPv6 header stays with the final SID in the DA. PHP removes a label and exposes what's underneath. PSP is controlled by the penultimate node's SID flavor, not by the egress. See the PSP vs PHP section for the full breakdown.

5. SRH Is Optional for Single-SID Packets

If there's only one SID (e.g., End.DT4 with no SR-Policy), the SRH can be omitted entirely. The packet is just outer IPv6 (DA = SID) + inner payload. Many engineers expect SRH to always be present — it's not. Check for Next Header = 43 (Routing) to know if SRH exists.

6. uA SID Kills ECMP (Unless Bundle uA)

A per-link uA pins traffic to one physical link, even if there are multiple parallel links to the same neighbor. If you're troubleshooting congestion on a TI-LFA backup path that uses uA, this is likely the cause. Use uN for ECMP or verify if your platform supports bundle/parallel uA.

7. Nested SRH in TI-LFA Is Not Like Nested Labels

In SR-MPLS, TI-LFA pushes extra labels onto the same flat stack — the egress just pops more labels. In SRv6, TI-LFA creates a fully nested IPv6+SRH encapsulation (H.Encaps). The endpoint must decapsulate the entire outer layer, then process the inner layer. Some platforms don't handle this correctly, especially for non-SRv6 (native IPv6) prefixes.

8. Locator ≠ Loopback

In MPLS, the loopback address and the Node-SID prefix are often the same /32. In SRv6, the locator is a /48 (or /32) prefix — it's a range, not a single address. All SIDs on the node are addresses within this range. Don't confuse the locator prefix with the node's management loopback address — they're different things, though the loopback may be within the locator range.
