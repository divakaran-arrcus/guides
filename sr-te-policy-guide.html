SR-MPLS Recap

A quick refresher on the building blocks before we add traffic engineering

Core Concepts in 60 Seconds

Segment Routing MPLS encodes forwarding instructions as a stack of MPLS labels. Each label represents a segment — either a node to traverse or a specific link to use. The ingress router pushes the label stack; transit routers process only the top label.

Prefix SID (Node SID)

A globally unique index assigned to a router's loopback. Advertised via IGP (IS-IS or OSPF). Any router in the domain can compute a label for any prefix SID. Traffic follows the IGP shortest path to the owning node.

Scope: Global. PE1's prefix SID index 1 means label 16001 everywhere in the network (SRGB base 16000 + index 1).

Adjacency SID (Adj-SID)

A locally significant label assigned to a specific link (adjacency). Allocated dynamically by the router. Forces traffic out a specific interface — regardless of IGP shortest path. This is the key to traffic engineering.

Scope: Local. Adj-SID 24001 on PE1 means "use PE1's link to P1." That label is only meaningful at PE1.

Why Traffic Engineering?

When IGP shortest path is not enough — the case for SR-TE policies

The Problem: Shortest Path Is Not Always Optimal

IGP shortest-path forwarding is simple and robust, but it has a fundamental limitation: every packet to the same destination takes the same path. The network may have abundant alternate capacity, but IGP cannot use it unless the metrics are manually tuned — and metric tuning affects all traffic, not just the flows you want to move.

The dilemma: Both paths PE1→P1→P2→PE3 and PE1→P3→P4→PE3 have identical IGP metric (30). With ECMP, traffic may split across both. But what if you need all latency-sensitive traffic on the southern path (lower latency, 2ms/hop) and all bulk transfers on the northern path (lower TE metric)? IGP metrics alone cannot express this intent. If ECMP is not available or P1-P2 is congested, you have no lever to shift traffic to P3-P4 for specific flows.

Traffic engineering gives you the ability to steer specific traffic flows onto specific paths, independent of IGP shortest path. With SR-TE, this is done entirely at the headend by pushing a label stack — no signaling, no per-flow state on transit routers.

What Can't IGP Do?

Consider these real-world scenarios where IGP shortest-path forwarding falls short:

Use Case 1
Low-Latency Paths
Voice, video, and trading traffic need the lowest-latency path. IGP metric is typically based on bandwidth (or set to a constant), not latency. The shortest IGP path may traverse high-latency links. SR-TE can pin latency-sensitive flows to a path optimized for delay metric.
Use Case 2
Link/Node Avoidance
A link is experiencing errors, or a node is under maintenance. You want to move traffic away from it without shutting it down or changing IGP metrics (which would affect all traffic). SR-TE steers specific flows around the problem without global impact.
Use Case 3
Capacity Balancing
The northern path is 80% utilized while the southern path sits at 20%. ECMP doesn't help because ECMP splits by flow hash — it can't guarantee balanced distribution. SR-TE lets you explicitly move specific large flows (elephant flows) onto the underutilized path.
Use Case 4
Disjoint Paths
Two VPN customers each require a dedicated path with no shared links or nodes (SRLG-disjoint). IGP cannot guarantee disjointness. SR-TE can pin Customer A to the northern path and Customer B to the southern path, ensuring zero shared fate.

How SR-TE Solves This

Instead of changing IGP metrics (which is a blunt instrument affecting all traffic), SR-TE uses policies at the headend. Each policy defines a label stack that forces the packet through a specific sequence of nodes or links.

The top-level container, identified by the 3-tuple (headend, color, endpoint). It groups one or more candidate paths and is assigned a Binding SID (BSID).

2

Each candidate path has a preference value (higher = more preferred). Only the highest-preference valid candidate path is active at any time. Sources can be local config, PCE (via PCEP), or BGP SR-Policy.

3

Each candidate path contains one or more segment-lists. If multiple segment-lists exist, traffic is weighted ECMP across them based on the assigned weights. Each segment-list is an ordered sequence of segments (labels) that defines a specific path through the network.

4
Segments

The individual forwarding instructions — prefix SIDs, adjacency SIDs, or other segment types. These are the actual MPLS labels pushed onto the packet at the headend.

Binding SID (BSID)

Every SR policy is assigned a Binding SID — a local MPLS label that acts as a shorthand for the entire policy. Any packet arriving at the headend with the BSID as its top label gets steered into the policy (the BSID is popped and the policy's segment-list is pushed).

Why BSID Matters

  • Scalability: A remote headend can steer traffic into a policy by pushing a single BSID label instead of the full segment-list
  • Policy chaining: Multi-domain TE — each domain's headend uses BSID to hand off to the next domain's policy
  • Stability: The BSID stays constant even if the underlying segment-list changes (re-optimization)

Segment Types for SR-TE

The building blocks you use inside segment-lists — and when to use each one

SR-TE Segment Type Taxonomy

SR-TE policies define paths using typed segments. Each type specifies how the segment is resolved to an MPLS label. The types are defined in RFC 9256 and identified by a single letter.

TypeNameDescriptionUsage
Type A SR-MPLS Label A raw 32-bit MPLS label value. No validation against IGP — just push this label. Can represent any SID (Prefix, Adj, or BSID). Common
Type C IPv4 Prefix + SR Algorithm Specifies a prefix SID by IPv4 address and algorithm ID. The headend resolves the address to the correct label (SRGB base + index) for the specified algorithm (0 = SPF, 128+ = Flex-Algo). Most Common
Type D IPv6 Node + SR Algorithm Same as Type C but for IPv6-addressed nodes. Used in dual-stack or IPv6-only SR-MPLS networks. Rare
Type E IPv4 Adj + Interface Identifies an adjacency by its local and remote IPv4 interface addresses. The headend resolves these addresses to the corresponding Adj-SID label from the IGP database. Moderate
Type F IPv4 Adj + Link-Local Like Type E but uses link-local addresses for unnumbered interfaces. Resolves to an Adj-SID label. Uncommon
Type G IPv6 Adj (global) Identifies an adjacency by IPv6 global addresses. Resolves to the Adj-SID label for that adjacency. Rare
Type H IPv6 Adj (link-local) Identifies an adjacency by IPv6 link-local addresses. Resolves to the Adj-SID label. Rare
Note: Type B is an SRv6 SID type (128-bit IPv6 address) and is not used in SR-MPLS policies. Types I, J, and K are also SRv6-only.
The workhorses: In practice, Type C (prefix + algorithm) and Type E (adjacency by address) cover 90%+ of SR-MPLS deployments. Type C for loose hops (follow IGP to this node via a given algorithm), Type E for strict hops (use this exact link). Type A (raw label) is used for inter-domain segments or when the controller pushes pre-resolved labels.

Node SID vs Adjacency SID in Segment-Lists

Choosing between node SIDs and adjacency SIDs in your segment-list is the most important design decision in SR-TE. Each has distinct trade-offs.

Node SID (Prefix SID) — Type C

  • Traffic follows IGP shortest path to the node
  • TI-LFA protection is available — transit routers can compute backup paths
  • If a link along the path fails, traffic is automatically rerouted via the post-convergence path
  • Globally significant — same label on all routers
Protected by TI-LFA. Resilient to failures.

Adjacency SID — Type E

  • Traffic is pinned to a specific link
  • TI-LFA protection is NOT available by default — the Adj-SID means "use this link," so there is no alternate
  • If the pinned link fails, the segment becomes invalid (unless protected Adj-SID is used)
  • Locally significant — only meaningful on the allocating node
No TI-LFA protection. Link failure = traffic loss.
Design rule of thumb: Use node SIDs as waypoints to guide traffic through specific nodes while maintaining TI-LFA protection. Use adjacency SIDs only when you must pin to a specific link (e.g., to avoid a parallel link, or for SRLG-disjoint path requirements). The fewer adjacency SIDs in your segment-list, the more resilient your policy.

Protected vs Unprotected Adjacency SIDs

Routers advertise two flavors of adjacency SID in the IGP:

Unprotected Adj-SID (B-flag=0)

Strict pin to the link. If the link goes down, the packet is dropped. No backup behavior. This is what you use when you need absolute path determinism.

Protected Adj-SID (B-flag=1)

Prefers the link, but if the link fails, the router has a pre-computed TI-LFA backup installed for this Adj-SID. Traffic is rerouted to an alternate path that still reaches the neighbor. Best of both worlds — link preference with protection.

Practical note: Most routers advertise both a protected and an unprotected Adj-SID for each link. The SR-TE controller or headend config should explicitly choose which to use based on the SLA requirements. When in doubt, use the protected variant.

Static SR Policies

Manually specifying every segment — maximum control, minimum automation

When to Use Static Policies

A static SR policy is one where you manually define every SID in the segment-list. The headend does not compute the path — you tell it exactly which nodes and links to traverse. This is the simplest form of SR-TE, and it maps directly to how you might think about explicit-path RSVP-TE LSPs.

Small Networks
When the topology is small enough to reason about manually (fewer than ~20 nodes), static policies are manageable and straightforward.
Specific Override Paths
When you need a very precise path that no dynamic algorithm would naturally compute — e.g., a maintenance bypass or a path that violates normal routing metrics.
Lab & Testing
Static policies let you validate forwarding behavior hop-by-hop. Ideal for proof-of-concept and troubleshooting because every segment is explicit.
Last-Resort Backup
A low-preference static candidate path can serve as an emergency fallback if all dynamic paths fail — a "known-good" path hardcoded by the operator.
Operational reality: Static policies do not adapt to topology changes. If a link or node in your segment-list goes down, the policy becomes invalid. There is no automatic re-computation. You must either pre-provision backup candidate paths or accept a failover gap. For production at scale, dynamic paths (covered in Section 8) are strongly preferred.

Configuration Concept

A static policy consists of three elements: the policy identity (headend, endpoint, color), one or more candidate paths, and within each candidate path, one or more explicit segment-lists where every SID is specified by the operator.

1
Define the segment-list

Create a named segment-list containing an ordered sequence of SIDs. Each SID can be a Prefix-SID (node SID), an Adjacency-SID, or a Binding-SID. The order matters — the first SID is the first hop, the last SID is the final destination.

2
Attach the segment-list to a candidate path

Associate the segment-list with a candidate path inside the SR policy. Give the candidate path a preference value (higher = more preferred). You can attach multiple segment-lists to a single candidate path for weighted ECMP.

3
The headend validates and installs

The headend verifies that each SID in the segment-list is reachable (present in the LSDB/TED). If all SIDs are valid, the policy is installed in the forwarding table. If any SID is unreachable, the candidate path is marked invalid.

Segment-List Deep Dive

How the headend builds the MPLS label stack and what happens at every hop along the path

From Segment-List to Label Stack

The segment-list is an ordered sequence of SIDs that defines the path. The headend converts this into an MPLS label stack by mapping each SID to a label and stacking them in order. The key rule:

Stack ordering rule: The first segment in the list becomes the top label in the stack. The last segment becomes the bottom label (with the S-bit set). Each transit node processes and removes the top label, revealing the next instruction underneath. Think of it as a stack of instructions — the top one is executed first.

Label Operations at Each Hop

Three fundamental MPLS label operations drive SR forwarding. If you know MPLS, this is familiar — SR uses the same data-plane mechanics:

PUSH (Headend)
The headend pushes the entire label stack onto the packet. All labels are imposed at once — no signaling needed because the headend has the full segment-list.
SWAP (Transit — Prefix SID)
A transit node that is NOT the destination of the top label swaps it with the next-hop's label for the same Prefix SID. Standard MPLS label switching — the node forwards toward the Prefix SID destination.

The node that owns the top label pops it, exposing the next label. For Adjacency SIDs, the owning node pops the label and forwards out the specified interface. For Prefix SIDs, PHP pops the label one hop before the destination.

Mixed SID Types: ECMP Behavior

A critical operational distinction: the type of SID in your segment-list directly affects load balancing.

Node SID (Prefix SID)

Forwarded via IGP best-path. If multiple equal-cost paths exist to that node, traffic is load-balanced across all ECMP paths. The SID says "reach this node" without specifying how.

ECMP — uses all available parallel paths

Adjacency SID

Pinned to a specific link (or link bundle). Even if multiple parallel links exist between two nodes, the Adj-SID forces traffic onto one specific link. This is the whole point — precise path control.

No ECMP — single link only

Dynamic Paths

Let the headend compute the optimal path based on constraints and real-time topology

Headend-Computed Dynamic Paths

With dynamic path computation, you do not specify individual SIDs. Instead, you define an optimization objective (which metric to minimize) and optional constraints (which links/nodes to include or avoid). The headend router uses its local Traffic Engineering Database (TED) — built from IGP (IS-IS or OSPF with TE extensions) — to run a constrained shortest-path computation and produce a segment-list automatically.

1
TED is populated from IGP TE extensions

IS-IS or OSPF floods TE attributes for every link: TE metric, latency, available bandwidth, admin-groups (affinities), SRLGs. The headend builds a complete graph with these attributes.

2
Policy specifies: dynamic with metric-type

Instead of a static segment-list, the candidate path is configured as dynamic with a metric type (IGP, TE, latency, or hop-count). Optionally, constraints are added (affinities, disjointness, bandwidth).

3
Headend runs CSPF (Constrained SPF)

The headend computes the shortest path in the TED graph that satisfies all constraints, optimizing for the selected metric. The result is a set of SIDs (typically node SIDs for the waypoints along the computed path).

4
Segment-list auto-generated and installed

The computed path is translated into a segment-list. If the topology changes (link failure, metric change), the headend automatically re-computes and updates the segment-list. No operator intervention needed.

Affinities (also called administrative groups or link colors) are 32-bit bitmasks assigned to links. They let you tag links with properties (e.g., "submarine cable," "encrypted," "high-capacity") and then write policy constraints that include or exclude links based on these tags.

Disjointness Constraints

Disjointness ensures that two or more SR policies use completely separate paths, so a single failure cannot affect both. This is essential for protecting primary/backup policy pairs.

Link-Disjoint
Primary and backup paths share no common links. A single link failure affects at most one path. Nodes may be shared.
Node-Disjoint
Primary and backup paths share no common transit nodes (and therefore no common links). Stronger than link-disjoint — protects against entire node failures.
SRLG-Disjoint
Primary and backup paths share no common Shared Risk Link Group. Protects against correlated failures — e.g., two links in the same fiber conduit both failing when the conduit is cut.

Bandwidth & Margin Constraints

Bandwidth

Ensures every link on the computed path has at least the specified available bandwidth. The headend checks the "unreserved bandwidth" attribute from the TED. If a link cannot satisfy the bandwidth requirement, it is excluded from the computation.

Use case: guaranteed-bandwidth VPN services

Margin

Adds slack to the computed metric before accepting the path. Can be absolute (e.g., path cost must be within 10 of optimal) or relative (e.g., within 20% of optimal). Allows sub-optimal but "good enough" paths, increasing path diversity.

Use case: avoid flapping between near-equal paths
Important caveat: SR-TE bandwidth constraints are not enforced in the data plane. Unlike RSVP-TE, SR does not have per-hop admission control or bandwidth reservation. The bandwidth constraint is checked during path computation against the TED's advertised values, but actual traffic policing must be done separately (e.g., QoS policies). Think of it as "bandwidth-aware routing" rather than "bandwidth-guaranteed routing."

Weighted ECMP & Segment-Lists

Multiple paths within a single candidate path — controlled traffic distribution by weight

Multiple Segment-Lists in One Candidate Path

An SR-TE candidate path can contain more than one segment-list. When multiple segment-lists are present and valid, the headend distributes traffic across them. Each segment-list has an associated weight that controls the proportion of traffic it carries.

This is conceptually similar to having multiple ECMP next-hops for a route, but with explicit control over the ratio. Standard ECMP divides traffic equally — weighted ECMP lets you send 75% on one path and 25% on another.

Not to be confused with candidate paths: Multiple candidate paths provide redundancy — only the highest-preference valid path is active. Multiple segment-lists within one candidate path provide load balancing — all valid segment-lists are active simultaneously.

Use Cases for Weighted ECMP

Gradual Traffic Migration
Moving traffic from an old path to a new path incrementally. Start at 10:90, shift to 50:50, then 90:10, then 100:0. Minimizes risk by validating the new path under partial load before full cutover.
Capacity Optimization
When north and south paths have different link capacities (e.g., north = 100G, south = 40G), use weights proportional to capacity: weight 100 for north, weight 40 for south. Fills both paths evenly relative to their capacity.
A/B Testing & Canary Paths
Send 5% of traffic on a new or experimental path (weight 1) while keeping 95% on the proven path (weight 19). Monitor error rates and latency on the canary before committing.
Cost-Based Routing
Prefer cheaper transit paths but overflow to expensive paths under load. Assign higher weight to the cheaper path and lower weight to the expensive path.

Traffic share for SL-n = weight(SL-n) / sum(all weights)

On-Demand Nexthop (ODN)

Automatic SR policy creation triggered by BGP color communities — the production deployment model

The Problem ODN Solves

Without ODN, every SR-TE policy must be manually configured on the headend. In a network with 100 PE routers and 5 color/SLA classes, that could mean 500 policies per headend — configured, maintained, and updated by hand. This approach does not scale.

ODN flips the model: instead of pre-configuring policies, you define templates that describe how to build a policy for a given color. When a BGP route arrives with a color extended community, the headend automatically instantiates an SR policy using the matching template. No per-destination, per-policy configuration required.

Without ODN

100 PEs x 5 colors = 500 policies per headend. Each one manually configured with endpoint, color, and path type. Adding a new PE means touching every other headend's config.

O(n x m) manual config: n endpoints, m colors

With ODN

5 ODN templates (one per color). Policies auto-created when BGP routes arrive. Adding a new PE requires zero config changes on existing headends — BGP does the work.

O(m) config: m templates, regardless of endpoint count

PCE Architecture

Centralised path computation for SR-TE — the Path Computation Element and its role in the network

Why a Centralised Path Computer?

A headend router running CSPF sees only its own IGP link-state database. That is sufficient for single-area, single-metric optimization. But three real-world scenarios break the local-computation model:

  • Multi-domain / multi-area: The headend does not have link-state visibility beyond its own IGP area or AS boundary.
  • Global bandwidth optimization: Each headend optimizes independently, leading to suboptimal aggregate bandwidth placement. Only a controller with a network-wide view can compute globally optimal paths.
  • Complex constraints: Disjoint path pairs, bandwidth calendaring, and multi-layer coordination require computation beyond what a router's embedded CSPF can offer.

The Path Computation Element (PCE) solves all three by centralizing path computation into a dedicated controller that maintains a global Traffic Engineering Database (TED).

Key point: The PCE computes paths — it does NOT forward packets. Data-plane forwarding remains distributed on the routers. This separation of computation from forwarding is fundamental to the PCE architecture.

PCE Components

PCE (Path Computation Element)

The external controller/server that computes SR-TE paths. It collects topology via BGP-LS, maintains a global TED, runs constrained shortest-path algorithms, and pushes computed paths to headend routers via PCEP.

Can be a dedicated appliance (e.g., Cisco XTC, Juniper Paragon Pathfinder) or an SDN controller with PCE functionality.

PCC (Path Computation Client)

The headend router that requests or receives computed paths. Each PCC maintains a PCEP session to one or more PCEs. The PCC installs the computed segment-list in its forwarding plane and reports policy status back to the PCE.

Every SR-TE headend router is a PCC. Transit and tail-end routers are NOT PCCs — they have no PCEP session.

Topology Collection: BGP-LS

The PCE needs a complete view of the network topology, including TE metrics, latency, bandwidth, SRGBs, prefix SIDs, adjacency SIDs, and Flex-Algo definitions. BGP Link-State (BGP-LS) is the standard mechanism for exporting this information from the IGP domain to the PCE.

1
IGP floods link-state information

IS-IS or OSPF distributes topology, TE attributes, and SR information (prefix SIDs, adj SIDs, SRGB ranges, Flex-Algo FADs) within each area.

2
BGP-LS exporter sends topology to PCE

One or more routers in each area are configured as BGP-LS exporters. They convert IGP LSDB entries into BGP-LS NLRIs and advertise them to the PCE via a BGP session. The PCE receives node, link, and prefix TLVs with all TE extensions.

3
PCE builds global TED

The PCE aggregates BGP-LS feeds from all areas/ASes into a single unified Traffic Engineering Database. This global TED is the input for all path computations. Updates arrive in near-real-time as IGP converges.

PCE Operation Modes

The PCE can operate in two fundamentally different modes — and both can be active simultaneously on the same PCE.

Stateless PCE

Computes paths on request (PCRequest/PCReply). Does not track active LSPs or policies. Each computation is independent — the PCE does not consider previously computed paths when handling new requests. Simple but cannot do global optimization.

Maintains a database of all active SR-TE policies (the LSP-DB). PCCs report policy state via PCReport messages. The PCE can optimize globally — it knows what paths are already in use and can redistribute traffic. Can also initiate new policies (PCE-initiated mode).

When to Use Headend Computation

Single IGP Domain
All destinations reachable within one IGP area. Headend has full topology visibility. No need for a PCE unless you want global bandwidth optimization.
Simple Constraints
Metric-type optimization (min latency, min TE metric) or affinity-based path selection. Headend CSPF handles these natively.
Maximum Resilience
No dependency on external controller. If the router is up, it can compute and install paths. Ideal for networks where PCE HA is not yet deployed.
ODN + headend CSPF is the most common starting point for SR-TE deployments. It provides automated policy creation with dynamic path computation — all without an external controller. Graduate to PCE when you hit the limits of local computation.

When to Use PCE

Multi-Domain
Paths that cross IGP area boundaries or AS boundaries. The headend cannot compute these — it lacks topology visibility beyond its own area.
Bandwidth Optimization
When you need to optimally place hundreds of policies across shared links without over-subscription. The PCE's global view enables bin-packing and load balancing.
Disjoint Paths
Working + protect path pairs that must be node-disjoint or link-disjoint. Computing truly disjoint paths requires knowledge of BOTH paths simultaneously — only a PCE can do this.
Centralized Policy
When network operations wants a single point of control for all SR-TE policies. PCE-initiated mode lets the controller push policies to any PCC without touching router config.

Coexistence: Local + PCE Together

In production, headend and PCE computation typically coexist on the same router. SR-TE's candidate path preference mechanism determines which path wins:

1
Multiple candidate paths per policy

A single SR-TE policy (identified by headend, color, endpoint) can have multiple candidate paths from different sources: static, dynamic (headend-computed), and PCE-computed. Each candidate path has a preference value (higher = preferred).

2
Preference-based selection

The headend selects the candidate path with the highest preference as the active path. If that path fails (e.g., segment unreachable), it falls back to the next-highest preference.

3
Typical preference hierarchy

PCE-computed paths often get higher preference (e.g., 200) than headend-computed (e.g., 100) and static (e.g., 50). If the PCEP session to the PCE fails, the headend-computed candidate path takes over automatically.

PCE failure resilience: If the PCEP session drops, the PCC retains the PCE-computed path for a configurable delegation timeout. After timeout, the PCC marks the PCE candidate path as invalid and falls back to the next-best candidate. This ensures traffic is never black-holed due to PCE failure — the headend always has a locally computed backup.
PCEP Protocol Details

The Path Computation Element Protocol — messages, objects, and SR-TE extensions

PCEP Overview

PCEP (RFC 5440) is the protocol between PCE and PCC. It runs over TCP (port 4189) and supports TLS (RFC 8253) for encryption. Stateful PCE extensions (RFC 8231) enable delegation and PCUpdate; PCE-Initiated capability (RFC 8281) adds PCInitiate for creating/deleting policies. The SR-TE extensions (RFC 8664) add segment-list encoding to PCEP messages, enabling the PCE to push SR label stacks to the headend.

TCP-Based
Reliable transport over TCP port 4189. Session keepalives via Keepalive messages. Dead-timer for session failure detection (typically 120s, with 30s keepalive).
Session-Oriented
Each PCC maintains a persistent PCEP session to the PCE. Session capabilities (stateful, SR, initiated) are negotiated during the Open handshake.
TLV-Extensible
All PCEP objects use a TLV (Type-Length-Value) structure, making the protocol easily extensible. SR-TE, Flex-Algo, and association groups are all TLV extensions.

2
LSP-DB Synchronization

After session establishment, the PCC sends PCReport messages for ALL existing SR-TE policies — both locally configured and previously PCE-initiated. This synchronizes the PCE's LSP-DB with the PCC's actual forwarding state. The sync is bounded by an end-of-sync marker.

Full sync on every reconnect: If the PCEP session flaps, the PCC re-sends all PCReports. The PCE rebuilds its LSP-DB for that PCC from scratch. This ensures consistency even after network partitions.
3

During normal operation, the PCC sends PCReports on any policy state change (path change, failure, BSID change). The PCE sends PCUpdates to modify delegated policies or PCInitiates to create/delete policies. Keepalive messages maintain the session.

4
Session Failure Handling

If keepalives stop arriving, the dead-timer expires and the session is torn down. The PCC starts a delegation cleanup timer — during this period, PCE-computed paths remain active. After the timer expires, the PCC reverts delegated policies to locally computed paths (if available) or marks PCE-initiated policies as orphaned.

PCE High Availability

In production, PCE is a critical control-plane component. A single PCE failure should not disrupt SR-TE policy forwarding. Standard HA patterns include:

Active/Standby PCE

PCC connects to two PCEs. One is primary (active), the other is standby. If the active PCE fails, the PCC switches delegation to the standby. The standby PCE's LSP-DB is kept in sync via PCReports from all PCCs.

Active/Active PCE

Both PCEs are active and can compute independently. Policies are partitioned or load-balanced across PCEs. Requires inter-PCE state synchronization to avoid conflicting path assignments. More complex but higher throughput.

Graceful degradation: Even with both PCEs down, the PCC retains previously computed paths for the delegation timeout period. After that, locally computed (headend CSPF) candidate paths take over. This layered resilience — PCE primary, headend fallback — means SR-TE traffic is never black-holed due to PCE infrastructure failure.

Traffic Steering

How traffic finds its way into SR-Policies — automated steering, color communities, and per-flow control

Automated Steering (AS) — The Primary Mechanism

Automated Steering is the production-standard method for directing traffic into SR-Policies. The idea is simple: BGP routes carry a color extended community, and the headend router automatically maps that color to a matching SR-Policy. No static routes, no tunnel interfaces, no manual next-hop configuration.

Color-Only (CO) Bits — Fallback Behavior

The BGP color extended community includes two CO (Color-Only) bits that control what happens when no SR-Policy with the matching color exists at the headend. This is critical for production resilience — you need to decide: should traffic be dropped, use the default path, or try an alternate policy?

CO BitsBehavior When No Matching PolicyUse Case
CO=00 Fall back to the default BGP path (IGP shortest path to the endpoint). Traffic is not dropped. Best-effort traffic that benefits from TE but must not be blackholed. Most common default.
CO=01 Drop traffic. If no matching SR-Policy exists, the prefix is unreachable. Strict SLA traffic that MUST follow a TE path — better to drop than violate the SLA (e.g., financial trading, real-time control).
CO=10 Fall back to any available SR-Policy to the same endpoint, regardless of color. Traffic that needs some form of TE steering but is flexible about which specific policy. Useful during migration or multi-color deployments.

CO=00 (safe fallback)

BGP route for 10.3.0.0/24 has color=200 but PE1 has no policy with color=200 to PE3. PE1 falls back to IGP shortest path to PE3. Traffic still flows — just without TE optimization. This is the safest production default.

CO=01 (strict drop)

Same scenario — but with CO=01, PE1 has no matching policy, so the prefix 10.3.0.0/24 is unreachable. Traffic is dropped. Use only when SLA violation is worse than packet loss.

Operational caution: CO=01 (strict) can cause unexpected blackholes if an SR-Policy goes down or is misconfigured. Always have monitoring in place before deploying strict CO bits. Most operators start with CO=00 and only move to CO=01 for specific high-value prefixes.

BSID & Policy Chaining

Using Binding SIDs to abstract SR-Policies into single labels — and chaining policies across domain boundaries

Binding SID Mechanics — Deep Dive

A Binding SID (BSID) is a local MPLS label allocated by the headend router to represent an entire SR-Policy. When a packet arrives at the headend with the BSID as the top label, the router performs two operations in sequence:

POP
Pop the BSID label from the stack

The BSID itself is removed. It served its purpose: identifying which SR-Policy to use.

PUSH
Push the active candidate path's segment list

The SR-Policy's current active segment list is pushed onto the remaining label stack. The packet is now forwarded according to the SR-Policy's path.

Why BSID Exists — The Abstraction Problem

Without BSID, any router that wants to steer traffic through another router's SR-Policy must know the full segment list of that policy. This creates two serious problems:

Without BSID

  • Remote router must know the full segment list of every policy it references
  • If the policy's path changes (failover, recompute), all remote routers must be updated
  • Label stack depth explodes: local labels + remote policy's labels + service labels
  • Cross-domain paths require knowledge of internal topology in other domains
  • No abstraction — tight coupling between domains

With BSID

  • Remote router pushes 1 label (BSID) — done
  • If the policy's path changes, only the headend's forwarding table is updated. BSID stays the same.
  • Label stack depth is minimal: 1 BSID per domain traversed
  • Cross-domain paths require only BSID knowledge — no internal topology leaking
  • Clean abstraction — loose coupling between domains
Think of BSID as a function call: You call the function (push BSID), the headend executes the implementation (expands to segment list), and you don't need to know how the function works internally. If the implementation changes (different path), your call still works.

Flex-Algo + SR-TE

Two complementary tools — distributed topology planes meet per-flow path engineering

Flex-Algo Alone: Distributed Topology Planes

Flex-Algo (algorithm IDs 128-255) lets you define independent forwarding topologies on the same physical network. A Flex-Algo Definition (FAD) specifies a metric type (IGP, delay, or TE metric) and optional link constraints (affinity include/exclude). Every participating node runs a separate SPF for each algorithm and installs per-algo Prefix-SIDs in its LFIB.

Strengths
Simple to deploy — define the FAD once, all nodes compute consistently. No per-flow config. One label (the algo-specific Prefix-SID) encodes both destination AND path intent. Scales to every prefix in the network automatically.
Limitations
Paths must be SPF-computable — you cannot create arbitrary waypoint paths. Every node in the algo sees the same topology, so you cannot engineer per-flow deviations. If the SPF-optimal path for algo 128 is PE1-P3-P4-PE3, every source in the network uses that same path to PE3. No per-headend customization.

SR-TE policies give the headend full control over path selection by specifying an ordered segment-list. Strengths
Maximum flexibility — any path, any constraint. Per-headend, per-flow granularity. Supports disjoint paths, bandwidth constraints (via PCE), affinity, and arbitrary waypoints. Can be computed locally, by PCE, or programmed via PCEP/NETCONF.
Limitations
Requires per-policy configuration (or PCE delegation). Explicit Adj-SID paths grow the label stack — one label per strict hop. A fully strict path across 5 hops needs 5 labels, which may approach hardware imposition limits. Does not scale as a network-wide topology mechanism.

Extending traffic engineering across IGP domain boundaries — the multi-domain challenge

The Inter-Domain Challenge

SR-TE works beautifully within a single IGP domain because the headend (or its PCE) has full visibility of the topology via IGP (ISIS/OSPF) and BGP-LS. But real SP networks span multiple IGP domains — different areas, different ASes, different administrative zones. The headend in Domain A cannot see the internal topology of Domain B or Domain C.

Reference: 3-Domain Topology

PE1 wants to steer traffic to PE3 along a specific engineered path. But PE1's IGP only knows Domain A's topology. It has no knowledge of P5, P6, P7, or the links between them. The Prefix-SIDs of Domain B and Domain C nodes are not in PE1's SRGB-resolved LFIB.

The core problem: Segment Routing labels are resolved via the IGP. If a node's Prefix-SID is not in your IGP domain, you cannot push its label — you don't know the label exists, and your LFIB has no entry for it. Inter-domain SR-TE requires mechanisms to bridge this visibility gap.

Solution 1: BSID Stitching

The simplest inter-domain approach. Each domain runs its own SR-TE policy for its segment of the end-to-end path. The policies are chained together using Binding SIDs (BSIDs) at domain boundaries.

1

ASBR4 has full visibility of Domain C. It creates an SR-TE policy: endpoint=PE3, segment-list=[16003]. BSID=40003 is allocated for this policy. ASBR4 advertises BSID 40003 via BGP to its peers.

2
Domain B: ASBR2 creates SR-Policy to ASBR3, stitches to Domain C

ASBR2 has visibility of Domain B. It creates an SR-TE policy: segment-list=[path-through-B, Adj-SID to ASBR3, BSID 40003]. The BSID 40003 from Domain C becomes a segment in Domain B's policy. ASBR2 allocates its own BSID=40002 for the combined B+C path.

3
Domain A: PE1 creates SR-Policy to ASBR1, stitches to Domain B

PE1 sees Domain A's topology. It creates an SR-TE policy: segment-list=[path-through-A, Adj-SID to ASBR1, BSID 40002]. PE1 pushes this label stack, and each domain boundary expands its BSID into the local segment-list.

A parent PCE sits above domain-specific child PCEs. Each child PCE has full visibility of its domain (via BGP-LS from that domain's nodes). The parent PCE knows the domain interconnections and coordinates end-to-end path computation.

Solution 3: BGP-LU + SR-TE

BGP Labeled Unicast (BGP-LU) provides inter-domain label stitching at the BGP level. Each ASBR allocates a local label for remote prefixes and advertises it via BGP. Within each domain, SR-TE handles intra-domain path engineering.

How it works: PE3's loopback (10.0.0.3) is advertised via BGP-LU across domain boundaries. Each ASBR along the way allocates a per-next-hop label. The BGP-LU label at each boundary stitches to the next domain's SR transport. Intra-domain, traffic follows SR-TE policies normally.

ODN + Color: Automated VPN Steering

On-Demand Next-Hop (ODN) is the mechanism that makes SR-TE + VPN scale to thousands of services without per-VPN policy configuration. Here is how the pieces fit together:

1
PE3 advertises VPN route with Color Extended Community

PE3 originates a BGP VPNv4 route for 10.3.1.0/24 with next-hop=10.0.0.3, VPN label=100100, RT=1:100, and Color=100. The color encodes intent — for example, color 100 = "low-latency service." The color is attached via a BGP Extended Community (Type 0x030b).

2
PE1 receives the route and triggers ODN

PE1 imports the VPNv4 route into VRF CUSTOMER-A. It sees color=100 on the route. PE1 checks: "Do I have an SR-TE policy matching (color=100, endpoint=10.0.0.3)?" If not, and an ODN template for color 100 exists, PE1 automatically creates the SR-TE policy.

3
ODN template computes or delegates the path

The ODN template for color 100 specifies how to compute the path. Options: dynamic computation with metric=latency (local CSPF), delegate to PCE, or use sid-algorithm 128 (Flex-Algo). The resulting segment-list is installed as the SR-TE policy's candidate path.

4
BGP resolves VPN next-hop via the SR-TE policy

The VPN route's next-hop (10.0.0.3 with color 100) now resolves via the SR-TE policy instead of the default IGP/SR path. All VPN traffic for 10.3.1.0/24 is steered through the engineered path. If the policy goes down, traffic falls back to the default SR path (or drops, depending on configuration).

Migration from LDP

LDP to SR-MPLS to SR-TE — an incremental journey with no flag day

The Migration Path

Migrating from LDP to SR-TE is not a single event — it is a three-phase progression where each phase adds capability without breaking the previous one. At every step, traffic continues to flow. No forklift upgrades. No flag days. Each phase is independently reversible.

Migration Timeline
Phase 1: LDP + SR Phase 2: SR Only Phase 3: SR + SR-TE
Coexistence LDP removed Traffic engineering
Key principle: Each phase is incremental and backwards-compatible. You can stay at Phase 1 indefinitely if needed. Phase 2 requires all nodes to support SR. Phase 3 can be deployed on a subset of headend PEs while the rest continue with plain SR. There is no big-bang cutover at any point.

Phase 1: LDP + SR Coexistence

Enable SR on all capable nodes while LDP continues to run. Both label distribution mechanisms operate simultaneously. This is the safe starting point — zero traffic impact, full rollback capability.

1
Enable SR on IGP (ISIS or OSPF)

Configure Segment Routing on the IGP process and assign Prefix-SIDs to loopbacks. The IGP begins advertising SR extensions (ISIS TLV 135 sub-TLV 1, or OSPF Extended Prefix LSA). LDP continues to run on all interfaces. Both SR and LDP labels are installed in the LFIB.

2
SR Preference: Configure SR labels to be preferred over LDP

When both SR and LDP have a label for the same prefix, the router must choose one for forwarding. Configure SR-prefer so that SR labels take priority. LDP labels remain in the LFIB as backup but are not used for forwarding when an SR label exists. This is the default on most modern platforms.

3
Mapping Server for Non-SR Nodes

If some nodes cannot run SR (older hardware/software), deploy an SR Mapping Server. The mapping server advertises Prefix-SID mappings for LDP-only nodes' loopbacks. SR-capable nodes install these mapped SIDs in their LFIB. LDP-only nodes continue to use LDP labels. The mapping server bridges the gap — SR nodes can reach LDP-only nodes via mapped SIDs, and LDP-only nodes can reach SR nodes via LDP.

4
Verify: Both Label Sets Active

Validate that the LFIB contains both SR and LDP entries. Verify that SR labels are preferred for forwarding (active). Confirm that VPN services (L3VPN, L2VPN) continue to work — the VPN label (inner) is unchanged; only the outer transport label source changes from LDP to SR.

Phase 2: Remove LDP

Once all nodes in the network support SR (either natively or via mapping server for legacy), LDP can be removed. This phase eliminates the LDP control plane entirely.

1
Verify SR coverage is complete

Confirm that every prefix reachable via LDP also has an SR label (either native Prefix-SID or mapping-server-derived). Audit the LFIB on every node. Any prefix with only an LDP label and no SR label will become unreachable when LDP is disabled. This is the critical pre-check.

2
Enable TI-LFA on all links

LDP relied on LFA (Loop-Free Alternate) and rLFA (remote LFA) for fast reroute. SR replaces these with TI-LFA (Topology-Independent LFA), which provides 100% coverage for any topology. Enable TI-LFA on all interfaces before removing LDP to ensure sub-50ms protection is maintained.

3
Disable LDP — node by node or region by region

Remove LDP configuration from each node. Since SR-prefer was active in Phase 1, traffic was already forwarding via SR labels. Removing LDP removes only the backup LDP labels from the LFIB. No traffic impact if Phase 1 was correctly validated.

4
Remove mapping server entries for upgraded nodes

Once all legacy nodes have been upgraded to native SR, remove their entries from the mapping server. Eventually, decommission the mapping server entirely. The network is now pure SR-MPLS.

What you gain by removing LDP: No more LDP sessions (TCP connections between every LDP peer), no more LDP LIB (Label Information Base — one entry per prefix per neighbor), no more LDP-IGP sync timers, no more targeted LDP sessions for rLFA. The control plane becomes dramatically simpler. The IGP alone distributes all label information.

Phase 3: Add SR-TE

With a clean SR-MPLS underlay in place, you can now deploy traffic engineering. This phase is entirely additive — it does not change the base SR forwarding at all.

1
Deploy Flex-Algo topologies

Define FADs for your SLA intents (e.g., Algo 128 = min-delay, Algo 129 = disjoint-A). Configure participation on all nodes. Assign algo-specific Prefix-SID indices. The network now has multiple forwarding planes. No impact on existing algo-0 traffic.

2
Configure ODN templates on headend PEs

Define ODN templates for each color (SLA class). Map colors to computation strategies: sid-algorithm for Flex-Algo, dynamic with PCE delegation for complex constraints. ODN templates are dormant until a VPN route with a matching color arrives.

3
Deploy PCE (optional but recommended)

For dynamic path computation, bandwidth-aware routing, and inter-domain SR-TE, deploy a PCE. Configure PCEP sessions from headend PEs to the PCE. ODN can delegate path computation to PCE. PCE provides centralized optimization that complements Flex-Algo's distributed approach.

4
Start steering VPN traffic via color communities

Attach Color Extended Communities to VPN routes at the egress PEs (or via route-policy). As soon as a colored VPN route arrives at a headend PE with a matching ODN template, the SR-TE policy is created automatically and traffic is steered. Start with non-critical VPN services, validate, then expand.

Incremental adoption: You do not need to enable SR-TE on every PE at once. Start with one pair of PEs for a pilot VPN. Add more PEs and colors as confidence grows. PEs without ODN templates simply ignore color communities and forward via default SR — no harm done.

Common Migration Pitfalls

SRGB Mismatch
All nodes must use the same SRGB range (e.g., 16000-23999). If one node has a different SRGB, label 16003 means different things on different nodes. Verify SRGB consistency across the entire network before Phase 1. Most platforms default to 16000-23999 but older software may differ.
Missing TI-LFA Before LDP Removal
LDP's rLFA provides FRR for non-ECMP topologies. If you remove LDP without enabling TI-LFA, you lose fast reroute protection. Always enable and validate TI-LFA coverage before Phase 2. Use "show isis fast-reroute" to confirm 100% prefix coverage.
Mapping Server Gaps
If a legacy node's prefix is missing from the mapping server, SR nodes cannot reach it via SR. Traffic falls back to LDP (Phase 1) or is dropped (Phase 2). Audit the mapping server entries against the full IGP prefix table. Automate the mapping if possible.
LDP-IGP Sync During Transition
During Phase 1, LDP-IGP synchronization is still active. If LDP goes down on a link but SR is working, the link may be penalized in IGP (max-metric). Consider disabling LDP-IGP sync on links where SR is confirmed working, or ensure SR-prefer is active so the LDP state does not affect SR forwarding.
Prefix-SID Index Conflicts
Each Prefix-SID index must be globally unique within the SR domain. During migration, as you assign indices to hundreds of loopbacks, conflicts can cause black-holing. Maintain a central SID index registry. Use automation to assign and validate indices.
Rollback at any point: Phase 1 rollback = disable SR on IGP, remove Prefix-SIDs. LDP continues as before. Phase 2 rollback = re-enable LDP on all nodes. Phase 3 rollback = remove ODN templates and SR-TE policies. Base SR forwarding continues. Each phase is independently reversible with no traffic loss during rollback if the previous phase's infrastructure is still in place.
Protection & Convergence

How SR-TE policies survive failures — from sub-50ms TI-LFA repair to PCE re-optimization

TI-LFA + SR-TE Interaction

TI-LFA and SR-TE work at different layers of protection. TI-LFA protects individual segments within a segment-list, while SR-TE policy mechanisms protect the policy as a whole. Understanding where each kicks in is critical for designing resilient networks.

When an SR-TE policy steers traffic via a segment-list like [16011, 16012, 16003] (PE1→P1→P2→PE3), TI-LFA independently protects each segment hop. If the P1→P2 link fails, P1's TI-LFA repair kicks in — P1 pushes a repair label stack to detour around the failed link. The SR-TE policy headend (PE1) does not even know the failure happened. Traffic continues flowing with sub-50ms interruption.

TI-LFA Scope (per-segment)

  • Protects each individual segment hop in the segment-list
  • Repair happens at the point of local failure (PLR)
  • Headend is unaware — no policy state change
  • Sub-50ms with BFD detection
  • Pre-computed repair label stack in FIB

SR-TE Policy Scope (end-to-end)

  • Protects the overall path intent (candidate path failover)
  • Headend detects when a segment-list or candidate path is invalid
  • Failover to next candidate path (seconds)
  • PCE re-optimization for new optimal path (seconds to minutes)
  • Requires IGP convergence or PCE notification
Key insight: TI-LFA is your first responder. It handles single failures instantly and invisibly. SR-TE policy failover is your strategic response — it activates when TI-LFA alone cannot maintain the segment-list validity (e.g., a node failure that invalidates an entire segment-list).

TI-LFA Repair Within an SR-TE Segment-List

SR-TE policy segment-list: PE1→P1→P2→PE3  |  P1→P2 link fails
PE1 (headend) ───→ P1 (PLR) ──✗── P2 ───→ PE3
PE1 ───→ P1 ─repair─→ P3 (detour) ───→ P2 ───→ PE3
P1 detects failure via BFD (sub-50ms). P1 pushes TI-LFA repair labels to reroute through P3.
PE1 is unaware — the SR-TE policy remains active and unchanged. TI-LFA handles it locally.

Protection Hierarchy

SR-TE protection is a layered defense system. Each layer handles failures the previous layer cannot, with increasing scope but also increasing convergence time.

1
TI-LFA — First Responder (sub-50ms)

Automatic, per-segment, local repair. Pre-computed repair label stack in FIB. Triggered by BFD or IGP hello timeout. Handles single link or node failures transparently — the SR-TE headend does not even notice. This is always your first line of defense.

2
Segment-List Failover — Within Candidate Path (sub-second)

If a candidate path has multiple segment-lists and one becomes invalid (a segment cannot be resolved), traffic shifts to the remaining valid segment-lists. This adjusts the weighted ECMP distribution. Only relevant when using multiple segment-lists for redundancy or load-sharing.

3
Candidate Path Failover — Preference-Based (seconds)

If all segment-lists in the active candidate path become invalid, the policy activates the next-best candidate path (the one with the next-highest preference value). This requires the headend to detect that segments are unresolvable — typically after IGP convergence propagates the topology change. Timescale: 1-5 seconds typical.

4
PCE Re-Optimization — Global Optimal (seconds to minutes)

When topology changes, PCE detects the change (via BGP-LS or IGP flooding), re-computes optimal paths, and pushes PCUpdate messages to the headend (PCC). This restores the globally optimal path after TI-LFA's local repair has stabilized traffic. Timer-driven: depends on PCE re-optimization timer (default often 60s, configurable to 1-5s).

Why this hierarchy matters: TI-LFA keeps traffic flowing while slower mechanisms find the best long-term path. Without TI-LFA, you would drop traffic during the seconds it takes for candidate path failover or PCE re-optimization. With it, you get sub-50ms recovery first, then path optimization follows.

Convergence Timeline

When a link failure occurs, the protection mechanisms activate in sequence. This timeline shows a typical failure event in an SR-TE network with TI-LFA + PCE.

Failure Event Timeline — P1→P2 link failure
T=0ms — Link Failure
P1→P2 link goes down. BFD session detects within 3x interval (e.g., 50ms with 3x15ms timers).
T=~50ms — TI-LFA Activates
P1 activates pre-computed repair: pushes repair labels, traffic detours via P3→P4→P2. Zero packet loss window is sub-50ms. Traffic is flowing again.
T=~1-5s — IGP Converges
IS-IS/OSPF floods the topology change. All routers run SPF. New IGP shortest paths computed. TI-LFA repair is replaced by post-convergence optimal path.
T=~2-10s — PCE Detects Change
PCE receives BGP-LS update reflecting the topology change. PCE starts re-optimization timer for affected policies.
T=~5-60s — PCE Re-Optimizes
PCE computes new optimal segment-lists for affected policies. Sends PCUpdate to PCC (headend). New segment-lists installed. Network is now globally optimal again.
No traffic loss: The key takeaway is that traffic never stops flowing. TI-LFA ensures sub-50ms repair, and every subsequent step merely improves the path toward global optimality. This is a massive improvement over RSVP-TE FRR, which required pre-signaled bypass tunnels.

Candidate Path Failover in Action

When the active candidate path becomes invalid, the headend immediately activates the next-best candidate path. This is preference-based — higher preference wins. Here is a concrete example.

P200
Active: Explicit path via P1→P2 (pref 200)

Segment-list: [16011, 16012, 16003] — PE1→P1→P2→PE3. This is the preferred low-latency path. Active as long as P1 and P2 are reachable and the segments resolve.

P100
Standby: Dynamic (PCE-computed, pref 100)

PCE computes the best available path with latency constraint. Ready to activate if pref-200 fails. PCE can pre-compute this to speed up failover.

P10
Last resort: IGP shortest path (pref 10)

Segment-list: [16003] — just the Prefix-SID of PE3. Always valid as long as PE3 is reachable via IGP. No TE guarantees, but traffic keeps flowing.

If P1 suffers a node failure, the explicit segment-list [16011, 16012, 16003] becomes invalid (16011 is unresolvable). The headend detects this after IGP convergence and activates pref-100 (PCE-computed dynamic path). If PCE also cannot find a valid constrained path, pref-10 (IGP default) takes over.

Monitoring & Telemetry

Observability for SR-TE policies — state, counters, liveness, and streaming telemetry

Policy State Machine

Every SR-TE policy has two independent state dimensions: administrative state (configured intent) and operational state (actual forwarding status). Understanding these states is essential for troubleshooting.

Admin State Oper State Meaning Traffic Behavior
Up Up Policy is configured and has at least one valid candidate path with resolvable segment-lists Traffic steered into policy via BSID
Up Down Policy is configured but no valid candidate path exists (all segment-lists unresolvable) Traffic falls back to IGP or is dropped (depends on config)
Down Down Policy is administratively shut down No traffic steered; BSID not programmed in FIB
State transitions
Admin Down / Oper Down ──admin up──→ Admin Up / Oper Down ──valid CP──→ Admin Up / Oper Up
Policy goes Oper Up only when admin-enabled AND at least one candidate path has all segments resolvable.
Troubleshooting tip: If a policy is Admin Up / Oper Down, check: (1) Can the headend resolve all SIDs in the segment-lists? (2) Is the endpoint reachable in the IGP? (3) Has the PCE delegated a valid path? These are the three most common root causes.

Path Verification & Liveness Detection

An SR-TE policy can be operationally up (segments resolvable) but the actual data-plane path might be broken. Liveness detection verifies the end-to-end path is working.

MPLS Ping Per-Policy Verification
Sends an MPLS echo request along the segment-list of the active path. Validates that every transit router correctly forwards the labeled packet. Use: ping mpls sr-te policy name <name>
MPLS Traceroute Hop-by-Hop Path Discovery
Traces the actual forwarding path of labeled packets through the segment-list. Reveals the exact sequence of routers and labels. Use: traceroute mpls sr-te policy name <name>
Performance Measurement Delay/Loss Probes
Periodic probe packets measure one-way delay, round-trip delay, and packet loss along the SR-TE path. Results feed into PCE for SLA-aware re-optimization. Often called PM (Performance Measurement) or STAMP.
Loopback Liveness Endpoint Reachability
BFD session over the SR-TE policy (BFD over BSID). If the BFD session goes down, the policy can be brought operationally down to trigger candidate path failover. Provides continuous liveness monitoring.
Caution: MPLS ping/traceroute validates the control-plane path (label stack is correct). It does not guarantee the data-plane works identically under load. For production SLA monitoring, use Performance Measurement probes combined with BFD-over-policy liveness.

Design Patterns

Proven SR-TE deployment patterns for common network architectures

Pattern 1: WAN Traffic Engineering Most Common

The most widely deployed SR-TE use case: classifying WAN traffic into tiers and steering each tier through a different path optimized for that traffic class. Each tier maps to a BGP color, and an ODN (On-Demand Next-hop) template per color defines the path constraints.

Color 100 — Low Latency
Voice, video conferencing, real-time trading. ODN constraint: metric-type latency. PCE or headend computes the minimum-delay path. Typically traverses fiber routes with lowest propagation delay.
Color 200 — Best Effort
General enterprise traffic, web, email. ODN constraint: metric-type igp (default). Uses standard IGP shortest path — effectively plain SR-MPLS. Low operational cost, no special path computation needed.
Color 300 — Bulk Transfer
Backups, replication, large file transfers. ODN constraint: metric-type te with affinity to prefer high-capacity links. Can be steered to overnight paths or links with available bandwidth. Tolerates higher latency.
Three traffic tiers through the WAN — PE1 to PE3
PE1 ──color 100──→ P1(low-delay) ───→ P2(low-delay) ───→ PE3 Voice/Video
PE1 ──color 200──→ P1(IGP) ───→ P2(IGP) ───→ PE3 Best Effort
PE1 ──color 300──→ P3(hi-cap) ───→ P4(hi-cap) ───→ PE3 Bulk Transfer
How it works: BGP routes for voice destinations carry color:100 extended community. PE1 has an ODN template for color 100 with metric-type latency. When PE1 receives such a route, it automatically instantiates an SR-TE policy to the route's next-hop with latency optimization. No per-prefix policy configuration needed — ODN handles it.

Pattern 2: DC Interconnect High Availability

Connecting two data centers across a WAN with disjoint primary and backup paths. The goal: if any single link or node in the primary path fails, the backup path is guaranteed to be unaffected because it shares no common elements.

Disjoint paths: DC-West (PE1) to DC-East (PE3)
Primary: PE1 ───→ P1 (north) ───→ P2 (north) ───→ PE3
Backup: PE1 ───→ P3 (south) ───→ P4 (south) ───→ PE3
Primary uses north core (P1→P2). Backup uses south core (P3→P4). Zero shared nodes or links.

Implementation: PCE Disjointness

  • Two SR-TE policies in an association group
  • PCE computes both paths simultaneously with node-disjoint or link-disjoint constraint
  • PCEP Association Object (RFC 8697) signals the disjointness requirement
  • If disjoint paths exist, PCE guarantees them

Failover Behavior

  • Primary path (pref 200): via P1→P2
  • Backup path (pref 100): via P3→P4
  • TI-LFA protects both paths individually
  • If primary goes down, candidate path failover to backup in seconds
  • PCE re-optimizes if topology allows a better pair
Disjointness constraint: This is a PCE feature — the headend alone cannot compute disjoint paths because it does not know the other policy's path. The PCE sees both policies, computes them together, and ensures they share no common elements. This is one of the strongest arguments for using PCE in DC interconnect designs.

Pattern 3: Regional Breakout Latency Optimization

Hub-and-spoke WAN where branch traffic should exit to the internet or cloud at the nearest regional breakout point rather than backhauling to a central site. SR-TE steers traffic to the optimal breakout PE based on latency or geography.

Regional breakout — branch at PE1 (west), breakout PEs at PE2 (west) and PE4 (east)
Cloud (west): PE1 (branch) ───→ PE2 (breakout) ─→ Cloud low latency
Cloud (east): PE1 (branch) ───→ P1 → P2 → PE4 (breakout) ─→ Cloud higher latency
Traffic to west-coast cloud exits at PE2 (1 hop). Traffic to east-coast cloud exits at PE4 (3 hops). SR-TE + latency metric ensures optimal breakout selection.

Flex-Algo + SR-TE Combination

Use Flex-Algo for the base topology plane (e.g., Algo 128 = low-latency plane) and SR-TE policies for specific traffic overrides that need path constraints beyond what Flex-Algo provides.

Flex-Algo (baseline)

Algo 128: latency-optimized topology. All PEs participate. Automatic path computation by IGP. Simple — but limited to metric optimization, no arbitrary path constraints.

SR-TE Policy (overrides)

Specific policies for traffic needing affinity constraints, disjointness, or explicit path pinning. More powerful but requires per-policy configuration or ODN templates.

Design guideline: Start with Flex-Algo for broad traffic classes (latency-optimized, TE-metric-optimized). Layer SR-TE policies on top only for traffic that needs specific constraints Flex-Algo cannot express. This minimizes the number of SR-TE policies you need to manage.

Pattern 4: Traffic Optimization PCE + Bandwidth

PCE with bandwidth awareness distributes traffic across multiple paths based on real-time link utilization. This is the most advanced pattern — it requires a feedback loop between the network and the controller.

1
Telemetry Collection

Routers stream interface utilization, queue depth, and SR-TE policy counters to a telemetry collector via gRPC. Data is aggregated and fed to the PCE or a traffic optimization application.

2
Traffic Matrix Computation

The optimization engine builds a traffic matrix from the telemetry data — it knows how much traffic flows between every PE pair and which paths are being used. It identifies congested links and underutilized paths.

3
Path Computation & Rebalancing

PCE computes new segment-lists (or adjusts weighted ECMP ratios) to shift traffic away from congested links toward underutilized ones. This is a global optimization across all policies.

4
PCUpdate to Headends

PCE pushes updated segment-lists and weights to headend routers via PCEP PCUpdate. Headends install the new paths. Traffic shifts. The loop repeats at the next telemetry interval.

Requirements: This pattern needs (1) a PCE with bandwidth-aware path computation (e.g., Cisco WAE, Juniper Paragon Pathfinder), (2) streaming telemetry from all routers, (3) careful tuning of re-optimization timers to avoid oscillation. Start conservative — re-optimize no more frequently than every 5 minutes until you understand the traffic dynamics.

Segment types, key terms, RFCs, and a decision tree for SR-TE policy design

Segment Types

SR-TE policies use typed segments in their segment-lists. Each type encodes a different forwarding instruction.

Type Name Description Common Use
Type A SR-MPLS Label A raw MPLS label value. No validation against IGP — just push this label. Inter-domain segments, labels from external controllers
Type B SRv6 SID (IPv6 addr) An IPv6 address representing an SRv6 segment. Not used in SR-MPLS policies. SRv6 policies only
Type C IPv4 Prefix + SR-Algo Resolved to an MPLS label via IGP: SRGB base + prefix SID index for the given algorithm. Node SIDs, Flex-Algo SIDs — the most common type in SR-MPLS
Type D IPv6 Node + SR-Algo Same as Type C but for IPv6-addressed nodes. IPv6-only SR-MPLS networks
Type E IPv4 Adj + Interface Resolved to the Adj-SID label for a specific adjacency (local address + remote address). Explicit link pinning — force traffic over a specific interface
Type F IPv4 Adj + Link-Local Resolved to Adj-SID using link-local addresses (for unnumbered interfaces). Unnumbered interfaces (less common)
Type G IPv6 Adj (global) Resolved to Adj-SID using IPv6 global addresses. IPv6 adjacency pinning
Type H IPv6 Adj (link-local) Resolved to Adj-SID using IPv6 link-local addresses. IPv6 link-local adjacency pinning
Type I IPv6 Node + SR-Algo (SRv6) IPv6 prefix resolved to SRv6 SID for a given algorithm. SRv6 policies only
Type J/K IPv6 Adj (SRv6) IPv6 adjacency resolved to SRv6 End.X SID. SRv6 policies only
For SR-MPLS, focus on Type A, C, and E. Type C (prefix + algorithm) is the workhorse — it is how you reference node SIDs and Flex-Algo SIDs. Type E (adjacency) is for explicit link pinning. Type A (raw label) is for inter-domain or special cases.

Decision Tree: Do You Need SR-TE Policies?

Not every SR-MPLS network needs SR-TE policies. Use this decision tree to determine the right level of traffic engineering for your deployment.

?
Do you need traffic engineering at all?

No: Use plain SR-MPLS with IGP shortest path + TI-LFA. This covers most enterprise and small SP networks. Prefix-SIDs route traffic along IGP best path, TI-LFA provides sub-50ms protection. You are done.
Yes: Continue below.

?
Is simple metric optimization enough?

Yes: Use Flex-Algo (RFC 9350). Define algorithm 128 = latency-optimized, algorithm 129 = TE-metric-optimized. IGP computes paths automatically. No per-policy configuration. Scales to thousands of destinations with zero policy management.
No (need affinity, disjointness, explicit paths): Continue below.

?
Single domain or multi-domain?

Single domain: Headend computation may suffice. The headend has full IGP visibility and can compute constrained paths locally. Use ODN templates with dynamic path computation at the headend.
Multi-domain: You need a PCE. No single headend has visibility across domain boundaries. PCE aggregates topology from multiple domains via BGP-LS.

?
How many SR-TE policies will you have?

<100 policies: Headend computation works fine. Local dynamic path computation scales well for this range. PCE is optional (but still useful for centralized management).
>100 policies: Strongly consider PCE. Centralized computation avoids redundant SPF runs across headends, provides global optimization, and simplifies management. PCE also enables advanced features like disjointness and bandwidth-aware computation.

?
Do you need SLA guarantees with dynamic adaptation?

Yes: Deploy ODN + Color + PCE with performance measurement. BGP routes carry color communities encoding SLA intent. ODN templates auto-instantiate policies. PCE computes paths with SLA constraints. Performance measurement probes verify SLA compliance. PCE re-optimizes when SLA is at risk. This is the full SR-TE stack.
