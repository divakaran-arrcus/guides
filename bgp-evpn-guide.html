Why Layer 2 Matters

You know how to route everything — so why can't we just route everything?

The L3VPN Engineer's Question

If you come from an L3VPN background, your instinct is: "Give every host a routed interface, run a routing protocol, and call it done." That works beautifully in WAN and service provider networks. But modern data centers throw a wrench into this model.

The fundamental issue is that some workloads require L2 adjacency — they depend on being in the same broadcast domain, sharing the same subnet, or communicating at the Ethernet level. Routing between them breaks these assumptions.

Key insight: In a WAN you control the applications. In a data center you host other people's applications — virtual machines, containers, databases — that were designed expecting a LAN environment. You can't always redesign the application.

Why Applications Expect L2

Many protocols and behaviors are fundamentally tied to being on the same Ethernet segment:

  • ARP — resolves IP addresses to MAC addresses via broadcast. Works only within a broadcast domain (same subnet).
  • Gratuitous ARP — a host announces its own IP-to-MAC mapping to update neighbors. Requires L2 reachability to all potential communicators.
  • IP multicast (local) — many database clustering protocols (Oracle RAC, VMware vSAN heartbeats) use link-local multicast.
  • License servers — some commercial software licenses are tied to MAC addresses and refuse to run if the MAC changes.

Use Case 1: VM Live Migration

This is the most common and critical use case. When a virtual machine is migrated from one physical server to another (VMware vMotion, KVM live migration), the VM must keep its same IP address and MAC address. If either changes, all active TCP connections drop.

The L2 vs L3 Boundary

Think of the network as two planes. In a traditional three-tier data center, the L2 boundary was a single ToR switch — traffic between racks immediately hit a router (L3). Modern multi-tenant data centers need to push that L2 boundary outward so a VLAN can span many racks, even multiple sites.

What Is a MAC Address?

A MAC address (Media Access Control address) is a 48-bit identifier burned into every network interface card. It is the address used by Ethernet for delivering frames within a single network segment. Think of it as the Layer 2 equivalent of an IP address — but unlike IP, MACs are not hierarchical and not routable.

Special MAC addresses: The multicast/group bit is bit 0 (the least-significant bit) of the first byte of a MAC address. Any MAC with an odd first byte is a multicast (group) address — for example 01:xx:xx:xx:xx:xx, 03:xx:xx:xx:xx:xx, 05:xx:xx:xx:xx:xx, etc. 01:xx:xx:xx:xx:xx is the most common multicast prefix (used by Ethernet multicast, STP, LACP, etc.) but not the only one. ff:ff:ff:ff:ff:ff is a special case: the broadcast address (every device on the segment receives it), which is technically also a multicast address since its first byte (0xff = 255) is odd.

How Ethernet Differs from IP Routing

IP Routing (L3)

Router reads destination IP, looks up in routing table (prefix match), decrements TTL, rewrites MAC header, forwards out interface. Each hop is a new L2 frame.

Ethernet Switching (L2)

Switch reads destination MAC, looks up in MAC table (exact match), forwards out the port. No IP addresses involved. No TTL. Frame is delivered as-is within the segment.

BUM Traffic: Broadcast, Unknown Unicast, Multicast

These three traffic types are grouped together because they all share one painful property: you cannot forward them to a single destination. They must be sent to many (or all) members of the VLAN. This is the core scalability challenge for L2 at scale.

B — Broadcast
Destination MAC = ff:ff:ff:ff:ff:ff. Every device in the VLAN receives it. ARP requests are the most common source. DHCP discover is another.
U — Unknown Unicast
Destination MAC is a specific host, but the switch has no MAC table entry for it. Must flood to all ports in the VLAN hoping the destination will see it and reply.
M — Multicast
Destination MAC in the range 01:xx:xx:xx:xx:xx. Without IGMP snooping, switches treat multicast like broadcast and flood to all ports in the VLAN.
Why BUM matters for EVPN: When you stretch a VLAN across a VXLAN fabric, BUM traffic must somehow reach all VTEPs (tunnel endpoints) that have hosts in that VLAN. BGP EVPN provides two solutions: ingress replication (unicast copies to each VTEP) or underlay multicast. This is a central design question in any EVPN deployment.

Spanning Tree Protocol (STP) — Loop Prevention at a Cost

Ethernet networks are inherently vulnerable to loops. Unlike IP (which has TTL), Ethernet frames have no hop limit. A loop causes a broadcast storm: a broadcast frame circulates forever, consuming all bandwidth within seconds.

The loop problem: If you connect two switches with two cables (for redundancy), a broadcast frame from Host A enters Switch 1, gets sent out both cables to Switch 2, which sends it back to Switch 1 on the other cable, which sends it out again... infinitely. The network collapses in milliseconds.

Spanning Tree Protocol (STP — IEEE 802.1D) solves this by blocking redundant links. It elects a root bridge and builds a loop-free tree topology by putting certain ports into a "blocking" state.

This is exactly why VXLAN matters: VXLAN runs over an IP fabric. The underlay uses ECMP routing — all links are active, no STP. The L2 behavior is emulated in the overlay. You get the L2 adjacency that applications need, without STP's limitations.

BGP EVPN's answer: Instead of flooding ARP, the VTEP (leaf switch) intercepts the ARP request and answers it locally using its BGP-learned MAC/IP table. Zero flooding. This is called ARP suppression and is one of EVPN's key benefits.

MAC Table Size Limits

Physical switches have a finite amount of TCAM (Ternary Content-Addressable Memory) for MAC address tables. TCAM is expensive and power-hungry. A typical top-of-rack switch might support 32K–128K MAC entries. In a large virtualized environment, this fills up fast.

What about MLAG/VPC — the pre-EVPN workaround? Before EVPN, data centers used Multi-Chassis LAG (MLAG, also called vPC on Cisco Nexus) to work around STP's link-blocking problem. MLAG pairs two physical switches into a virtual single switch, so a dual-homed server sees one logical link — STP does not block it. But MLAG has hard limits: it is vendor-proprietary, restricted to exactly 2 switches per pair, and requires tight state synchronization over a dedicated peer-link. Scaling beyond 2 switches means stacking MLAG domains with growing complexity. EVPN multi-homing (Type-1 + Type-4 routes) replaces MLAG entirely with a standards-based solution that scales to any number of VTEPs sharing an Ethernet Segment — no proprietary peer-link required.

What Is VXLAN?

Virtual eXtensible LAN — the tunnel technology that carries L2 over an IP fabric

The Overlay / Underlay Model

VXLAN uses the same overlay/underlay architecture you already know from MPLS. The mental model is identical — only the technology changes.

MPLS (you know this)

Underlay: IGP (OSPF/IS-IS) provides IP reachability to all router loopbacks.

Overlay: MPLS labels identify VPN paths. Traffic is encapsulated with MPLS labels and tunneled through the underlay IP network.

Control plane: BGP distributes VPN routes and labels (MP-BGP with VPNv4/VPNv6 AFI).

VXLAN EVPN (what we're learning)

Underlay: IP fabric (OSPF/IS-IS/eBGP) provides IP reachability to all leaf loopbacks (VTEP addresses).

Overlay: VXLAN encapsulation carries L2 frames (with VNI identifier) through UDP tunnels between VTEPs.

Control plane: BGP EVPN distributes MAC/IP reachability and VNI mappings (MP-BGP with L2VPN EVPN AFI).

Key Terms: VTEP and VNI

VTEP — VXLAN Tunnel Endpoint

A VTEP is the device that encapsulates and decapsulates VXLAN traffic. It sits at the boundary between the L2 domain (where hosts live) and the IP underlay (where tunnels traverse). In a spine-leaf fabric, the leaf switch is the VTEP.

MPLS PE Router (you know this)

  • Sits at the edge of the MPLS core
  • Receives customer IP traffic on CE-facing interface
  • Pushes MPLS labels and forwards into core
  • Pops labels and delivers to customer on other side
  • Has a loopback used as tunnel endpoint (LDP/RSVP)

VTEP — VXLAN Tunnel Endpoint

  • Sits at the edge of the VXLAN fabric (leaf switch)
  • Receives Ethernet frames from hosts on access ports
  • Encapsulates in VXLAN/UDP/IP and forwards into underlay
  • Decapsulates on the remote side, delivers to local host
  • Has a loopback used as VTEP IP (reachable via underlay IGP)

VNI — VXLAN Network Identifier

The VNI is the 24-bit field in the VXLAN header that identifies which virtual network a frame belongs to. It is the VXLAN equivalent of the MPLS VPN label or the VLAN ID.

The Big Picture: What VXLAN Encapsulation Does

When Host-A sends a frame, it goes out as a plain Ethernet frame on the local segment. The VTEP (the leaf switch acting as a tunnel endpoint) intercepts this frame, wraps it entirely inside a new UDP/IP packet, and sends it across the IP underlay network. The remote VTEP unwraps it and delivers the original frame to the destination host.

Think of it like taking a physical letter (the inner frame), putting it in a new envelope (UDP/IP), and mailing it through a courier network (the IP underlay). The courier only sees the outer envelope address — not what's inside.

Key insight: VXLAN is MAC-in-UDP. The entire original Ethernet frame (including the original MAC headers) becomes the payload of a UDP packet. This is fundamentally different from MPLS, where you push labels in front of the IP header — VXLAN encapsulates the entire L2 frame.

VXLAN adds exactly 50 bytes of overhead: 14 (outer Ethernet) + 20 (outer IP) + 8 (UDP) + 8 (VXLAN header) = 50 bytes. If hosts send 1500-byte frames, the outer packet is 1550 bytes. Your underlay network MUST have MTU ≥ 1550, otherwise packets will be silently dropped or fragmented.

VXLAN vs MPLS Encapsulation — Side by Side

You know MPLS from L3VPN. Here's how the two encapsulations compare conceptually:

MPLS (L3VPN)

What gets encapsulated: Only the IP packet. The original Ethernet header is stripped at the PE.

Label stack: VPN label (bottom) + Transport label (top) pushed in front of IP header.

Overhead: 4 bytes per label, 8 bytes total for a typical 2-label stack.

Forwarding: Label switching at every hop — swaps transport label.

Service: L3VPN — IP routing between sites. No L2 (MAC) information carried.

MTU impact: Adds 8 bytes (2 labels) → underlay needs 1508 bytes for 1500B packets.

VXLAN

What gets encapsulated: The entire L2 Ethernet frame including the original MAC headers.

Headers added: Outer Ethernet + Outer IP + UDP + 8-byte VXLAN header.

Overhead: 50 bytes fixed — significantly more than MPLS.

Forwarding: Standard IP routing — underlay routes outer IP dst to remote VTEP.

Service: L2 extension — hosts appear on the same LAN segment, MAC addresses preserved end-to-end.

MTU impact: Adds 50 bytes → underlay needs 1550 bytes for 1500B inner frames.

The fundamental difference: MPLS-based L3VPN terminates at L3 — the PE does IP routing. VXLAN operates at L2 — the VTEP does MAC bridging. VXLAN allows you to extend a single broadcast domain (VLAN) across a routed IP network as if the hosts were all on the same LAN. This is why we need a completely different control plane to distribute MAC addresses — which is where BGP EVPN comes in.
Flood-and-Learn VXLAN

How VXLAN works without a control plane — and why it doesn't scale

No Control Plane — Pure Data Plane Learning

Basic VXLAN (RFC 7348) defines only the data plane encapsulation. It has no control plane to distribute MAC addresses or VTEP membership. Instead, it relies on the same mechanism that traditional Ethernet switches have used for decades: flood unknown destinations, learn from return traffic.

The challenge is that "flooding" in a traditional network means sending to all ports on a switch. In a VXLAN network, "flooding" means sending to all remote VTEPs that participate in the same VNI. This is called BUM traffic: Broadcast, Unknown-unicast, and Multicast.

BUM traffic = any traffic that must be sent to every endpoint in the network segment: ARP broadcasts, unknown destination frames, multicast. In flood-and-learn VXLAN, BUM traffic is replicated and sent to every VTEP in the VNI.
Static VTEP Flood Lists — The Operational Burden of Flood-and-Learn
  • No auto-discovery: In flood-and-learn VXLAN (RFC 7348, no EVPN), VTEPs do not automatically discover each other. Each VTEP has no idea which other VTEPs participate in a given VNI.
  • Static configuration required: The list of remote VTEP IP addresses (the flood list) must be manually configured on every VTEP, or a multicast group must be mapped to each VNI so BUM traffic uses IP multicast in the underlay.
  • Scaling problem: Adding a new VTEP to the fabric requires updating the flood list configuration on every existing VTEP. In a 50-leaf fabric, a new leaf = 49 config changes. This is operationally fragile and error-prone.
  • Why BGP EVPN solves this: EVPN Type-3 (IMET) routes automate VTEP discovery — when a new leaf comes up and joins a VNI, it advertises a Type-3 route, and all existing VTEPs automatically add it to their flood list. Zero-touch VTEP discovery.

BUM Traffic Handling Options

There are two main ways to handle BUM (Broadcast, Unknown-unicast, Multicast) traffic in flood-and-learn VXLAN:

Option 1: Ingress Replication (Head-End)

Each VTEP maintains a list of all other VTEPs in the VNI. When BUM traffic arrives, the ingress VTEP creates a separate copy for each remote VTEP and sends them individually.

  • No multicast required in underlay
  • CPU/memory intensive at the ingress VTEP
  • N-1 copies for N VTEPs — O(N) replication
  • The "replication list" must be manually configured or learned somehow

Option 2: Multicast Underlay

Each VNI is mapped to an IP multicast group (e.g., VNI 10100 → multicast group 239.1.1.100). BUM traffic is sent to the multicast group address. The underlay replicates it to all VTEPs that have joined the group.

  • Requires PIM multicast in the underlay
  • Efficient replication done by IP multicast tree
  • Scales better for large networks
  • Adds operational complexity (multicast routing)

Enter BGP EVPN.

Why BGP EVPN?

BGP as a universal control plane — from IP prefixes to MAC addresses

Bridge From L3VPN: Same Protocol, New Address Family

You already know BGP MPLS L3VPN (RFC 4364). In L3VPN, BGP carries IP prefixes as VPNv4/v6 routes with a VPN label, distributed between PE routers using MP-BGP. The key insight for EVPN is:

L3VPN (What You Know)

BGP carries: IP prefixes (e.g., 10.1.1.0/24)

Attached to: VPN label (identifies which VRF)

Route type: VPNv4 (AFI 1, SAFI 128) or VPNv6 (AFI 2, SAFI 128)

PE learns: "10.1.1.0/24 is reachable via remote PE 1.2.3.4, use VPN label 100"

Data plane: IP packets routed into VRF, pushed with MPLS labels

EVPN (What We're Building)

BGP carries: MAC addresses + optionally IP addresses

Attached to: VNI (identifies which virtual network segment)

Route type: L2VPN EVPN (AFI 25, SAFI 70)

VTEP learns: "MAC aa:bb:cc:00:02:01 is at remote VTEP 192.168.1.3, VNI 10100"

Data plane: Ethernet frames bridged into VNI, encapsulated in VXLAN

The elegance: Both L3VPN and EVPN use the same BGP mechanisms — Route Distinguishers, Route Targets, MP-BGP, Route Reflectors. If you can operate L3VPN, you already understand 80% of EVPN's operational model. The concepts map directly.

iBGP + Route Reflectors vs eBGP — Two Valid Models

This guide uses iBGP with spine Route Reflectors (single AS 65000) — a common model familiar to L3VPN operators. However, an equally popular approach — especially in hyperscale and Clos fabrics — is eBGP with per-device ASNs:

eBGP model: Each leaf and each spine gets its own unique ASN (e.g., Leaf-1 = AS 65001, Leaf-2 = AS 65002, Spine-1 = AS 65100). Every leaf peers with spines over eBGP. EVPN routes are carried over these eBGP sessions. No Route Reflector needed — eBGP inherently reflects routes to all peers.

Key differences:
• iBGP + RR: one AS, simpler policy, familiar to MPLS operators, AS_PATH is empty within the fabric
• eBGP: no RR required, each device is independent, AS_PATH provides built-in loop prevention, scales to very large fabrics (used by hyperscalers)
• Both carry EVPN routes identically — the NLRI format, Route Targets, and VNI semantics are identical

This guide uses iBGP + RR for simplicity and to leverage the L3VPN parallel (same RR model, same MP-BGP mechanics). In production, choose based on your operational preference and fabric scale.

ARP Suppression — EVPN's Most Impactful Feature

With flood-and-learn, every ARP request must be flooded across the entire fabric. In a data center with 100,000 VMs, ARP storms can consume significant bandwidth. EVPN's ARP suppression eliminates most of this:

1
VTEP learns MAC+IP binding via BGP (Type-2 route)

When Leaf-3 receives a BGP EVPN Type-2 route: "MAC aa:bb:cc:00:02:01, IP 10.1.1.20, at VTEP 192.168.1.3, VNI 10100" — it stores this in its local ARP suppression table.

2
Host-A sends ARP "Who has 10.1.1.20?" — Leaf-1 intercepts it

Instead of flooding the ARP across the fabric, Leaf-1 checks its ARP suppression table. It finds "10.1.1.20 → aa:bb:cc:00:02:01" — it already knows the answer.

3
Leaf-1 generates a proxy ARP reply — no fabric flood needed

Leaf-1 crafts an ARP reply on behalf of Host-B, using the MAC it learned via BGP. It sends this directly to Host-A on the local port. The ARP never leaves Leaf-1. The entire VXLAN fabric sees zero ARP traffic for this lookup.

ARP suppression can eliminate 90%+ of BUM traffic in a well-deployed EVPN fabric

The Five EVPN Route Types

EVPN defines five distinct route types (RT-1 through RT-5), each carrying different information for different purposes. Think of them as different BGP path attributes for different services — similar to how VPNv4, VPNv6, and labeled-unicast are different BGP route types for different L3VPN use cases.

Type-1: Ethernet Auto-Discovery (EAD)

Multi-homing only

Used when a host (or CE device) is dual-homed to two VTEPs. The pair of VTEPs form an "Ethernet Segment" identified by an ESI (Ethernet Segment Identifier). Type-1 routes signal the ESI membership and enable two things:

  • Aliasing: Remote VTEPs can load-balance traffic across both VTEPs in the ES, even if they've only learned the MAC behind one of them
  • Mass withdrawal: When a link fails, a single Type-1 withdrawal can invalidate ALL MACs behind that Ethernet Segment instantly — faster than withdrawing thousands of individual Type-2 routes

Type-3: Inclusive Multicast Ethernet Tag (IMET)

Core — every deployment

The Type-3 route does two things: it announces that a VTEP is participating in a given VNI ("I am part of this virtual network"), and it signals how BUM traffic should be delivered to that VTEP (via ingress replication or multicast).

Type-4: Ethernet Segment Route

Multi-homing only

When a host is multi-homed to two VTEPs (via a LAG or dual-homed connection), both VTEPs share the same Ethernet Segment (identified by the ESI). Only one VTEP should forward BUM traffic into the segment at a time to prevent duplicates — this VTEP is called the Designated Forwarder (DF).

Export RT

Scaling RT Distribution: RT-Constrain (RFC 4684)

The scale problem at 100+ VNIs: As a fabric grows, every VTEP receives every EVPN route from the Route Reflector — even routes for VNIs it does not participate in. A leaf that only has VNI 10100 and 10200 still receives and processes routes for all 1,000 VNIs in the data center. BGP table sizes and CPU churn grow proportionally with the total number of VNIs, not the number of local VNIs.

RT-Constrain (RFC 4684) solves this: each VTEP advertises to the RR the set of Route Targets it wants to import, using a special BGP address family (AFI 1, SAFI 132). The Route Reflector tracks these per-peer RT subscription lists and only sends routes whose RTs appear in the receiver's subscription. A leaf with 2 VNIs only receives routes for those 2 VNIs, regardless of how many VNIs exist in the fabric.

Result: BGP table size on leaf nodes stays proportional to local VNI count, not total fabric VNI count. Essential for large multi-tenant deployments with hundreds of VNIs — without RT-Constrain, EVPN route tables on leaves can reach millions of entries in large clouds.

Duplicate Detection: If two VTEPs keep incrementing and advertising the same MAC within a threshold window (e.g., 5 moves in 180 seconds), the control plane marks the MAC as a "duplicate" and stops processing mobility events — protecting against network loops from misconfigured hosts.
MAC Move Dampening — What Happens When Duplicate Detection Triggers
  • Threshold: Vendor-specific, but typically around 5 MAC moves within a 3-minute window. Once exceeded, the MAC is flagged as a duplicate by the control plane.
  • Frozen MAC: The MAC entry is "frozen" — depending on the implementation, traffic to that MAC may be blackholed (dropped silently) or flooded (treated as unknown unicast). Neither is desirable in production.
  • Recovery: Requires either manual intervention (clear evpn arp-cache / clear bgp evpn mac duplicate — vendor-specific command) or waiting for the hold timer to expire (typically 2–5 minutes). Traffic disruption lasts for the entire hold period.
  • Common root causes: VM live-migration loop (orchestrator keeps migrating a VM back and forth), misconfigured LAG where both member links appear as separate VTEPs, or an actual MAC address conflict between two hosts.
  • Operational note: MAC duplicate detection is one of the most common support escalations in production EVPN deployments. Always monitor for duplicate MAC events in your NMS/syslog pipeline.

Ingress Replication vs Multicast — Trade-offs

Ingress Replication (IR)

How: The ingress VTEP creates N unicast copies, one per remote VTEP in the flood list. Each copy is a separate IP packet sent to a specific remote VTEP.

Underlay needed: Unicast only. Works with any IP underlay — OSPF, IS-IS, eBGP. No PIM, no IGMP, no multicast routes.

Bandwidth cost: N copies leave the ingress switch's uplinks simultaneously. For 10 remote VTEPs with a 1500-byte frame: 10× the bandwidth on uplinks.

CPU/ASIC cost: Ingress VTEP replicates in hardware. Most modern ASICs handle this at line rate.

Recommended for most deployments. Zero underlay complexity.

Underlay Multicast (PIM)

How: The ingress VTEP sends 1 packet to a multicast group address. Network routers in the underlay replicate the packet as it fans out toward subscribed VTEPs.

Underlay needed: PIM-SM (or PIM-SSM) with RP, IGMP on all switches. Significantly more complex underlay.

Bandwidth cost: Only 1 copy leaves the ingress VTEP's uplinks. Network replicates at fan-out points — efficient for large fabrics with many VTEPs.

CPU/ASIC cost: Each switch in path does replication — distributed cost.

Added underlay complexity. Only beneficial with many VTEPs (>20-30).

ARP Suppression
How EVPN eliminates most ARP broadcast flooding — and why Type-2 routes include the IP address.
This is why Type-2 includes both MAC AND IP. Without the IP address in the Type-2 advertisement, each VTEP would have no way to answer ARP requests locally. The IP in the Type-2 NLRI feeds the ARP suppression cache, which is the single most important optimization EVPN provides over a traditional flood-and-learn VXLAN fabric.

The Problem: ARP is Broadcast, and Broadcasts Are Expensive

ARP is how IPv4 resolves an IP address to a MAC address. It works by sending a broadcast: "Who has IP X? Tell me your MAC." In a traditional network, this broadcast goes to every host in the VLAN. In a VXLAN fabric, this means BUM flooding — the ingress VTEP must send a VXLAN-encapsulated copy to every remote VTEP in the flood list.

Scale Problem

In a large data center with 1000 VMs and 10 VTEPs per VNI, each ARP request causes 9 copies to be sent to remote VTEPs. 1000 VMs each sending periodic ARPs = enormous BUM baseline.

Chatter Problem

VMs send gratuitous ARPs at boot, after live migration, after IP changes. Kubernetes pods generate ARP storms at scale. All of this is BUM traffic that must be replicated to every VTEP.

Without ARP suppression: Even with EVPN controlling MAC advertisement, ARP is still flooded as BUM. Type-3 routes build the flood list, but every ARP request still gets sent to all VTEPs in that list. ARP suppression is what makes EVPN truly efficient.

When a VTEP receives a Type-2 MAC+IP route, it stores the MAC-to-IP binding in a local ARP suppression cache. When a local host ARPs for an IP that's in this cache, the VTEP answers directly — no flood needed.

BFD for fast failure detection (production requirement): Production deployments always configure BFD (Bidirectional Forwarding Detection) on all underlay adjacencies. BFD detects link or node failures in milliseconds — a typical deployment uses 3 × 50 ms = 150 ms detection time. Without BFD, a silent link failure (no carrier loss, e.g., a unidirectional optical fault) can take 30–40 seconds to detect via IGP dead timers alone. When BFD detects the failure, it immediately notifies the IGP, which reconverges and reroutes all VXLAN tunnels through the surviving spine — minimizing traffic loss to sub-second. Enable BFD on every OSPF/IS-IS interface on both leaf uplinks and spine downlinks.
MTU Planning — Critical Before Enabling VXLAN

VXLAN adds 50 bytes of overhead to every packet (14 outer Ethernet + 20 outer IP + 8 UDP + 8 VXLAN header). If hosts send standard 1500-byte frames, the outer VXLAN packet is 1550 bytes. Underlay interfaces MUST support at least 1550-byte frames or VXLAN packets are silently dropped — no ICMP error, no log entry, just missing traffic.

Best practice: configure MTU 9216 on all spine-leaf links. This provides headroom for 1500-byte inner frames with VXLAN overhead, plus future encapsulations. Mismatched MTU is the #1 cause of silent packet drops in VXLAN fabrics — it typically manifests as large TCP transfers stalling while small pings succeed, making it extremely difficult to diagnose without knowing to check MTU.

Verify MTU before enabling VXLAN:
ping 192.168.1.3 size 1550 df-bit source 192.168.1.1 between VTEP loopbacks. If this fails but size 100 succeeds, you have an MTU mismatch. Fix it before proceeding to Step 2.

Security Considerations for BGP EVPN Deployments
  • Control plane: Secure BGP EVPN sessions with TCP-AO (TCP Authentication Option — RFC 5925) to prevent session hijacking and route injection. Use MD5 as a fallback if TCP-AO is not supported, but prefer TCP-AO for stronger protection.
  • Data plane: VXLAN has NO built-in authentication or encryption. Any device that can send UDP port 4789 packets to a VTEP can inject traffic into the overlay — this is a significant attack surface if the underlay is not trusted.
  • Mitigations: Deploy infrastructure ACLs on underlay interfaces to permit VXLAN (UDP 4789) only from known VTEP loopback addresses. Apply control-plane policing (CoPP) on spine and leaf management planes to rate-limit unexpected VXLAN sources.
  • Encryption: Consider MACsec (IEEE 802.1AE) on underlay physical links if data confidentiality is required. MACsec encrypts at Layer 2 and has near-zero overhead with hardware support — this protects both the VXLAN payload and the underlay routing traffic.

EVPN Multi-Homing — Type-1 & Type-4 Routes
Dual-homed servers with fast failover and per-VLAN load balancing

Why Multi-Homing?

In a data center, servers often have two NICs connected to two different leaf switches for redundancy. The server bonds them into a LAG (Link Aggregation Group). From the server's perspective, it's one fat pipe. From the network's perspective, traffic can arrive on either leaf.

Without EVPN Multi-Homing

You'd need traditional MLAG (Multi-chassis LAG) — proprietary protocol to synchronize the LAG state between two switches. Complex, vendor-specific, doesn't scale across fabrics.

With EVPN Multi-Homing

EVPN provides standards-based multi-homing using BGP. Type-1 and Type-4 routes handle DF election, aliasing, and fast failover. Works across any BGP-capable EVPN fabric.

Ethernet Segment (ES): The set of links from a host (or CE device) to multiple VTEPs is called an Ethernet Segment. It's given a unique identifier — the ESI — shared between all VTEPs that connect to that ES.

Split-Horizon Filtering — Preventing Overlay Loops

DF election controls who forwards BUM traffic from the fabric to the ES. But what about the reverse direction — traffic arriving from the fabric that a non-DF VTEP must not re-flood? This is handled by the split-horizon rule:

The rule: A BUM frame that arrives at a VTEP from the VXLAN overlay (i.e., decapsulated from a VXLAN tunnel) must NEVER be re-encapsulated and sent back into the overlay via the same Ethernet Segment. The frame can only be forwarded to local access ports — never back into the fabric.

Example: Remote Leaf-3 sends a BUM frame via VXLAN to both Leaf-1 (DF) and Leaf-2 (Non-DF) for VLAN 100. Leaf-1 decapsulates and forwards to Host-C via the ES. Leaf-2 also decapsulates it — but because the Non-DF rule blocks forwarding to the ES, and the split-horizon rule blocks re-flooding to the overlay, the frame is simply dropped. No loop occurs.

Without split-horizon: Leaf-2 would re-encapsulate the BUM frame and send it back into the fabric. Leaf-1 would receive it again and forward to Host-C — creating an infinite BUM loop between the two VTEPs. Traffic would consume bandwidth and trigger MAC table instability across the entire fabric.

Active/Active vs Active/Standby Multi-Homing

EVPN multi-homing supports two forwarding modes. The choice depends on whether the connected host or CE device can form a LAG (Link Aggregation Group).

Active/Active (shown in this guide)

What it means: Both VTEPs (Leaf-1 and Leaf-2) actively forward known-unicast traffic to and from the ES simultaneously.

Requirement: The host/server must support LAG (bonding/LACP) across both uplinks. Both physical ports are active members of the bond.

Traffic: Inbound unicast load-balanced via aliasing (Type-1 per-EVI routes signal both VTEPs as valid next-hops). BUM forwarding controlled by DF election.

Use case: Dual-homed servers with NIC bonding. The dominant model in modern DC deployments. Maximum bandwidth utilization.

Active/Standby

What it means: Only the DF VTEP forwards ALL traffic (both known-unicast and BUM). The non-DF VTEP is a hot standby — it monitors but does not forward.

Requirement: Host cannot do LAG. For example, a CE device with two separate uplinks (not bonded) where only one path should be active at a time.

Traffic: All forwarding through the DF. Remote VTEPs send unicast only to the DF's VTEP IP (no aliasing/ECMP). On failure, DF re-election activates the standby.

Use case: Legacy CE devices, routers without LAG capability, or scenarios requiring strict single-path forwarding. Less bandwidth efficiency but simpler.

Most DC deployments use Active/Active with LACP. When servers have dual NICs bonded with LACP (IEEE 802.3ad), Active/Active gives you full bandwidth across both uplinks plus sub-second failover via Type-1 mass withdrawal. Active/Standby is reserved for edge cases where the attached device cannot participate in a LAG.

Type-5 Routes & Inter-Subnet Routing (IRB)
Moving beyond pure L2: routing between subnets inside the EVPN fabric

The Problem: Hosts in Different Subnets

EVPN Type-2 routes carry L2 MAC addresses. That handles hosts in the same subnet beautifully. But what if Host-A (10.1.1.10 in VLAN 100) needs to talk to Host-C (10.2.2.10 in VLAN 200)? That's an L3 routed hop — you need a gateway.

Traditional approach: Route traffic to an external router or firewall. Spine becomes the L3 gateway. But that's a traffic trombone — data has to hairpin up to the spine and back down. EVPN solves this with Integrated Routing and Bridging (IRB).
IRB: Each VTEP (leaf) acts as both an L2 switch AND an L3 router simultaneously. The leaf has a gateway IP for each VLAN it hosts. When a host sends a packet to a different subnet, the leaf routes it locally — no external router needed.

Asymmetric IRB

In asymmetric IRB, all the routing is done at the ingress VTEP. The egress VTEP only does L2 forwarding. The model is called "asymmetric" because the ingress does more work than the egress.

Symmetric IRB — The Scalable Solution

Symmetric IRB introduces a dedicated L3 VNI for inter-subnet traffic. Both ingress and egress VTEPs do routing. The L3 VNI is a "transit" VNI associated with a VRF (tenant routing domain).

L3 VNI: A VNI reserved for routed traffic in a VRF. It doesn't correspond to any VLAN — it's just a "label" to say "this VXLAN packet is carrying L3 traffic for tenant VRF X." Think of it like an MPLS L3VPN label.

Anycast Gateway MAC — Same IP, Same MAC on Every Leaf

In symmetric IRB, every leaf switch (VTEP) configures its IRB/gateway interface with the same virtual IP AND the same virtual MAC address — for example, IP 10.1.1.1 with MAC 00:00:5e:00:01:01. This is called the anycast gateway MAC.

Why it matters for host mobility: When a VM migrates from Leaf-1 to Leaf-3, its ARP cache still holds the gateway MAC (00:00:5e:00:01:01). Because every leaf uses the exact same MAC, the cached ARP entry remains valid — the VM does not need to re-ARP. Traffic resumes immediately after the BGP Type-2 update propagates the new VTEP binding.

Without anycast MAC: Each leaf would have a unique gateway MAC. After migration, the VM would send frames to the old gateway MAC, which no longer exists on the new leaf. Traffic black-holes until the ARP cache expires (often minutes) and the host re-ARPs — unacceptable for live VM migration in a DC fabric.

Configuration note: The anycast MAC (e.g., 00:00:5e:00:01:01) is provisioned identically on every leaf in the fabric. It is a Day-1 design decision and must be consistent across all VTEPs sharing the same VRF and IRB interface.

Type-5: IP Prefix Route

Type-2 routes carry individual MAC+IP pairs (host routes /32 or /128). For advertising entire IP prefixes (subnets, summaries, or external routes), EVPN uses Type-5 IP Prefix Routes. This is where EVPN becomes a full L3VPN replacement.

Use cases for Type-5:
• Advertise an entire subnet (10.2.2.0/24) instead of individual /32 host routes
• Redistribute routes from external (non-EVPN) networks into the fabric
• Leak routes between VRFs (tenant route leaking)
• Connect EVPN fabric to WAN/internet

Control Plane Comparison

MPLS L3VPN Control Plane

  • IGP (OSPF/ISIS) for underlay reachability
  • LDP or RSVP-TE to build LSPs (transport labels)
  • MP-BGP VPNv4 (AFI 1, SAFI 128) for VPN routes
  • PE-CE routing: static, OSPF, BGP, RIP
  • RR for VPNv4 route reflection
  • Label allocation: per-prefix or per-VRF

EVPN Control Plane

  • IGP (OSPF/ISIS) for underlay reachability
  • No LDP/RSVP — IP routing is enough
  • MP-BGP L2VPN EVPN (AFI 25, SAFI 70) for all EVPN routes
  • Host attachment: L2 (no routing protocol to host)
  • RR for EVPN route reflection (Spines)
  • VNI allocation: manual or auto per VNI/VRF
Key insight: EVPN eliminated the need for a label distribution protocol entirely. Because VXLAN encapsulation is IP-based (UDP over IP), the underlay just needs to route IP. No LFIB, no label swap tables, no LDP sessions. The IGP you already run is sufficient.

What EVPN Adds That L3VPN Doesn't Have

L2 Extension
Stretch Layer 2 VLANs across a routed IP fabric. Hosts in the same VLAN on different leaves appear on the same broadcast domain. L3VPN only routes — it cannot bridge.
ARP Suppression
VTEPs answer ARP locally using MAC/IP bindings from BGP Type-2. Eliminates ARP broadcast storms across the fabric. L3VPN has no L2 visibility — cannot do this.
Standards-Based Multi-Homing
Type-1 and Type-4 routes provide per-VLAN DF election, aliasing, and sub-second mass withdrawal. No proprietary MLAG needed. L3VPN has no equivalent for L2 multi-homing.
Integrated L2+L3
Single BGP control plane handles both L2 MAC routes and L3 IP routes (Type-5). IRB does routing without external gateways. L3VPN requires separate L2 control planes (VPLS, etc.).
No STP
EVPN fabric has no loops at L2 (it's an IP underlay). No Spanning Tree Protocol needed. Multi-homing convergence is handled by Type-1 BGP, not STP timers (30s+).
VTEP Auto-Discovery
Type-3 IMET routes let VTEPs discover each other per-VNI automatically. No manual flood lists. In L3VPN, PE-to-PE paths are pre-provisioned or auto-discovered differently.

EVPN over MPLS: The BGP control plane is the same (AFI 25, SAFI 70), but the data plane uses MPLS labels instead of VXLAN. The VNI is encoded as an MPLS label. Service providers deploying this get all EVPN features (multi-homing, ARP suppression, integrated L2+L3) with their existing MPLS infrastructure.
EVPN over SRv6 — the emerging unified data plane: EVPN can also run over SRv6 (Segment Routing over IPv6) instead of VXLAN or MPLS. In this model, SRv6 SIDs (Segment Identifiers — 128-bit IPv6 addresses) replace VXLAN VNIs and MPLS labels as the data-plane encapsulation. The BGP EVPN control plane (AFI 25, SAFI 70) remains unchanged — only the encapsulation extended community changes to signal SRv6. This combines EVPN's rich L2/L3 control plane with SRv6's programmable, source-routing forwarding. It is emerging in service provider networks that want a single unified SRv6 data plane for both L3VPN and EVPN services — eliminating the need to maintain separate MPLS and VXLAN fabrics.

Troubleshooting Common Issues

What to check when things don't work

Underlay Connectivity

Before investigating EVPN control plane issues, verify that the IP underlay is healthy. EVPN depends entirely on the underlay for VTEP-to-VTEP reachability.

  • Verify IGP adjacencies are up on all links (OSPF neighbors / IS-IS adjacencies)
  • Ping between VTEP loopback addresses — these are the BGP next-hops and VXLAN tunnel endpoints
  • Check MTU end-to-end: VXLAN adds 50 bytes of overhead. Test with a large ping using the DF (Don't Fragment) bit to detect any underlay link with insufficient MTU
MTU test: ping -s 1550 -M do <remote-VTEP-loopback> — if this fails but a normal ping succeeds, an underlay link has MTU below 1600. Find it with tracepath or per-hop MTU probing.

BGP EVPN State

Once underlay is healthy, verify the BGP EVPN control plane:

  • BGP neighbor status: Is the session established? Confirm both peers negotiated AFI 25 / SAFI 70 (L2VPN EVPN). A session can be Up for IPv4 unicast but not EVPN if the address family was not activated.
  • Type-3 (IMET) routes present: Each VTEP participating in a VNI advertises one Type-3 route. If you see Type-3 routes from all expected VTEPs, VTEP discovery is working correctly.
  • Type-2 routes for expected MACs: When a host is active, the local VTEP should have a Type-2 route for its MAC. Remote VTEPs should have received and installed it. If Type-2 is missing on the remote side, check RT import policy.
