Why Data Center Fabrics?

The architectural shift from legacy 3-tier to modern DC fabrics — and why it matters

Why Data Center Fabrics?

Modern applications — cloud-native microservices, containerized workloads, distributed databases — fundamentally changed how traffic flows inside a data center. Networks built for the client-server era collapse under these demands. DC fabrics are the architectural response.

Cloud Scale
Thousands of servers in a single DC, millions of flows per second, horizontal scaling requiring any-to-any connectivity at line rate
Virtualization
VMs migrate across physical hosts; the network must stretch L2 domains without spanning tree or physical re-cabling
Containers & Microservices
A single application request fans out to dozens of backend pods — each hop between pods is East-West traffic that never leaves the DC
Multi-Tenancy
Shared physical infrastructure must carry isolated traffic for hundreds of customers, each with overlapping IP space — just like L3VPN in the WAN
Operational Agility
Workloads must be provisioned in seconds, not change-window weeks. Fabric automation replaces manual VLAN trunk configs
Resilience & Scale-Out
No single point of failure, no STP convergence timer, add capacity by bolting on another leaf — no redesign required

Problems with Traditional DC Architecture

Root Cause: Traditional DCs were built on L2 Ethernet with STP as the control plane. Every limitation below traces back to this fundamental choice.
1
Spanning Tree Protocol (STP)

STP blocks redundant links to prevent loops — deliberately wasting half your bandwidth. Convergence takes 30–50 seconds (classic STP) or 1–5 seconds (RSTP in production), but any convergence causes traffic loss. In a fabric where every link is redundant by design, STP is an existential problem.

2
VLAN Scale Limit (4094)

802.1Q VLAN IDs are 12 bits = 4094 usable. A large cloud DC needs millions of tenant segments. VXLAN solves this with a 24-bit VNI (16M segments), but traditional L2 fabrics are structurally limited.

3
Oversubscription Ratios

Legacy aggregation layers were designed for 20:1 oversubscription — fine when 80% of traffic exits the DC. With 80% East-West, you need near 1:1 on fabric links. Traditional chassis switches cannot deliver this economically.

4
Single Points of Failure

The aggregation layer is typically a pair of chassis switches. Losing one cuts capacity in half; losing both partitions the DC. MLAG/vPC helps but introduces complex dual-active state machines with their own failure modes.

5
MAC Table Exhaustion

Aggregation switches must learn MAC addresses for every host in every VLAN. At 100K+ VMs, ASIC MAC tables overflow. VXLAN with BGP EVPN distributes MAC/IP binding via control plane, eliminating data-plane flooding.

Leaf-Spine Deep Dive

Clos network mathematics, oversubscription ratios, and physical topology design

Clos Network Mathematics

The leaf-spine topology is a 2-stage Clos network, invented by Charles Clos at Bell Labs in 1953 for telephone switching. The math guarantees non-blocking connectivity.

Clos Non-Blocking Condition: A Clos network is strictly non-blocking when the number of middle-stage switches M ≥ (2N − 1), where N is the number of inputs per input-stage switch. In DC fabric terms: number of spines ≥ (2 × server ports per leaf − 1) ÷ port-density factor. In practice: match uplink count to desired oversubscription ratio.
Clos Port Formula — Scope: The formula total fabric-facing ports per leaf = n × m applies where n = number of downlink (server-facing) ports per leaf and m = number of spines. This gives the total number of fabric-facing (uplink) ports across all leaves. For a non-blocking (non-oversubscribed) fabric, each leaf needs exactly m uplinks — one to every spine (1:1 uplink-to-spine ratio). In non-square fabrics (where downlink count ≠ uplink count), the oversubscription ratio is n:m. Always define whether "ports" refers to downlinks, uplinks, or total switch ports when applying Clos math.

Max hosts = leaf count × server ports per leaf. Reference: 4 leaves × 48 ports = 192 servers. At full scale (64 leaves × 48 ports) = 3,072 servers with 2 spines. Non-blocking spine count depends on the uplink-to-downlink speed ratio: spines needed = (downlink count x downlink speed) / uplink speed. For example, with 48x25G downlinks and 100G uplinks, you need 12 spines for non-blocking — not 48 or 64. The "64 spines" figure only applies when each uplink matches a single downlink speed (e.g., 1:1 port speed ratio).

3
Uplink:Downlink Ratio Formula

Oversubscription = total host-facing bandwidth ÷ total uplink bandwidth. For example, a 48×10G leaf with 6×40G uplinks: 480G ÷ 240G = 2:1 oversubscription. More generally: (downlink ports × downlink speed) ÷ (uplink ports × uplink speed). For 1:1: uplinks = downlinks × (downlink_speed / uplink_speed). With 48 × 25G down: need 48 × 25 / 100 = 12 × 100G uplinks = 12 spines for non-blocking.

Leaf Roles in the Fabric

Not all leaves are equal. The reference topology demonstrates three distinct leaf roles — each analogous to a PE router variant in the SP world.

Compute Leaf Leaf-1, -2, -3
Connects servers directly. Terminates VXLAN tunnels as VTEP. Hosts VRFs for tenant routing. Distributes host MACs and IPs via BGP EVPN Type-2 routes. Analogous to a standard PE in L3VPN.
Border Leaf Border-L4
Connects DC fabric to external networks — WAN, internet, SR-MPLS backbone. Re-originates EVPN routes as BGP-LU or VPNv4 toward the WAN. Analogous to ASBR or inter-AS PE. In reference topology: connects to SR-MPLS WAN.
Service Leaf Optional
Connects shared services: firewalls, load balancers, IDS/IPS. Traffic is steered via EVPN route-map or policy to traverse the service leaf. Provides centralized security without hairpinning through a separate physical firewall cluster.
Storage Leaf Optional
Dedicated leaf for NVMe-oF, iSCSI, or NFS storage arrays. May use lossless Ethernet (PFC + ECN) for storage protocols. Separate VRF isolates storage traffic. RoCEv2 workloads require careful QoS configuration.
DCI Leaf Optional
Data Center Interconnect leaf — extends fabric across dark fiber or DWDM to a remote DC. Uses EVPN Type-5 prefixes for IP stretch or Type-2 for L2 stretch. Equivalent of an inter-AS option-B ASBR.
Spine (Transit-only)
No server attachments, no VRFs, no VTEP function. Pure IP forwarding — exactly like a P router in MPLS. Holds full BGP table of loopback /32s only. In eBGP overlay designs, spines do not participate in EVPN signaling. In iBGP designs, spines serve as route reflectors for the EVPN address family.
Reference Topology: Border-L4 peers with the SR-MPLS WAN cloud. It imports EVPN Type-5 routes from the fabric and re-advertises them as VPNv4 toward the WAN PE. Inbound WAN prefixes are imported into the appropriate tenant VRF and re-advertised as EVPN Type-5 into the fabric.

BFD: Sub-Second Failure Detection

BFD Timers
Tx/Rx interval: 50ms. Multiplier: 3. Detection time: 150ms. Every spine-leaf link runs BFD independently.
BGP + BFD
BGP default timers are keepalive 60s / hold 180s. DC fabric recommendation: keepalive 3s / hold 9s (keepalive = 1/3 of hold time). BFD shortcircuits even these — link loss detected in 150ms, BGP session torn down, routes withdrawn, ECMP reconverges in <1s total.
Without BFD: silent link failure → BGP hold-timer expires (180s default, 9s with aggressive timers) → prolonged traffic black-holing. With BFD: 150ms detection + BGP convergence = sub-second total.

  • BFD associated with each eBGP session
  • BFD for EVPN overlay sessions (loopback-based) uses multi-hop BFD
  • Hardware BFD (line-card offload) required at scale

No MLAG / vPC Between Spine and Leaf

One of the most common questions from engineers migrating from traditional DC: "Where is the MLAG between the spines?" Answer: there is none. This is a feature, not a missing component.

Why MLAG Exists in Legacy DC: STP sees two physical links as a loop and blocks one. MLAG/vPC creates a logical port-channel across two physical switches, making STP see a single uplink. This is purely a workaround for STP — it introduces a control-plane peer link, dual-active detection, consistency checks, and a failure mode where both switches may simultaneously become active.

eBGP Underlay Design

Running eBGP as the DC underlay routing protocol — configuration, ECMP, and best practices

The modern data center fabric runs eBGP as its underlay routing protocol. This may seem surprising to WAN engineers accustomed to using BGP only at the edge — but eBGP's properties map perfectly onto Clos fabric requirements.

Why eBGP Won the DC Underlay Debate

In the early 2010s, DC architects tried running OSPF and ISIS inside large-scale Clos fabrics. They hit scaling walls. eBGP became the industry standard because it solves those problems structurally.

No Flooding Domains
OSPF/ISIS flood LSAs/LSPs to every router in the area. In a 100-leaf fabric, every topology change reaches every node. eBGP only sends incremental updates to configured peers.
No Areas to Design
Multi-area OSPF or ISIS multi-level requires careful area boundary planning. eBGP's AS topology naturally matches the Clos tier structure with zero extra design work.
Native ECMP
With unique leaf ASNs and shared spine ASNs, ECMP paths are automatically unequal-ASN-path-distinct. No route reflectors, no add-path complexity needed for basic multipath.
Operational Simplicity
Every operator knows BGP. Configuration is explicit — you see exactly which peers exist. No implicit neighbor discovery, no DR/BDR elections, no area type confusion.
Policy at Every Tier
BGP route-maps apply naturally at spine-leaf boundaries. Controlling what loopbacks are advertised, suppressing /32 host routes, aggregating per-pod — all native BGP operations.
Scales to Hyperscale
Google, Meta, Microsoft all run eBGP underlays with thousands of nodes. The protocol has proven scaling properties that IGPs simply cannot match at that scale.
WAN Engineer Analogy: Think of each leaf as a PE router with its own unique ASN, and spines as P routers (AS 65100) that transit traffic without inspecting the payload. The underlay is just exchanging loopback /32s — exactly like exchanging PE loopbacks in an MPLS core, except the "core" is now eBGP instead of ISIS+LDP. Note: spines act as P routers for the underlay. They can separately act as iBGP route reflectors for the overlay (EVPN) — a different role covered in Section 11.

ASN Assignment Strategy

The ASN design is the most important underlay decision. The key principle: unique ASN per leaf, shared ASN per spine tier.

Leaf ASNs (Unique)

Each leaf gets its own ASN from the private range. This prevents AS path loop rejection: when Leaf1 (AS 65001) advertises its loopback to spines, the spines re-advertise to other leaves. Other leaves see AS path 65100 65001 — Leaf1's ASN appears only in paths at remote leaves, never back at Leaf1 itself. For ECMP: Leaf1 receives Leaf2's loopback via two paths — Spine-1 and Spine-2 — both with identical AS path 65100 65002. With maximum-paths 2, BGP installs both paths because they have equal AS path length, identical AS path content, and different next-hops.

  • Leaf-1: AS 65001
  • Leaf-2: AS 65002
  • Leaf-3: AS 65003
  • Border-L4: AS 65004

Spine ASNs (Shared Tier)

All spines in the same tier share one ASN. This is critical for ECMP: with shared spine ASN (65100), a leaf receives identical AS paths (65100 65002 via both spines). Standard BGP ECMP requires paths to have identical AS path content, not just equal length. If spines had unique ASNs (65101/65102), paths would be 65101 65002 and 65102 65002 — same length but different content. Most implementations would then require bgp bestpath as-path multipath-relax to treat these as ECMP candidates. Shared spine ASN avoids this complexity entirely.

  • Spine-1: AS 65100
  • Spine-2: AS 65100
  • Super-Spine tier (if present): AS 65000
The allowas-in Question: With unique leaf ASNs and shared spine ASNs, you do NOT need allowas-in. A leaf advertising 10.0.0.1/32 (its own loopback) up to the spines — the spines then advertise to other leaves. Those other leaves see AS path 65100 65001. No loop. The originating leaf's ASN only appears in the path at other leaves, not back to itself. Note: allowas-in permits a router to accept routes containing its own ASN — used in hub-and-spoke or migration scenarios, but generally avoided in leaf-spine fabrics due to loop risk.
4-Byte ASN Option: For large fabrics or to avoid coordinating with existing private ASN allocations, use 4-byte private ASNs: 4200000000–4294967294 (RFC 6996). Each leaf gets a unique 4-byte ASN. Spines share one. Operationally identical to 2-byte, just larger numbers.

By default, BGP installs only the best path. Enable maximum-paths to install multiple equal-cost paths into the FIB. This is how traffic load-balances across both spines.

Key Insight: Leaf-1 receives 10.0.0.2/32 (Leaf-2 loopback) via two paths: next-hop 10.1.1.1 (Spine-1) and next-hop 10.1.1.3 (Spine-2). Both paths have AS path length 2 (65100 65002). With maximum-paths 2, both are installed. ECMP hashing distributes flows across both spines.
4
Route Advertisement Policy — Loopbacks Only

The underlay only needs to carry loopback /32 reachability. PtP link prefixes (/31s) are NOT advertised — they are only used as BGP peer addresses. This keeps the BGP table small and prevents unnecessary churn from link flaps affecting routing on other nodes.

What gets advertised: Each node advertises only its loopback /32 into eBGP. Nothing else. Use a prefix-list or route-map to enforce this outbound on every BGP session.
What does NOT get advertised: /31 PtP links, management prefixes, connected routes other than loopback, any overlay (VXLAN/EVPN) routes. The underlay BGP table is tiny: each leaf learns 5 BGP routes (3 other leaf loopbacks + 2 spine loopbacks); each spine learns 5 (4 leaf loopbacks + the other spine's loopback, re-advertised by leaves). No /31 PtP noise.

OSPF/ISIS Alternative Underlay

When and how to use IGP-based underlays in DC fabrics

While eBGP is the modern DC standard, IGP-based underlays (OSPF or ISIS) remain valid and sometimes preferable. Understanding when to choose IGP — and which IGP — requires knowing their tradeoffs against eBGP in the fabric context.

When IGP Makes Sense

Small Fabrics (< 20 nodes)
In a 4-leaf/2-spine fabric, the flooding domain is tiny. OSPF or ISIS convergence is essentially instantaneous. The operational simplicity of not managing ASNs may outweigh eBGP's theoretical advantages.
SP-Managed DCs
Service provider teams managing DC infrastructure often have deep ISIS expertise from their WAN. Running a consistent protocol across WAN and DC reduces operational friction and tooling requirements.
WAN Integration
When the DC fabric is SR-MPLS and must extend a single ISIS topology to the WAN PE (e.g., BGP-free core or seamless MPLS), running ISIS in the DC avoids redistribution complexity at the DC edge.
Existing Infrastructure
Brownfield expansion of an existing campus or enterprise DC already running OSPF. Introducing eBGP requires ASN management and potentially new automation tooling that may not be justified.
WAN Engineer Note: If your team runs Segment Routing with ISIS on the WAN and you're building a DC that will integrate with that domain, running ISIS in the DC as well eliminates a protocol boundary. The fabric becomes an extension of your SR domain.

OSPF Underlay Design

OSPF is familiar to most enterprise engineers. The main choice is area design: single Area 0 for small fabrics, or area-per-pod for medium fabrics to contain flooding.

Single Area 0 (Small Fabric)

All nodes in Area 0. Simple. Works fine for fabrics up to ~20 nodes. Every topology change floods to every router, but at small scale this is negligible.

  • All spines: Area 0
  • All leaves: Area 0
  • No ABRs, no type-3/5 LSAs
  • SPF runs on full topology
  • Recommended for: ≤ 4 spines, ≤ 16 leaves

Area-per-Pod (Medium Fabric)

Spines in Area 0, each pod's leaves in an NSSA (Not-So-Stubby Area). Reduces SPF scope per pod. Spines become ABRs. Increases design complexity but improves scale.

  • Spines: Area 0 (backbone)
  • Pod-1 leaves: Area 1 (NSSA)
  • Pod-2 leaves: Area 2 (NSSA)
  • Spines act as ABRs
  • Recommended for: > 16 leaves, multi-pod
OSPF Network Type: Always use network point-to-point on /31 PtP links. This eliminates DR/BDR election and the type-2 LSA it generates, reducing LSA count and simplifying troubleshooting.

ECMP Hash Deep Dive

ECMP in the underlay ensures traffic distributes across both spines. The hash function must provide sufficient entropy to spread flows without polarization.

5-Tuple Hash (Standard)
Hash(Src IP, Dst IP, IP Proto, Src Port, Dst Port). Most DC hardware uses this. For VXLAN, the outer UDP source port provides inner-flow entropy — different inner flows get different outer src ports.
VXLAN Entropy Source Port
RFC 7348 recommends setting VXLAN outer UDP src port to a hash of the inner frame's 5-tuple. This means the outer 5-tuple hash at the spine naturally distributes inner flows — even though all VXLAN flows appear to go between the same two VTEP IPs.
Hash Polarization Problem
If two consecutive ECMP stages use the same hash algorithm and same inputs, flows that were split at Leaf-1 may reconverge at Spine-1 and re-split identically at Spine-2. This is called polarization. Mitigate by using different hash seeds per stage.
Flow Affinity (Consistency)
All packets in a TCP flow must take the same path to avoid reordering. The 5-tuple hash guarantees this — the same 5-tuple always hashes to the same spine. Flow affinity is preserved through ECMP even during normal operation.
Verify ECMP distribution in production: run show ip route 10.0.0.2 detail on Leaf-1 — you should see two next-hops with equal weight. Use traffic generators to confirm both spine links carry load across many flows.
Why Overlays?

L3 fabric solves the underlay — but servers still need L2. Overlays bridge that gap with a familiar WAN pattern.

The Problem Space

A pure L3 routed fabric is operationally clean and scales well. But the server world has not caught up: VMs need to live-migrate across racks without changing IP addresses, tenants need isolated L2 segments, and the same VLAN must span multiple ToR switches. These are inherently L2 requirements sitting on top of an L3 network.

Core tension: The underlay wants everything routed. The overlay wants everything flat. Tunneling resolves this — stretch L2 as a service on top of L3 infrastructure, exactly as MPLS stretches L2/L3 VPNs over an IP core.
Problem 1
Server-A (Leaf1, VLAN 10) and Server-B (Leaf2, VLAN 10) need to be on the same broadcast domain — but there are 3 routed hops between their ToRs.
Problem 2
Tenant-A and Tenant-B share the same physical fabric. Their traffic must be completely isolated end-to-end, including at the control plane.
Problem 3
Workload mobility: Server-A's VM migrates from Leaf1 to Leaf3 without changing 192.168.10.10. The fabric must follow the move.

Overlay = Tunnel + Control Plane

An overlay is not just encapsulation. It requires two components working together:

Data Plane
VXLAN encapsulation tunnels the original L2 frame inside a UDP/IP packet across the routed underlay. The VTEP adds and removes headers at each edge.
Control Plane
BGP EVPN distributes MAC/IP binding information so VTEPs know where each endpoint lives — without flooding unknown unicast.
Separation of Concerns
Underlay BGP carries only loopback reachability. Overlay BGP (EVPN) carries tenant MAC/IP state. They run in parallel and do not pollute each other.
Multi-Tenancy
Each tenant gets its own VNI namespace. L2VNI per segment, L3VNI per VRF. Traffic between tenants never mixes, mirroring VPN label separation in MPLS.

VNI Types: L2VNI vs L3VNI

VNIs serve two distinct purposes. Confusing them is the most common mistake for engineers new to DC fabric design.

L2VNI — Bridge Domain
Bridging

Extends a single VLAN (broadcast domain) across multiple VTEPs. Traffic stays within the same subnet. MAC learning is suppressed at the hardware level — BGP EVPN distributes MAC/IP bindings instead.

In our topology:
VNI 10010 → VLAN 10, Tenant-A
VNI 10020 → VLAN 20, Tenant-A

Server-A (Leaf1) and Server-B (Leaf2) share VNI 10010. Their frames are tunneled between VTEPs as if on the same switch.
Analogous to: VPLS service in MPLS

Routing

Enables inter-subnet routing between VTEPs within the same tenant VRF (symmetric IRB model). Asymmetric IRB does not use L3VNI — it routes locally and bridges to the destination VLAN’s L2VNI instead. When a packet must be routed (different subnets, different VLANs), it is encapsulated with the L3VNI rather than the L2VNI at the ingress VTEP.

BGP EVPN Control Plane

EVPN replaces flood-and-learn with a proper control plane. If you know MP-BGP L3VPN, you know 80% of EVPN already.

Why EVPN? The Problems with Flood-and-Learn

Traditional VXLAN (no EVPN):

BUM flooding: Unknown unicast, broadcast, multicast traffic must be flooded to all VTEPs in the segment. This requires either multicast in the underlay (complex) or ingress replication to every VTEP (bandwidth waste at scale).
Data-plane MAC learning: VTEPs learn remote MAC addresses by snooping on received traffic. Until a MAC is learned, every frame to that destination is flooded. In a VM environment with thousands of endpoints, this causes significant BUM traffic.
No ARP suppression: ARP requests are broadcast. In a segment with thousands of VMs, ARP storms can saturate the overlay. Every VTEP must flood ARP to all others.

EVPN solves all three:

Control-plane MAC learning: BGP EVPN Type-2 routes advertise MAC/IP bindings proactively. A VTEP knows where every MAC lives before the first packet arrives.
ARP suppression: The ingress VTEP (Leaf1) holds a BGP-learned ARP table. When Server-A ARPs for Server-B, Leaf1 responds locally — the ARP never hits the overlay. Eliminates broadcast amplification.
Selective BUM: EVPN Type-3 routes build an ingress replication list per VNI. Flooding is still possible but targeted only to VTEPs that have active members in the segment.

RD, RT, and the L2VPN EVPN Address Family

EVPN reuses the MP-BGP machinery of L3VPN wholesale. If you have configured VPNv4, the concepts are identical.

Route Distinguisher (RD)

Makes EVPN NLRIs globally unique in the BGP table. Each VTEP assigns an RD per VRF or per EVI (EVPN Instance). Common format: loopback-ip:VNI — e.g., Leaf1 Tenant-A: 10.0.0.1:10010.

Same purpose as L3VPN RD. Two VTEPs may have the same MAC address in their local tables — the RD makes these routes distinct in the BGP RIB so the route reflector does not collapse them. Auto-derived RD = <Router-ID>:<VNI> or <Loopback-IP>:<VLAN-ID>, depending on vendor. Auto-RD ensures uniqueness without manual configuration.
Route Target (RT)

Controls import/export: which VTEPs import which EVPN routes. Common format: ASN:VNI — e.g., 65000:10010 for VLAN 10 Tenant-A. All VTEPs participating in VNI 10010 export and import this RT.

Same purpose as L3VPN RT. Leaf1, Leaf2 both export RT 65000:10010. Both import it. Leaf3 hosts Server-C (VLAN 20, VNI 10020) in Tenant-A — it exports and imports RT 65000:10020 (L2VNI) and RT 65000:50001 (L3VNI) for Tenant-A, just like Leaf1 and Leaf2 participate in Tenant-A with their respective L2VNIs.
Address family configuration: EVPN runs as address-family l2vpn evpn under BGP. This is a separate AF from IPv4/IPv6 unicast (underlay) and from VPNv4 (if present). The underlay BGP sessions carry IPv4 unicast (loopbacks). The overlay BGP sessions carry L2VPN EVPN. These can be the same session (both AFs activated) or separate sessions — vendor and design dependent.

BGP Session Topology: Underlay + Overlay

Two independent BGP session meshes coexist. Understanding which session carries what is essential for troubleshooting.

Option A: iBGP Overlay with Spine as Route Reflector

The underlay uses eBGP with per-leaf ASNs (65001–65004) peering with spine ASN (65100). Separately, the overlay runs iBGP EVPN sessions where all nodes share a common overlay ASN (65100). Leaves configure local-as 65100 (or use the spine ASN as their iBGP AS) for the overlay sessions only. This is the most common deployment model.

Option B: eBGP Overlay (same AS as underlay)

Multi-Homing: ESI, DF Election & All-Active Forwarding

When a server connects to two leaves (LACP bond), both leaves must appear as one logical attachment point. EVPN uses the Ethernet Segment Identifier (ESI) — analogous to a pseudowire group ID in MPLS — to coordinate between PEs attached to the same CE.

ESI Concepts
ESI
10-byte identifier shared by all leaves connected to the same CE/server. Derived from LACP system MAC or configured manually.
Type-1 Route
Ethernet Auto-Discovery route. Advertised per-ESI and per-ES/EVI. Used for fast convergence and aliasing.
Type-4 Route
Ethernet Segment route. Peers discover each other and elect the Designated Forwarder using this route.
DF Election
Only the Designated Forwarder sends BUM (flood) traffic to the CE. Prevents duplicate delivery. All-active = both forward unicast.
All-Active Forwarding Steps
1
Server bonds to Leaf-1 and Leaf-2

Both leaves are configured with ESI = 00:01:01:01:01:01:01:01:01:01. Server sends LACP on both uplinks. Leaves advertise Type-4 routes to each other.

2
DF Election (per EVI)

Leaves exchange Type-4 ES routes containing their loopback IPs and are sorted in ascending IP order. For each EVI/VLAN V with N leaves in the ES, the DF is the leaf at ordinal position (V mod N). This distributes BUM forwarding responsibility across VLANs — different VLANs have different DFs on the same ES, achieving load balance. Example: with Leaf-1 (10.0.0.1) and Leaf-2 (10.0.0.2), N=2. For VLAN 10 (EVI 10): 10 mod 2 = 0 → leaf at position 0 = Leaf-1 is DF. For VLAN 20 (EVI 20): 20 mod 2 = 0 → Leaf-1 again. For VLAN 11 (EVI 11): 11 mod 2 = 1 → Leaf-2 is DF. This ensures load is spread across leaves for different EVIs.

3
Unicast aliasing — remote leaves load-balance

Remote Leaf-3 sees Type-1 aliasing routes from both Leaf-1 and Leaf-2 for the same ESI. ECMP hashes to either VTEP. Both leaves accept and forward — no MAC move needed.

4
Split-horizon: no local loops

If Leaf-1 receives a BUM frame from Leaf-3 destined to the ES, it does NOT forward to the bonded server if it's a non-DF. Prevents dual delivery.

5
Mass-withdrawal on link failure

On link failure, the affected leaf withdraws its per-ESI Type-1 route (Ethernet A-D per-ES). Remote leaves immediately remove that VTEP from their ECMP set for ALL MACs behind that ESI — no individual per-MAC Type-2 withdrawals needed. This is O(1) convergence regardless of how many MACs are behind the segment.

MPLS Parallel: ESI multi-homing in EVPN is analogous to VPLS multi-homing with PW redundancy groups, but operates via BGP signaling rather than LDP. DF election replaces the "active/standby" PW selection that was manual in legacy VPLS.
MAC Move Detection: MAC move detection identifies hosts that migrate between ports/VTEPs. Repeated moves trigger MAC dampening to prevent control-plane churn from loops or misconfigurations.

RT Design: Auto-Derived vs Manual

Route Targets control which VRFs import which routes. Consistent RT assignment across all leaves is critical — a mismatch means routes silently drop.

Auto-Derived RT
RT = AS:VNI — automatically computed from BGP ASN and VNI
! Leaf-1 auto-RT for L2VNI 10010:
RT = 65001:10010
! Leaf-2 auto-RT for L2VNI 10010:
RT = 65002:10010
Problem: Different ASNs per leaf → different RTs → routes NOT imported
Auto-RT only works when all leaves share the same ASN (iBGP model). In eBGP spine-leaf (different ASN per leaf), auto-RT produces different values per leaf — use manual RT instead.
Manual RT (Recommended for eBGP)
RT explicitly configured — identical on all leaves for same VNI

 rd auto

 route-target export 65000:10010
! ALL leaves for L3VNI 50001 (VRF):

 route-target import evpn 65000:50001

  • No VNI policy or filtering at boundary
Failure Domain: One flat L2 domain across both DCs. A rogue VM sending BUM traffic in DC-1 floods DC-2. Use only for tightly controlled, low-latency DCI (sub-5ms).

L2 Stretch: When To (and Not To)

Avoid L2 stretch unless required

L2 across DCs merges failure domains. A broadcast storm, STP topology change, or ARP flood in DC-1 directly impacts DC-2. Most workloads do NOT require L2 stretch.
Do NOT stretch L2 when:
  • Application can handle an IP address change on failover
  • Load balancers or proxies sit in front of servers
  • Kubernetes / container workloads (re-IP is native)
  • WAN latency > 10ms (ARP timeouts, STP issues)
L2 stretch may be justified when:

Legacy applications with hardcoded IPs, active/active clustering (MSCS, Oracle RAC), live VM migration requiring same L2 segment, or specific compliance/licensing tied to MAC address.
If you MUST stretch L2:
  • Use EVPN Multi-Site BGW (Option B) — not back-to-back VXLAN
  • Suppress ARP flooding: enable ARP proxy at each site BGW
  • Rate-limit BUM traffic at BGW ingress
  • Keep stretched VNIs to minimum — never stretch /16 subnets

Common Failure: MTU Black-Hole

Symptom: Small pings work, large file transfers fail or are very slow. VXLAN tunnels show as UP but application traffic drops. TCP sessions established but stall.
1
Confirm MTU Black-Hole

Ping with DF-bit set and increasing sizes. Find the breaking point: ping <vtep> size 1472 df-bit (success) vs ping <vtep> size 1500 df-bit (fail). Breaking point identifies where the MTU mismatch is.

2
Locate Undersize Interface

Check MTU on each hop: leaf uplinks, spine downlinks, inter-spine links. One interface at 1500 in a jumbo-frame fabric will silently drop large VXLAN-encapsulated packets.

3
Verify ICMP Unreachable Path

If ICMP "fragmentation needed" messages are blocked (as in many DC environments), TCP PMTUD fails silently. Enable ICMP unreachable on transit interfaces or apply TCP MSS clamping on border interfaces.

4
Fix and Re-Test

Set all fabric interfaces to MTU 9216. Set server NICs to MTU 9000 (uniform fabric-wide MTU simplifies operations; the 50-byte VXLAN overhead is handled on fabric-facing links, not access ports). Re-run jumbo ping sweep: sizes 1500, 4000, 8972 — all should pass.

Common Failures: EVPN Control Plane Issues

EVPN RT Mismatch
Symptom: BGP EVPN sessions up, but no routes exchanged between certain VNIs. MAC/IP routes present on source leaf but absent on remote leaf.
1
Verify RT Values

Show EVPN route on source: confirm export RT. Show EVPN import config on destination: confirm import RT. They must match exactly — e.g., "65000:10010" must equal "65000:10010", not "65001:10010".

2
Check Auto-RT Config

Auto-derived RT = AS:VNI. If auto-RT is configured on some leaves and manual RT on others, they will not match. Standardize: either all-auto or all-manual across the fabric.

3
Fix and Soft-Reset

After correcting RT config, issue BGP soft-reset to re-advertise routes: clear bgp l2vpn evpn soft. Verify routes now appear in remote leaf EVPN table.

Asymmetric Routing / BUM Storm
Symptom: Traffic works in one direction only. Or: unusually high VXLAN counters on fabric links, CPU spikes on leaves, broadcast/multicast rate spike.
1
Check IRB Mode

Asymmetric routing (IRB without L3VNI) requires local ARP entry. If ARP suppression is off, return traffic may flood. Verify symmetric IRB is configured: both L2VNI and L3VNI defined on all leaves hosting a VNI.

2
Verify ARP Suppression

ARP suppression uses EVPN Type-2 MAC/IP routes to answer ARP locally. If disabled, every ARP floods all VTEPs as BUM. Check: show vxlan arp-suppression — table should be populated.

3
BUM Rate Limiting

Apply BUM storm control on VNI — limit unknown unicast/broadcast/multicast rate. Check for any rogue VMs sending gratuitous ARP floods: show vxlan bum statistics vni 10010.

VXLAN Tunnel Troubleshooting
0
Check VXLAN Source-Interface (NVE)

Verify VXLAN source-interface (NVE) is configured and up — this is the VTEP IP used for all tunnel endpoints. Run show vxlan interface or show nve interface. If the source-interface loopback is down or misconfigured, no VXLAN tunnels will form regardless of BGP state.

BGP Session Drop Flowchart
1
Identify the Drop Reason

Check BGP notification message in logs: show bgp neighbors | include Notification. Common codes: HOLD_TIMER_EXPIRED (keepalive issue), UPDATE_MSG_ERROR (route attribute mismatch), OPEN_MSG_ERROR (capability negotiation fail).

2
Timer Mismatch

If HOLD_TIMER_EXPIRED: verify keepalive/hold timers match on both peers. DC fabric typically uses aggressive timers (3s/9s) for fast failover — ensure both sides agree. BFD is preferred for sub-second detection.

3
EVPN Capability Mismatch

Verify both peers negotiate L2VPN EVPN address family in BGP OPEN. If one side lacks EVPN capability, session may establish for IPv4 but no EVPN routes exchanged. Check: show bgp neighbors | include AFI.

4
Route Policy Filtering

Check if route-map or prefix-list is dropping EVPN routes. Common mistake: a catch-all deny in an IPv4 route-map also affects EVPN neighbors sharing the same peer-group. Verify with: show bgp l2vpn evpn neighbors <ip> policy.
