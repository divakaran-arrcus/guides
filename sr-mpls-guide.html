Why SR-MPLS?

The case for replacing LDP and RSVP-TE with Segment Routing

The Problem with LDP

Core Issue: LDP creates per-prefix LSP state on every router in the network. As the network grows, state grows with it — on every single node.

If you've run LDP in production, you know the operational pain. Every router maintains a full table of Label FEC (Forwarding Equivalence Class) bindings for every prefix in the network. With 1000 PE routers, every transit router holds LDP sessions with its directly connected neighbors (typically a handful) but must maintain FEC-label bindings for all 1000+ loopback prefixes distributed by LDP.

LDP State Per Transit Router

1000 PE network:
• LDP sessions with each neighbor (2-8 typical)
• 1000+ FEC bindings per router
• Hello adjacencies per neighbor
• Keepalive timers per session
• Label database × 1000 entries

Total: ~1,000,000 state entries network-wide just for PE loopbacks

SR-MPLS State Per Router

1000 PE routers:
• 0 LDP sessions
• 1000 LFIB entries (from IGP)
• IGP already running
• No extra sessions
• No extra timers

Total: 1000 LFIB entries per router, computed from existing IGP

LDP's Four Fundamental Problems

Problem 1 No Traffic Engineering
LDP is hop-by-hop, following IGP shortest paths only. You cannot steer traffic over a specific path. If you want TE, you must run RSVP-TE alongside LDP — two separate control planes.
Problem 2 No Built-in FRR
LDP has no fast reroute mechanism. You must deploy RSVP-TE FRR tunnels alongside LDP for sub-50ms recovery. This means running both protocols simultaneously.
Problem 3 IGP-LDP Sync Issues
After a link comes up, IGP converges faster than LDP. Traffic can black-hole during the window where IGP is using the link but LDP hasn't yet established labels. Requires LDP-IGP synchronization (RFC 5443) as a workaround.
Problem 4 Session Scaling
LDP uses targeted LDP sessions for remote neighbors (e.g., for inter-area LSPs). This adds yet more session state. Large networks with inter-area TE can have thousands of targeted LDP sessions.

The Problem with RSVP-TE

Core Issue: Full-mesh RSVP-TE requires n×(n-1)/2 tunnels and per-tunnel state on every transit router along every path.

RSVP-TE solved LDP's TE problem but created a scaling nightmare. Every transit router along an LSP must maintain PATH and RESV state for that LSP. The head-end must know the full topology to compute explicit paths. And to provide protection, you double the state.

The Head-End Knows Everything Problem

For RSVP-TE to compute an explicit path, the head-end PE must have a full, accurate view of the network topology including link bandwidth utilization. This requires OSPF-TE or ISIS-TE with TE extensions, CSPF computation, and a controller that can update paths when utilization changes. Any topology inaccuracy causes suboptimal or failed path computation.

!
Transit Router State is Proportional to LSPs Through It

A core router in a 100-PE network may have thousands of RSVP LSPs passing through it. Each LSP requires a PATH state entry and a RESV state entry. Memory consumption is significant. Any software restart or process crash on a transit router must restore all this state.

RSVP-TE Requirements

  • TE database (OSPF-TE or ISIS-TE)
  • CSPF at head-end
  • RSVP state on every transit hop
  • Bandwidth reservation database
  • Periodic RSVP refresh messages
  • Make-before-break for re-optimization
  • FRR: facility bypass or detour tunnels

SR-MPLS (SR-Policy) Requirements

  • IGP already running
  • No per-LSP state on transit
  • Path encoded in label stack at head-end
  • No bandwidth reservation (use flex-algo or metrics)
  • No refresh messages
  • TI-LFA provides automatic FRR
  • SDN controller optional, not required

What SR-MPLS Changes

Key Insight: SR-MPLS is source routing — the ingress node encodes the entire path in the label stack. Transit nodes just process the top label. No per-flow state on transit routers.
1
Ingress Encodes the Path

The ingress PE pushes a stack of MPLS labels representing the sequence of segments the packet must traverse. Transit routers don't need to know the intent — they just look at the top label, swap/pop, and forward. The "routing decision" is made once, at the edge.

2
IGP Distributes Segments — No LDP, No RSVP

ISIS or OSPF is extended with new TLVs to advertise Prefix-SIDs and Adjacency-SIDs. Every router already runs the IGP. SR just adds a few new sub-TLVs to existing LSPs. No new protocol, no new sessions, no new adjacencies to form.

3
Same MPLS Data Plane — Same Hardware

PUSH, SWAP, and POP operations are unchanged. The LFIB lookup is unchanged. SR-MPLS runs on existing MPLS-capable hardware. There is no new encapsulation format — the MPLS label header is identical.

4
Stateless Transit Routers

Transit routers (P routers) have no per-tunnel, per-LSP, or per-flow state. They only need LFIB entries for their local Prefix-SIDs and Adjacency-SIDs — a bounded, small set derived from the IGP. A transit router's LFIB complexity is O(nodes) not O(LSPs).

The Postal Analogy: With LDP, every post office (router) decides where to send the package based on the destination address — decentralized, hop-by-hop decisions. With SR-MPLS, the sender writes the full routing instructions on the envelope at the origin — the post offices just follow the instructions without needing to think.

SR-MPLS Architecture

Source routing with existing MPLS forwarding

Source Routing Paradigm

Core Principle: The ingress node (PE1) decides the complete path and encodes it as an ordered list of segments in the MPLS label stack. Every transit router processes only the top label — no path-level intelligence required at transit nodes.

LDP / Hop-by-Hop Routing

Each router independently decides the next hop for every packet. Like a postal system where each post office decides where to send the package next, based on its own routing table.

CE1→CE2:
PE1: "Destination CE2 — next hop is P1, push label 100 (P1's label for PE2)"
P1: "Top label 100 — swap to 200 (P2's label for PE2)"
P2: "Top label 200 — swap to 300 (PE2's label for PE2)"
Each router holds state for every FEC. Labels change at every hop.

SR-MPLS / Source Routing

PE1 makes all routing decisions. Transit routers just execute SWAP/POP on the top label. Like a postal system where the sender writes the full delivery route on the envelope — post offices just follow the next instruction.

CE1→CE2:
PE1: "Push label 16004 (PE2's Prefix-SID)"
P1: "Top label 16004 — SWAP 16004 (uniform SRGB, same label)"
P2: "Top label 16004 — POP (I am penultimate hop)"
PE2: "Receive unlabeled — IP lookup for CE2"
Zero per-LSP state on P1 or P2.
!
Why "Source Routing" is the Right Mental Model

In traditional networking, "source routing" (IP strict source routing) was considered dangerous and disabled everywhere. SR-MPLS implements source routing safely at the MPLS layer — the routing instructions are opaque to the IP layer, controlled by the network operator (not end hosts), and verified against the IGP topology. The ingress PE is the "source" that does the routing, not the CE or end host.

Segment Types Overview

Segments are the fundamental building blocks of SR-MPLS. Each segment is an MPLS label that represents an instruction: "forward via shortest path to this node," "forward out this specific interface," or "forward to any of these nodes."
Global Prefix-SID (Node SID)
Identifies a node or prefix globally. The same index maps to the same label on every router (with uniform SRGB). Allocated from the SRGB. Advertised in IGP. Represents the shortest IGP path to that node.

Example: PE2's Prefix-SID index=4 → label 16004. This label means "forward toward PE2 via shortest path" on every router in the domain.

Use for: Basic transport, L3VPN next-hop resolution, BGP next-hop forwarding
Local Adjacency-SID
Identifies a specific link (adjacency) on a specific router. Typically allocated from the SRLB (or dynamic label range if SRLB is exhausted). Only meaningful on the router that advertised it. Tells the router: "forward out this specific interface, regardless of IGP metric."

Example: PE1's link to P1 gets Adj-SID 15001. Pushing 15001 on PE1 forces traffic out that specific interface — even if IGP would normally prefer a different path.

Use for: Traffic engineering, explicit paths, TI-LFA repair paths
Global Anycast-SID
The same Prefix-SID index is configured on multiple nodes. Traffic is forwarded toward the nearest node advertising that SID (shortest IGP path to any of them). Provides load balancing and automatic failover.

Example: P1 and P2 both advertise Anycast-SID index=50 → label 16050. Traffic sent to 16050 goes to whichever of P1/P2 is closest.

Use for: Service nodes, firewall clusters, anycast services, inter-area TE
Key Distinction: Prefix-SIDs and Anycast-SIDs are globally significant — the same label value has the same meaning on every router. Adjacency-SIDs are locally significant — the same label value on two different routers means two completely different things. Never confuse these.

What Changed: Label Allocation

Before (LDP): Label allocated dynamically by each router, distributed via LDP sessions. Label changes at every hop.

After (SR): Label = SRGB_base + index. Index advertised in IGP. Every router computes the label independently. No sessions needed.
For Hardware Teams: No ASIC changes required. Line cards that forward MPLS today forward SR-MPLS traffic with the same microcode. The only software change is in the control plane: label allocation and LFIB programming now comes from the IGP process instead of LDP.

Control Plane: IGP Replaces LDP

ISIS and OSPF are extended with new TLVs/sub-TLVs to flood segment information. Every router computes its own LFIB entries from the Link State Database — no targeted sessions, no PATH/RESV signaling.
1
ISIS Floods Prefix-SID Advertisements

Each router advertises its Prefix-SID index in its ISIS LSP using an extended TLV (TLV 135 sub-TLV for IPv4, TLV 236 sub-TLV for IPv6). Every router receives every other router's LSP via normal ISIS flooding — no new adjacencies needed.

2
Each Router Independently Computes LFIB

Using SPF results and the Prefix-SID advertisements, each router independently computes: "To forward toward PE2 (index=4, SRGB=16000), incoming label 16004 should SWAP to 16004 (uniform SRGB) and forward out interface toward P2." No controller needed for basic forwarding.

3
Adjacency-SIDs Are Self-Announced

Each router allocates Adjacency-SIDs for its local links and advertises them in its ISIS LSP. Neighbors learn the Adj-SID value and can use it in label stacks they build. The advertising router programs the corresponding LFIB entry to forward out that specific interface.

SRGB & SRLB

Label range planning for Segment Routing

Segment Routing Global Block (SRGB)

Definition: The SRGB is a reserved range of MPLS labels from which global segment labels (Prefix-SIDs) are allocated. The SRGB base and range are configured on each router and advertised in the IGP.

SRGB in Our Topology

Base: 16000
Range: 8000 labels
Upper bound: 23999
All routers: PE1, P1, P2, PE2 — all use 16000-23999

Formula: Label = SRGB_base + index
PE1 (index=1): 16000 + 1 = 16001
P1 (index=2): 16000 + 2 = 16002
P2 (index=3): 16000 + 3 = 16003
PE2 (index=4): 16000 + 4 = 16004

SRGB Key Properties

  • Configured per-router (not globally imposed)
  • Advertised in IGP (ISIS: Router Capability TLV 242, SR-Capabilities sub-TLV 2)
  • Default on most platforms: 16000-23999 (IOS-XR, Junos)
  • Must not overlap with other label blocks
  • Indexes must be unique per router (within the SR domain)
  • Index=0 maps to the SRGB base label itself (16000). Some implementations do not support it — best practice is to avoid using index 0.
  • An index is a 32-bit value; label space limits practical range
Important: The SRGB itself is just a reservation. Labels within the SRGB are only "active" if a Prefix-SID has been configured with that index. Unused labels in the SRGB are simply not programmed in the LFIB.

Why Uniform SRGB Matters

Recommendation: All routers in an SR domain should use the same SRGB base and range. This is not technically required but has major operational advantages.

Uniform SRGB (Recommended)

All routers: SRGB = 16000-23999
PE2 index=4 → label 16004 everywhere

PE1 LFIB: 16004 → SWAP 16004, out → P1
P1 LFIB: 16004 → SWAP 16004, out → P2
P2 LFIB: 16004 → POP, out → PE2

Benefit: Label 16004 means "toward PE2" on every router. Traceroute shows the same label at every hop. Debugging is trivial.

Non-Uniform SRGB (Avoid)

PE1 SRGB: 16000-23999
P1 SRGB: 20000-28000
P2 SRGB: 16000-23999
PE2 index=4 → different labels per router!

PE1 LFIB: 16004 → SWAP 20004, out → P1 (P1's SRGB base 20000 + 4)
P1 LFIB: 20004 → SWAP 16004, out → P2 (P2's SRGB base 16000 + 4)
P2 LFIB: 16004 → POP, out → PE2 (PHP)

Problem: Each router must SWAP to the downstream neighbor's label (based on that neighbor's SRGB). More complex LFIB programming. Harder to debug. Traceroute shows different labels at different hops.

Segment Routing Local Block (SRLB)

Definition: The SRLB is a reserved label range for locally significant segments — primarily Adjacency-SIDs. Labels in the SRLB are only meaningful on the router that allocated them.

SRLB in Our Topology

Base: 15000
Range: 1000 labels
Upper bound: 15999

Each router allocates Adj-SIDs from its own SRLB independently. The same label value (e.g., 15001) on PE1 and on P1 refers to completely different adjacencies — this is correct and expected.

SRLB Key Properties

  • Local to the advertising router only
  • Advertised in IGP so neighbors can learn the Adj-SID value
  • Not globally unique — same label on different routers is fine
  • Dynamically allocated per adjacency (two per link: protected + unprotected)
  • Must not overlap with SRGB
  • Default: 15000-15999 on most platforms
!
SRLB and TI-LFA

TI-LFA (Topology-Independent Loop-Free Alternates) uses Adj-SIDs extensively for repair paths. When a link fails, the pre-computed backup path is encoded as a label stack using Adj-SIDs to "steer around" the failure. The SRLB must be large enough to accommodate all adjacencies plus the TI-LFA repair paths, which may use additional local labels.

Prefix-SID (Node SID)

Global segments identifying network nodes

What is a Prefix-SID?

Definition: A Prefix-SID is a globally significant segment that identifies a node (via its loopback prefix) or any IP prefix in the SR domain. It is represented as an index, which every router independently converts to a label using: Label = SRGB_base + index.
Global Significance
The same index means the same destination on every router in the SR domain. Label 16004 means "forward toward PE2 via shortest path" whether you're on PE1, P1, or P2.
IGP-Advertised
The index is configured on the router that owns the prefix and advertised in the IGP (ISIS TLV 135 sub-TLV, or OSPF Extended Prefix LSA). Every router floods it network-wide.
Shortest Path Semantics
A Prefix-SID always represents the shortest IGP path to that node. It does not pin a specific physical path — it follows whatever IGP computes as the best path, including ECMP.
Admin-Assigned Index
The index is manually configured by the operator (or by a controller). It must be unique within the SR domain. Convention: assign sequentially based on router role or number.
!
Node SID vs Prefix SID

A "Node SID" is a Prefix-SID associated specifically with a router's loopback (the router itself as a node). This is the most common usage. A Prefix-SID can also be assigned to any other prefix (e.g., a P2P link prefix or an aggregate), but in practice, the majority of Prefix-SIDs are Node SIDs assigned to loopback interfaces.

Index vs Label

Critical Distinction: What is configured and advertised in the IGP is the index — a small integer. What is programmed in the LFIB and carried in the MPLS header is the label — computed as SRGB_base + index. Never confuse the two.

Prefix-SID in IGP

Prefix-SID information is carried in ISIS or OSPF extensions. The index, flags, and algorithm are flooded to every router, which then independently programs its LFIB.

ISIS (RFC 8667)

TLV 135 (Extended IP Reachability): carries IPv4 prefixes
Sub-TLV 3: Prefix-SID sub-TLV
Fields: Flags (1 byte), Algorithm (1 byte), SID/Index (3 or 4 bytes)

TLV 236 (IPv6 Reachability): same structure for IPv6
Router Capability TLV (242): advertises SRGB range (SR-Capabilities sub-TLV 2)

PE2 ISIS LSP contains:
• TLV 135: 10.0.0.4/32
  • Sub-TLV 3: index=4, algorithm=0, flags=N|P

OSPF (RFC 8665)

Extended Prefix LSA (Opaque Type 7 in LSA Type 10): carries prefix + SID
Prefix-SID sub-TLV: inside Extended Prefix TLV
Fields: Flags (1 byte), MT-ID (1 byte), Algorithm (1 byte), SID (4 bytes)

Router Information LSA: advertises SRGB range

Key flags:
• N-flag: Node-SID (this SID identifies the node)
• P-flag: No-PHP (do NOT do penultimate hop popping — keep the label until the egress node)
• E-flag: Explicit-Null (use explicit null instead of PHP)
Every router receives these advertisements via normal IGP flooding. No new protocol is needed. The existing IGP LSDB now carries segment information alongside topology information.

Adjacency-SID

Local segments identifying specific links

What is an Adjacency-SID?

Definition: An Adjacency-SID is a locally significant segment that identifies a specific link (adjacency) on a router. It is typically allocated from the SRLB (or dynamic label range if SRLB is exhausted) and tells the router: "send this packet out this specific interface, bypassing IGP shortest-path selection."

Prefix-SID (for comparison)

  • Global significance
  • Allocated from SRGB (16000-23999)
  • Represents shortest IGP path to a node
  • Same label everywhere (uniform SRGB)
  • ECMP-aware (follows all equal-cost paths)
  • Computed by every router from IGP

Adjacency-SID

  • Local significance only
  • Allocated from SRLB (15000-15999)
  • Represents a specific physical link
  • Different on every router (local allocation)
  • Forces traffic out one specific interface
  • Advertised in IGP so neighbors can use it
In our topology, PE1 has one link to P1. PE1 allocates an Adj-SID (e.g., 15001) for that link. If a packet arrives at PE1 with top label 15001, PE1 sends it out the PE1→P1 interface — regardless of what IGP says the best path is. This is the building block for traffic engineering with SR-MPLS.

Why Adjacency-SIDs?

The Problem Prefix-SIDs Can't Solve: Prefix-SIDs always follow the IGP shortest path. If you need to steer traffic over a specific link — for traffic engineering, load balancing, or to avoid a congested segment — you need Adjacency-SIDs.
1
Specific Link Selection

If PE1 has two parallel links to P1 (a LAG member or two different physical paths with equal cost), a Prefix-SID will ECMP across both. An Adj-SID pins traffic to one specific physical link — critical for scenarios where you need deterministic path selection.

2
Non-Shortest Path Forwarding

Suppose you want traffic from CE1 to CE2 to go PE1→P1→PE2 (skipping P2), even though the shortest path might be PE1→P2→PE2. You can't do this with Prefix-SIDs alone. By stacking [Adj-SID(PE1→P1), Prefix-SID(PE2)], you force the first hop via the PE1→P1 link and then let P1 forward to PE2 via its shortest path.

3
TI-LFA Repair Paths

When TI-LFA pre-computes a backup path around a failed link, it often needs to specify an exact sequence of links to bypass the failure point. Adj-SIDs are the mechanism TI-LFA uses to encode these repair instructions in a label stack.

SR-Policy Building Blocks: Real-world SR-Policies (the SR-MPLS TE mechanism) are combinations of Prefix-SIDs and Adj-SIDs. A segment list in an SR-Policy might look like:
[Adj-SID(PE1→P1), Adj-SID(P1→P2), Prefix-SID(PE2)]
This fully specifies every link in the path while using a Prefix-SID for the final hop.

Anycast-SID

Shared segments for redundancy and load-sharing

What is an Anycast-SID?

Core Concept: An Anycast-SID is the same Prefix-SID index advertised by multiple nodes simultaneously. The IGP sees multiple origins for the same SID — ECMP or nearest-node selection determines which node receives the traffic.

In traditional MPLS, a label uniquely identifies a path to a single node. With Anycast-SID, multiple routers advertise the same SID index. The IGP best-path algorithm (SPF) determines which router wins — the closest one (by metric) gets the traffic. It's the MPLS-domain equivalent of anycast IP routing.

Unicast Prefix-SID

One node advertises index 1 → label 16001

Traffic always goes to PE1 regardless of topology

PE1 failure = unreachable until reconvergence

Anycast Prefix-SID

Two nodes advertise index 100 → label 16100

Traffic goes to whichever node is closest (by IGP metric)

Node failure = IGP shifts traffic to surviving node automatically

Key Property: The forwarding plane behavior is identical to a normal Prefix-SID. Only the control plane differs — multiple IGP advertisements, SPF picks the best. No new MPLS opcodes, no new hardware requirements.
IGP Behavior: When two nodes advertise the same Prefix-SID index, SPF computes the shortest path to each origin independently. If both are equidistant, ECMP occurs. If one is closer, all traffic goes to the closer node. When the closer node fails, SPF reconverges and routes to the surviving node.

OSPF SR Overview

OSPF implements SR using Opaque LSAs — a mechanism for carrying non-OSPF-native data within the OSPF flooding domain. SR information is carried in LSA Type 10 (area-scope Opaque LSAs) with specific Opaque Type values inside: Opaque Type 7 (Extended Prefix LSA) for Prefix-SIDs, Opaque Type 8 (Extended Link LSA) for Adj-SIDs, and Opaque Type 4 (Router Information LSA) for SR capabilities.

Opaque LSA Refresher: Opaque LSAs (RFC 5250) allow OSPF to carry arbitrary data that is flooded but not used by the OSPF SPF computation itself. SR information rides in opaque LSAs alongside normal OSPF LSAs.
OSPFv2 Opaque Type 7
Extended Prefix Opaque LSA. Area-scoped. Carries Prefix-SID sub-TLVs for IP prefixes. One LSA per router announcing SR prefix-SIDs.
OSPFv2 Opaque Type 8
Extended Link Opaque LSA. Area-scoped. Carries Adj-SID sub-TLVs for OSPF adjacencies. One LSA per link/adjacency.
OSPFv2 Opaque Type 4 (Router Information LSA)
Router Information LSA (RFC 7770). Carries SR capabilities (SRGB, algorithms) — equivalent to ISIS TLV 242 Router Capability.
OSPFv3 (IPv6)
OSPFv3 uses Router Information LSA (Type 12) for SR capabilities and Extended Prefix/Link LSAs for SID advertisement. Same concepts, different LSA types.
OSPF SR LSA hierarchy:

Backward Compatibility: Routers that don't support SR simply flood the Opaque LSAs without processing them. The standard OSPF SPF is unaffected. Non-SR routers can coexist in the same OSPF area during migration.

PHP & Penultimate Hop in SR

Controlling label disposition at the egress

What is PHP?

PHP = Penultimate Hop Popping: The second-to-last router in an LSP removes (pops) the MPLS label before forwarding to the egress router. The egress router receives a plain IP packet and performs a normal IP lookup — no MPLS LFIB lookup required.

This optimization has existed since the earliest MPLS deployments. The penultimate hop (the router one hop before the egress) pops the label and sends the unlabeled packet. The egress router is spared a redundant MPLS lookup — it would just have popped the label itself anyway.

Without PHP (egress pops):
P2 → PE2: [16004 | IP]   ← PE2 does LFIB lookup for 16004, then pops, then IP lookup
PE2 → CE2: IP packet

With PHP (penultimate pops):
P2 → PE2: IP packet      ← PE2 does IP lookup only (one lookup saved)
PE2 → CE2: IP packet
Performance Benefit
Saves one LFIB lookup at the egress PE. On high-scale routers processing millions of packets per second, this adds up. Historical hardware had separate MPLS and IP forwarding paths — PHP bypassed the slower MPLS path.
Modern Relevance
On modern ASICs, the lookup cost difference is minimal. PHP is still the default for compatibility and because it's a well-understood behavior. Explicit-null is used when QoS TC bits must be preserved to the egress.
PHP in LDP
In LDP, PHP is signaled by advertising label 3 (implicit-null). When a downstream LSR advertises label 3 for a prefix, the upstream router knows to pop before forwarding.
PHP in SR
In SR, PHP behavior is controlled by the P-flag in the Prefix-SID sub-TLV. P=0 (default) means PHP is on. The egress router communicates its preference via the IGP — no label distribution protocol needed.

PHP in SR-MPLS — Default Behavior

PHP is enabled by default in SR-MPLS. When the P-flag (no-PHP flag) is NOT set in the Prefix-SID advertisement, all penultimate hops perform PHP for that SID.

Default SR Behavior: P-flag = 0 → PHP is ON. The penultimate hop pops the Prefix-SID label and forwards the IP packet to the egress router. This is identical to LDP implicit-null behavior.

TI-LFA Fundamentals

Sub-50ms fast reroute using SR label stacks — no RSVP tunnels, no extra state

What TI-LFA Solves

With MPLS + LDP, getting sub-50ms fast reroute means deploying RSVP-TE with FRR — facility backup or 1:1 protection. That means signaling bypass tunnels in advance, maintaining tunnel state on every transit router, and manually coordinating PLR (Point of Local Repair) and MP (Merge Point) nodes. It is operationally expensive at scale.

Topology-Independent LFA (TI-LFA) solves this within SR-MPLS itself. The IGP (IS-IS or OSPF) pre-computes a backup path using the SR topology. The result is a repair label stack that gets pre-installed in the FIB. No extra tunnels. No extra protocols. No extra state on transit routers.

RSVP-TE FRR — What You Had to Do

  • Signal bypass tunnels with RSVP PATH/RESV
  • Every transit router holds tunnel state (soft state, refreshed every 30s)
  • BFD sessions on every protected link
  • Manual PLR / MP configuration in many cases
  • Scales poorly: N protect links = N bypass tunnels

TI-LFA — What You Get Now

  • Backup computed automatically from IGP topology
  • Repair label stack pre-installed in FIB — zero signaling
  • Zero state on transit routers
  • 100% topology coverage (any topology, any failure)
  • Just enable TI-LFA in your IGP — done
Key insight: TI-LFA works because SR label stacks are source-routed. The protecting router can pre-compute "if link X fails, push these labels to steer around it" — and the entire detour is encoded in the label stack. No transit router needs to know about the repair.

FRR Evolution

Fast reroute has evolved through several generations. Each solved problems of the previous but introduced new complexity — until TI-LFA.

Computed by running SPF from S with the S→F link removed from the topology.

All nodes that can reach destination D without traversing the failed link. Computed from D's perspective (reverse SPF).

Computed by running reverse SPF from D with S→F removed. These nodes have a valid post-failure path to D.

A PQ-node is a node that exists in both P-space and Q-space — S can reach it after the failure, and it can reach D after the failure. It is the ideal repair node.

TI-LFA Repair Lists & Computation

How the protecting router builds and uses repair label stacks during failure

Link Protection vs Node Protection

TI-LFA can protect against two types of failures:

Link Protection

Protects against the link S→F going down. The repair path avoids the failed link but may still traverse node F via a different link. The P/Q-space computation excludes the S→F link.

Simpler to compute. Higher backup path availability. Does not protect if the entire node F fails.

Node Protection

Protects against node F itself failing. Stronger guarantee — the repair path must avoid F entirely. The P/Q-space computation excludes all links touching F.

More restrictive — may require more segments. Subsumes link protection. TI-LFA attempts node protection first, falls back to link if no node-protecting path exists.
Best practice: Always enable node protection where possible. If a line card or whole router fails (not just a single link), link protection fails too. Node protection handles both.

Microloop Avoidance

Preventing transient forwarding loops during IGP convergence

Why Microloops Happen

After a topology change (link failure, link recovery, metric change), each router in the network runs SPF independently and updates its FIB. The problem: routers don't all update at the same time. Router A may converge in 50ms; router B may take 200ms. During this window, A's FIB says "send toward B" while B's FIB still says "send toward A." Packets bounce between them until B converges.

Classic 3-node loop: A─B─C. Link B─C fails. B converges first, routes C-destined traffic via A (A─C still up). But A hasn't converged yet — A's FIB still says "send C-traffic to B" (it was the shortest path before). Loop: A→B→A→B... until A converges.
Microloop formation timeline
T=0 Link B→C fails
T=50ms B converges: FIB[C] = via A (new path)
T=50ms–200ms LOOP: B sends C-traffic to A, A sends C-traffic back to B
T=200ms A converges: FIB[C] = direct via A─C link. Loop breaks.

In LDP/RSVP networks, this loop also causes LDP label changes to propagate in waves — packets may hit routers with inconsistent label bindings, causing drops or misdelivery, not just looping.

SR Microloop Avoidance Mechanisms

SR-MPLS provides two complementary mechanisms. They can be used together.

(a) Local Delay Timer

After detecting a topology change, each router delays updating its FIB by a configurable time (e.g., 5 seconds). This gives all routers time to converge before any FIB update takes effect. Simple and reliable.

Trade-off: During the delay, traffic uses the (now possibly failed) primary path. TI-LFA still handles the fast switchover — the delay just controls when the post-convergence path becomes primary.

(b) Ordered FIB Update (uLoop Prevention)

Each router computes its distance from the failure point, then delays its FIB update proportionally to that distance. Routers farther from the failure update before routers closer to it, ensuring each router installs its new FIB only after all downstream routers (those between it and the failure) have already converged. This distance-based delay prevents the transient inconsistency window that causes microloops.

Based on: RFC 8333 / draft-ietf-rtgwg-uloop-delay. IS-IS and OSPF both support this with SR extensions.
SR advantage: With LDP, label bindings propagate hop-by-hop — each router waits for its neighbor's label before it can update. This creates a dependency chain that can amplify the microloop window. With SR, every router independently knows all labels (from the SRGB + Prefix-SID index). No label propagation dependency. The microloop is purely an IP/metric issue, not a label binding issue.

Basic SR-TE (Explicit Paths)

Traffic engineering without signaling — just push the right label stack

Explicit Paths Without Signaling

In RSVP-TE, traffic engineering works by signaling an explicit route: the headend sends a PATH message with an Explicit Route Object (ERO), each router in the path reserves resources and sends RESV back, and the resulting tunnel is maintained with periodic refreshes. Every router in the path holds soft state. Tear down requires more signaling.

In SR-TE, there is no signaling at all. The ingress node pushes a label stack that encodes the entire path. Transit routers just forward based on the top label — they hold zero extra state. The path IS the label stack.

RSVP-TE Explicit Path

  • PATH message with ERO: PE1→P1→P2→PE2
  • Each hop allocates label and returns RESV
  • Soft state on P1, P2 (refresh every 30s)
  • Failure → re-signaling required
  • Bandwidth reservation optional but tied to state

SR-TE Explicit Path

  • PE1 pushes label stack [Adj-SID(PE1→P1), Adj-SID(P1→P2), 16004]
  • P1 and P2: pop top label, forward. Zero state.
  • No refresh, no teardown, no signaling
  • Failure → push different label stack (or TI-LFA kicks in)
  • No bandwidth reservation (use constraints + PCE for that)
The philosophical shift: In RSVP-TE, the network holds the path (as state in every hop). In SR-TE, the packet holds the path (as labels in the header). Transit routers are stateless. This is the source routing model.

SR-Policy Structure

The framework for intent-based traffic steering in SR-MPLS

What Is an SR-Policy?

An SR-Policy is a named, intent-driven steering object. It is defined by a 3-tuple:

Headend
The router that steers traffic into the policy. This is where the label stack is pushed. In our topology: PE1.
Color
An intent tag (a 32-bit number). It encodes traffic engineering intent: "low-latency," "avoid-region-X," "high-bandwidth." BGP routes tagged with matching color are steered into this policy.
Endpoint
The far-end PE (or any destination node). The policy steers traffic toward this endpoint. In our topology: PE2 (10.0.0.4).

The 3-tuple (PE1, color=100, PE2) uniquely identifies an SR-Policy. Multiple policies can exist for the same headend+endpoint pair, differentiated by color. This allows different traffic types (voice vs. data, premium vs. best-effort) to take different paths.

BGP color extended community: BGP routes carry a Color extended community. The headend router matches the color against installed SR-Policies. Matching traffic is steered into the policy. Non-matching traffic uses the IGP shortest path (or a default policy).

RSVP-TE tunnel equivalent

An RSVP-TE tunnel from PE1 to PE2 with a specific ERO. Traffic steering via autoroute or forwarding adjacency. One tunnel = one path.

SR-Policy equivalent

An SR-Policy is richer: multiple candidate paths (preferred, backup), dynamic computation, weighted ECMP, automatic TI-LFA, Binding SID for simplification. One policy = multiple path options.

Candidate Paths

An SR-Policy contains one or more candidate paths, each with a preference value. The candidate path with the highest preference that is operationally valid (all segments resolvable) is the active path. If it fails, the next-highest preference takes over automatically.

Binding SID (BSID)

A Binding SID is a single MPLS label that represents an entire SR-Policy. Any packet arriving at the headend with the BSID on top of its label stack is steered into the SR-Policy — the BSID is replaced with the active candidate path's segment list.

Why BSID Matters

  • Label stack simplification: Instead of pushing 4–5 specific segment labels, a remote router pushes one BSID label. The headend expands it.
  • Cross-domain stitching: A BSID can stitch SR-Policy segments across domain boundaries. BGP-EPE uses BSIDs to chain paths across AS boundaries.
  • BGP next-hop: A BGP route can use a BSID as next-hop — the color-based steering and SR-Policy selection is abstracted behind the BSID label.

Advanced SR-TE (ODN, PCE, Constraints)

Automating traffic engineering at scale — color communities, PCE delegation, and multi-dimensional constraints

On-Demand Next-Hop (ODN)

In classic SR-TE, the headend engineer pre-provisions each SR-Policy by hand: destination, color, segment list. That works for a handful of policies. At scale — hundreds of VPNs, dozens of PEs — it becomes unmaintainable. On-Demand Next-Hop (ODN) is the answer.

ODN inverts the model. Instead of the operator defining every tunnel, the network automatically creates SR-Policies on demand, triggered by BGP route advertisements. The mechanism:

Classic SR-TE (Static)

  • Operator defines: "create policy to PE2, use path P1→P2"
  • Policy exists regardless of whether traffic needs it
  • N PEs × M colors = N×M manual policy configs
  • Adding a new PE requires updating headend configs
  • No automation — purely operational burden
Does not scale beyond ~10 policies

ODN (Dynamic)

  • BGP route carries color extended community (e.g., color 100)
  • Headend PE sees color 100 + next-hop PE2 → auto-creates policy
  • Policy is created on-demand, destroyed when route withdrawn
  • New PEs self-advertise with colors — zero headend config change
  • Color = intent (100=low-latency, 200=high-bandwidth, 300=diverse)
Scales to thousands of policies automatically

PCE-Initiated vs PCC-Initiated

Once you need path computation, there are two architectural models for who does the computing:

PCC-Initiated (Headend Computes)

The headend PE (PCC — Path Computation Client) runs CSPF locally using its IGP topology database. It computes the path, creates the SR-Policy, and installs the segment list. The PCE (if present) is only notified via PCEP for visibility — it does not control.

When to use: Single-domain networks, simple constraints (IGP metric, latency), when PCE infrastructure is not available or not needed.
  • Local IGP view sufficient for single domain
  • No PCE required — simpler architecture
  • Constraints: metric, affinity (if TE extensions enabled)
  • Cannot do global optimization across domains

PCE-Initiated (PCE Computes and Programs)

An external PCE (Path Computation Element) has global topology visibility via BGP-LS. The PCE computes the path and pushes it to the headend PE via PCEP (PCInitiate). The PE installs the PCE-programmed segment list as an SR-Policy candidate path.

When to use: Multi-domain, multi-area, bandwidth-aware (need global BW view), global optimization (minimize total network utilization), or when centralized control is required.
  • PCE sees entire multi-domain topology via BGP-LS
  • Can optimize across domain boundaries
  • Bandwidth-aware: PCE tracks BW reservations globally
  • Dynamic re-optimization: PCE pushes updates via PCUpd
Criterion PCC-Initiated PCE-Initiated
Topology scope Single domain (local IGP) Multi-domain (via BGP-LS)
Bandwidth awareness No (IGP does not carry BW reservations) Yes (PCE tracks globally)
Global optimization No Yes — PCE minimizes total utilization

PCEP with SR Extensions

PCEP (Path Computation Element Communication Protocol) was originally designed for RSVP-TE. RFC 8231 added stateful PCE support. RFC 8281 added PCE-initiated LSPs. RFC 8664 added SR-specific extensions — replacing RSVP EROs with SR SID lists.

The key SR extension: the PCE sends an SR-ERO (SR Explicit Route Object) containing an ordered list of SIDs. The PCC (headend PE) installs this SID list as an SR-Policy candidate path. No RSVP signaling. No state on transit routers.

PCEP Message Flow: ODN + PCE Path Setup

1
PCOpen — Establish PCEP Session

PE1 (PCC) opens a TCP session to the PCE (port 4189). They exchange Open messages advertising capabilities: stateful, PCE-initiated, SR capability, MSD (Maximum SID Depth = how many labels PE1 can push).

2
PCRpt — PCC Reports Existing State

PE1 sends PCRpt (Report) messages for all existing SR-Policies it has instantiated. This syncs the PCE's view of what PE1 currently has installed. After sync, the PCE has accurate state.

3
ODN Trigger → PE1 Requests Path from PCE

PE1 receives a BGP route with color 200. It needs a new SR-Policy. PE1 sends a PCReq (or for stateful: delegates via PCRpt with "D-bit" set) to the PCE, describing: endpoint=10.0.0.4, color=200, constraints=low-latency.

4
PCInitiate — PCE Programs SR-Policy on PE1

The PCE computes the path using its BGP-LS topology database. It sends a PCInitiate message to PE1 containing the SR-Policy definition: PLSP-ID, BSID, and SR-ERO with SID list [16005, 16004]. PE1 installs this as a candidate path.

5
PCRpt — PE1 Confirms Installation

PE1 responds with a PCRpt confirming the SR-Policy is active (operational=UP, in-use=true). The PCE now knows PE1 has this path installed and active.

6
PCUpd — PCE Re-Optimizes Path

Later, the PCE detects a better path (e.g., link latency changed). It sends a PCUpd with the updated SR-ERO: new SID list [16002, 16003, 16004]. PE1 updates its FIB atomically. Traffic shifts to the new path with no disruption.

MSD matters: The PCE must know PE1's Maximum SID Depth (how many labels PE1's dataplane can push). MSD is advertised in the PCOpen message and also in ISIS (RFC 8491). PCE will never send a SID list longer than MSD.

Flex-Algo

Virtual topologies within one physical network — algorithm IDs 128-255 for custom forwarding planes

What is Flex-Algo?

Standard SR-MPLS uses Algorithm 0 — shortest path by IGP metric. Every Prefix-SID you've seen so far (16001–16005) is computed using SPF with the standard IS-IS metric. Flex-Algo extends this with algorithm IDs 128–255, each defining a completely independent forwarding topology.

Think of Flex-Algo as running multiple parallel SPFs on the same physical network simultaneously — each with different rules:

Without Flex-Algo

One topology. One SPF. One forwarding plane. To use a different path for different traffic, you need SR-TE with explicit SID lists or RSVP-TE tunnels. Every hop must be engineered explicitly.

With Flex-Algo

Multiple topologies. One physical network. Algo 0 = IGP cost. Algo 128 = minimize delay. Algo 129 = diverse (red links excluded). Each algo has its own Prefix-SIDs, its own FIB, its own forwarding path. Just push the right label.

Flex-Algo is the "distributed TE" complement to SR-TE's "centralized TE." Instead of the headend computing a path node-by-node, every node independently computes the same SPF for each algorithm, using the Flex-Algo Definition (FAD) advertised in the IGP. The result is a distributed, consistent forwarding table for each topology.

Analogy for MPLS engineers: If RSVP-TE is like manually routing every truck along a specific highway route, Flex-Algo is like creating separate road networks — one for trucks (algo 0), one for ambulances (algo 128, fastest), one for hazmat (algo 129, avoids tunnels). Each vehicle type follows its own network's rules automatically.

Algorithm ID
128–255. Algo 0–127 reserved (0=SPF/IGP, 1=strict SPF). Network can run multiple Flex-Algos simultaneously.
Metric Type
0 = IGP metric, 1 = min unidirectional delay, 2 = TE metric. Each algo can use a different metric.

Include/exclude links by affinity bits. Only links matching constraints participate in that algo's topology.

Before any router can participate in Flex-Algo 128, a Flex-Algo Definition (FAD) must be advertised in the IGP. One or more defining routers originate the FAD as an IS-IS TLV (or OSPF TLV). All participating routers receive it and use it to compute their topology for that algorithm.

BGP-LS

Exporting the IGP topology to external consumers — the eyes of the PCE and SDN controller

What is BGP-LS?

BGP Link-State (BGP-LS) is a BGP address family defined in RFC 7752 with AFI 16388, SAFI 71. It exports the IGP topology database — nodes, links, prefixes, and all their attributes — as BGP NLRI (Network Layer Reachability Information). The IGP's link-state database (LSDB) becomes available over BGP to any consumer that can peer with a BGP-LS exporter.

Without BGP-LS, the IGP topology is trapped inside the IGP domain. Only routers running IS-IS or OSPF in that area can see the full topology. External systems like PCE controllers, SDN orchestrators, or network visualization tools have no direct window into the network's internal structure.

Without BGP-LS

  • PCE must join ISIS/OSPF in every area
  • Multi-domain: PCE needs IGP sessions in every domain
  • Tight coupling: PCE participates in routing convergence
  • SDN controller has no topology view
  • Visualization tools need SNMP polling or proprietary APIs

With BGP-LS

  • 1–2 routers per domain export LSDB via BGP to PCE
  • Multi-domain: BGP-LS route reflector aggregates all domains
  • PCE is completely out-of-band — does not participate in IGP
  • SDN controller gets real-time topology updates
  • Same BGP session carries SR SID data (Prefix-SIDs, Adj-SIDs, SRGB)
BGP-LS is what makes PCE-based SR-TE scalable. The PCE gets a live, consistent, multi-domain view of the entire network topology — including SR SIDs, link metrics, delay values, and Flex-Algo data — over a single BGP session. It can then compute optimal paths and program them via PCEP.

BGP-LS + SR + PCE Integration

This is the complete architecture showing how BGP-LS sits at the center of SR-TE automation:

1
ISIS Floods Topology + SR Attributes

IS-IS runs in the domain. Every router advertises its LSP (Link State PDU) containing: adjacency info, Prefix-SID TLVs (index + algorithm), Adj-SID TLVs, SR Capability TLV (SRGB), TE metric sub-TLVs, delay sub-TLVs, and Flex-Algo FAD/participation TLVs. Every router has the complete LSDB.

2
BGP-LS Exporter Converts LSDB to BGP-LS NLRIs

One or two routers (e.g., P1) are configured as BGP-LS exporters. They read the ISIS LSDB and generate BGP-LS Node, Link, and Prefix NLRIs. These are advertised to the PCE via a BGP session (AFI 16388, SAFI 71). Updates are sent as the IGP topology changes — real-time.

3
PCE Builds Complete Topology Graph with SR SIDs

The PCE receives all BGP-LS NLRIs and builds an in-memory topology graph. Each node has SRGB range. Each link has metric, TE metric, delay. Each prefix has Prefix-SID (index + algo). The PCE can now compute labels: PE2's algo-0 SID = 16000+4=16004, PE2's algo-128 SID = 16000+104=16104.

4
PCE Computes SR-TE Paths Using SID Lists

For each SR-Policy request (from ODN or operator config), the PCE runs CSPF on its graph with the specified constraints. It produces a SID list: e.g., [16005, 16004] for PE1→P3→PE2 low-latency path. This is expressed as an SR-ERO in PCEP.

5
PCE Programs SR-Policies on Headend PEs via PCEP

The PCE sends PCInitiate (or PCUpd) to PE1 via PCEP. PE1 installs the SR-Policy with the SID list. BGP VPN routes resolve via this policy. Traffic is steered. When topology changes (link failure, delay change), PCE recomputes and sends PCUpd — dynamic re-optimization without any operator involvement.

The information flow is unidirectional at each layer: ISIS → BGP-LS → PCE (topology in), PCEP → PE headend (policy out). The PCE never injects into the IGP. It observes via BGP-LS, computes, and acts via PCEP. Clean separation of concerns.

BGP-LS Configuration Overview

Enabling BGP-LS is typically a few configuration steps. The specifics are vendor-dependent, but the logical sequence is universal:

1
Enable BGP-LS Address Family on the Exporter Router

On the router that will export (e.g., P1), enable the BGP-LS address family under the BGP process. Configure the neighbor (PCE's IP address) with address-family link-state link-state. This tells BGP to advertise topology NLRIs to this neighbor.

2
Configure ISIS to Export into BGP-LS

Under the ISIS process, enable redistribution into BGP-LS (or configure a BGP-LS instance ID to associate the ISIS topology). This causes the ISIS LSDB to be reflected as BGP-LS NLRIs. Specify which ISIS instance/level to export (e.g., level-2, area 49.0001).

3
Establish BGP-LS Session to PCE

The BGP session to the PCE uses standard eBGP or iBGP mechanics — just with AFI/SAFI 16388/71 negotiated. The PCE (running software like Cisco SR-PCE, Juniper PCE, or an open-source ONOS/ODL controller) accepts BGP-LS sessions and populates its topology database.

4
Verify BGP-LS Operation

Key verification commands (vendor-neutral concepts):

  • show bgp link-state summary — BGP-LS neighbor state, prefix counts
  • show bgp link-state nlri node — all Node NLRIs being exported
  • show bgp link-state nlri link — all Link NLRIs with TE attributes
  • show bgp link-state nlri prefix — all Prefix NLRIs with SR SID TLVs
  • On PCE: verify node/link/prefix count matches expected topology size
BGP-LS exporter redundancy: Configure 2 routers as BGP-LS exporters (e.g., P1 and P2) for resilience. The PCE will receive duplicate NLRIs but will de-duplicate using the topology identifiers in each NLRI. If one exporter fails or its BGP session drops, the PCE continues receiving updates from the other.

LDP-SR Coexistence & Migration

Migrating from LDP to SR without a forklift upgrade — incremental, reversible, production-safe

Why Coexistence?

In the real world, you cannot flip an entire network from LDP to SR overnight. Hardware upgrade cycles, software qualifications, maintenance windows, and risk management all demand a gradual migration. SR coexistence mechanisms let you run LDP and SR simultaneously on the same network while migrating node by node.

Migration Drivers

  • Some routers may not support SR (older HW/SW)
  • SR requires IGP extensions — ISIS/OSPF must be updated
  • Risk management: migrate incrementally, roll back if needed
  • Operators need time to validate SR behavior before full cutover
  • LDP labels are per-hop; migrating requires coordination

Two Coexistence Models

  • Ship-in-the-Night: LDP and SR active, traffic uses one or the other — no interaction
  • SR-Prefer: Both active, router prefers SR label when both exist for same prefix
  • Ship-in-the-Night: safer, easier, but no migration progress
  • SR-Prefer: the actual migration mechanism — per-router, incremental
  • Mapping Server: for LDP-only nodes that cannot run SR at all

Ship-in-the-Night

Mapping Server

What about routers that cannot run SR at all? Hardware too old, software not upgraded, or a leased device you don't control. The Mapping Server (MS) solves this. It is an SR-capable router (typically a route-reflector or dedicated node) that advertises Prefix-SID mappings on behalf of SR-incapable nodes.

Without Mapping Server

  • LDP-only node (e.g., old P router) has no Prefix-SID
  • SR-capable routers cannot build SR label for that node's prefix
  • SR path computation cannot route through the LDP-only node
  • Island: SR-capable PEs cannot communicate via SR transport
SR path blocked by LDP-only node

With Mapping Server

  • MS advertises: "prefix 10.0.0.99/32 (LDP-node) has SID index 99"
  • SR-capable routers install label 16099 for that prefix
  • When SR packet arrives at LDP-only node, gateway router does label swap to LDP label
  • LDP-only node has no idea SR exists — it processes LDP label normally
Seamless SR transport through LDP islands

Mapping Server Operation

1
Mapping Server Advertises SID Bindings

The MS uses ISIS TLV 149 (SID/Label Binding TLV) to advertise prefix-to-SID mappings for LDP-only nodes. Example: "10.0.0.99/32 → SID index 99." This TLV is flooded throughout the SR domain just like a Prefix-SID Sub-TLV would be.

2
SR-Capable Routers Build Labels for LDP Prefixes

Every SR-capable router receives the binding and installs label 16099 (SRGB_base 16000 + index 99) in its LFIB for 10.0.0.99/32. Forwarding is identical to a native Prefix-SID — the router doesn't know the destination is LDP-only.

3
Gateway Router Performs SR-to-LDP Stitching

The SR-capable router directly adjacent to the LDP-only node acts as a stitching point. When it receives a packet with SR label 16099, it pops the SR label and pushes the LDP label it learned via LDP for 10.0.0.99/32. The LDP-only node receives a normal LDP-labeled packet.

The LDP-only node sees zero change. LDP sessions remain up. It receives LDP-labeled packets exactly as before. The mapping server and stitching are completely invisible to it. This is the key advantage — you can extend SR reachability to nodes you cannot touch.

Migration Steps: LDP → SR

A production-safe, step-by-step migration sequence using the reference topology (CE1—PE1—P1—P2—PE2—CE2). Every step has a defined rollback procedure.

1
Enable SR on All Routers (Alongside LDP)

Configure SRGB (16000-23999) and Prefix-SIDs on all four routers (PE1=1, P1=2, P2=3, PE2=4). Enable ISIS SR extensions. LDP remains fully active. This is ship-in-the-night — SR labels are installed in all LFIBs but no traffic uses them yet. Risk: Zero — LDP forwarding is unchanged. Rollback: Remove SR config on any router independently.

2
Verify SR Labels in All LFIBs

On every router, confirm: labels 16001-16004 are installed in the LFIB for the four loopbacks. Verify ISIS is advertising all Prefix-SIDs. Check for SRGB conflicts or index collisions. Use show mpls forwarding-table and show isis segment-routing prefix-sid-map. Risk: None — purely verification. Do NOT proceed until all labels are present everywhere.

3
Enable SR-Prefer on Edge Routers (PE1, PE2)

Configure segment-routing prefer (or equivalent) on PE1 and PE2. These routers now originate SR-labeled traffic. Core routers P1 and P2 still forward LDP labels for non-SR traffic from other networks, but SR-labeled packets from PEs transit correctly via global SIDs. Risk: Low — only PE-originated traffic changes. Rollback: Disable SR-prefer on PEs; LDP immediately reactivates.

4
Enable SR-Prefer on Core Routers (P1, P2)

Enable SR-prefer on P1 and P2. Now all routers prefer SR for all labeled forwarding. The entire network is running SR transport. LDP sessions remain up and LDP labels remain installed — SR is simply preferred in all LFIB lookups. Risk: Low-Medium — full network transition. Monitor traffic counters carefully. Rollback: Disable SR-prefer on P1/P2 (and PEs if needed) to instantly revert.

5
Verify All Traffic Uses SR Transport

Use traceroute with SR label verification to confirm traffic is following SR paths. Confirm LDP label counters are at zero (no traffic using LDP labels). Run this validation period for a minimum of 24-48 hours covering multiple traffic profiles (peak hours, failover events, BGP route changes). Risk: None — observation only. Rollback: Full rollback still available via SR-prefer removal.

6
Remove LDP Configuration

Disable LDP on all routers. LDP sessions will drop. LDP labels will be withdrawn and removed from LFIBs. Since all traffic is already using SR labels, this causes zero traffic impact. Risk: Medium — this is the only partially irreversible step (LDP reconverges in ~seconds if re-enabled, but there is a brief LDP label relearning period). Rollback: Re-enable LDP; it reconverges independently of SR.

7
Reclaim LDP Label Space

Adjust the dynamic label range to reclaim the space previously used by LDP (typically 100000-299999 or vendor-specific). This label space is now available for other uses (e.g., RSVP-TE if still running, VPN labels). Update any monitoring/alerting that referenced LDP-specific label ranges. Risk: Zero — administrative cleanup only. Migration complete.

Total migration time in production: Steps 1-2 can be done in one maintenance window. Steps 3-4 in separate windows. Steps 5-7 after a validation period. The entire migration can take weeks to months — that is normal and correct. Never rush past the verification step.
SR-MPLS OAM

Verifying, troubleshooting, and monitoring SR-MPLS paths — ping, traceroute, BFD, and common failure scenarios

MPLS Ping with SR Labels

Standard MPLS ping (RFC 8029, formerly RFC 4379) works directly with SR labels. The tool is the same one you know from LDP — the target FEC is simply the SR Prefix-SID instead of an LDP FEC element. The echo request packet carries the SR label stack; transit routers swap normally; the egress router processes the UDP echo request and sends a reply.

LDP Ping (Reference)

ping mpls ipv4 10.0.0.4/32
! FEC: IPv4 prefix 10.0.0.4/32
! Label stack: [LDP-label-per-hop]
! Each router validates LDP FEC

SR Ping

ping mpls segment-routing prefix 10.0.0.4/32
! FEC: SR Prefix-SID for 10.0.0.4/32
! Label stack: [16004]
! Each router validates SR FEC mapping

SR Ping/Traceroute Flow: PE1 → PE2

1
PE1 Sends Echo Request

PE1 builds an MPLS echo request: label stack [16004], UDP destination port 3503, TTL=255. The payload contains a Target FEC Stack TLV identifying the SR Prefix-SID for 10.0.0.4/32. PE1 sends it toward P1.

2
P1 and P2 Transit Normally

P1 receives the packet with label 16004, swaps 16004→16004 (uniform SRGB, same label), decrements TTL. P2 does the same. Neither P1 nor P2 terminates the ping — they are transit routers for this Prefix-SID. (For traceroute, TTL=1 at each hop causes the router to generate a time-exceeded ICMP, which is returned as an echo reply with return code 8.)

3
PE2 Terminates and Replies

PE2 is the egress for label 16004 (its own Prefix-SID). It pops the label, processes the UDP echo request, validates the FEC, and sends an MPLS echo reply back to PE1's source address. Reply carries return code 3 (Egress for the FEC).

Example Traceroute Output (Uniform SRGB)

traceroute mpls segment-routing prefix 10.0.0.4/32

Tracing MPLS Label Switched Path to 10.0.0.4/32, timeout is 2 seconds:

  1  10.1.1.2 (P1)   1 ms   [Label: 16004, Exp: 0]
  2  10.2.2.3 (P2)   2 ms   [Label: 16004, Exp: 0]
  3  10.3.3.4 (PE2)  3 ms   [Label: implicit-null]

Path 1 found, output interface GigabitEthernet0/0/0
Notice: The label 16004 appears at every hop — P1, P2, and then PHP at PE2. With uniform SRGB, you immediately know "16004 = PE2" without consulting a per-hop label mapping table. This is a significant debugging advantage over LDP where every hop shows a different label.

Nil-FEC Validation

Classic MPLS ping validates an LSP by including a Target FEC Stack TLV — the transit router checks that the label it received matches its FEC. This works for Prefix-SIDs because each label maps to a known FEC. But for SR-TE explicit paths (segment lists), the intermediate routers only see individual labels — they don't know the full intended path or the ultimate FEC being validated.

The problem: If you push an explicit stack [16002, 16003, 16004] to force traffic through P1→P2→PE2, a transit router receiving label 16002 has no way to validate whether this is part of the intended SR-TE policy or an accidental mismatch. Standard FEC validation cannot work here.

How Nil-FEC Solves This

RFC 8029 defines the Nil FEC — a special FEC TLV that means "I am not asking you to validate the FEC for this label. Just validate that you can forward it and tell me what you did." The echo request uses a Nil-FEC TLV with the label being tested. Each router responds with its next-hop and outgoing label, allowing full path verification without FEC knowledge.

Standard FEC Ping (Prefix-SID)

  • Works when label maps to a known FEC (Prefix-SID)
  • Each hop validates "this label = this prefix"
  • Fails for explicit SR-TE segment lists
  • Cannot debug path stitching errors

Nil-FEC Ping (SR-TE)

  • Works for any label, any context
  • Each hop validates "I can forward this label" and replies
  • Essential for debugging SR-TE explicit segment lists
  • Used for Adj-SIDs and Binding SIDs too
! SR-TE Nil-FEC traceroute — validate explicit path [PE1→P1→P2→PE2]
traceroute mpls nil-fec labels 16002,16003,16004 output-interface GigabitEthernet0/0

  1  P1 (10.1.1.2)   1 ms  [Nil-FEC, Label 16002 POP (own Node-SID), exposed 16003]
  2  P2 (10.2.2.3)   2 ms  [Nil-FEC, Label 16003 POP (own Node-SID), exposed 16004]
  3  PE2 (10.3.3.4)  3 ms  [Nil-FEC, Label 16004 egress — destination reached]
Note on PHP: In normal forwarding, PE1 would PHP-pop 16002 before sending to P1 (since PE1 is directly connected). Nil-FEC traceroute bypasses this by using TTL manipulation — it pushes the full label stack with TTL=1 on the outermost label, ensuring each hop responds with an ICMP TTL-exceeded. This lets you validate every label in the explicit path regardless of PHP behavior.

Traceroute Behavior: Uniform vs Non-Uniform SRGB

One of the most practically useful properties of SR for operations teams is how traceroute output changes depending on SRGB configuration across the network.

Uniform SRGB (Recommended)

All routers have the same SRGB base (16000). Index 4 = PE2. Label 16004 everywhere. Traceroute shows 16004 at every hop. You immediately know "16004 = PE2" from any vantage point. No per-hop label table lookups needed.

Hop 1: [Label: 16004] ← PE2
Hop 2: [Label: 16004] ← still PE2
Hop 3: [PHP/implicit-null]
Human-readable, globally meaningful

Non-Uniform SRGB

Routers have different SRGB bases (e.g., P1 base=17000, P2 base=18000). The same SID index maps to different labels on different routers. Traceroute shows changing labels per hop — similar to LDP. Requires per-router label tables to interpret.

Hop 1 (P1 SRGB=17000): [Label: 17004]
Hop 2 (P2 SRGB=18000): [Label: 18004]
Hop 3: [PHP/implicit-null]
Must consult per-router SRGB to decode
Operational recommendation: Configure identical SRGB ranges across your entire network. The uniform SRGB property — where labels have globally consistent meaning — is one of SR's biggest operational advantages over LDP. Non-uniform SRGB negates this advantage. The only reason to use non-uniform SRGB is if you are absorbing routers from different vendors with different default SRGB ranges into a single domain.

BFD over SR-MPLS LSPs

BFD (Bidirectional Forwarding Detection) provides sub-50ms failure detection for SR-MPLS paths. Two distinct BFD modes exist for SR, serving different purposes:

BFD over Prefix-SID Path

BFD packets are sent with the SR transport label (e.g., 16004) toward the remote node. Detects any failure on the shortest-path to that node. When BFD detects a failure, it triggers TI-LFA switchover to the backup path — the same backup that would be used for IGP failures.

  • BFD session: PE1 ↔ PE2 over Prefix-SID 16004
  • Detects: link failure, node failure, any mid-path failure
  • Triggers: TI-LFA reroute to backup path
  • Integration: works natively with TI-LFA timer

BFD over SR-Policy

BFD packets are sent with the full SR-Policy segment list as the label stack. Detects failure specifically on that explicit SR-TE path. If the policy path fails, BFD triggers failover to a lower-priority candidate path within the same SR-Policy.

  • BFD session: PE1 ↔ PE2 over explicit path [16002, 16003, 16004]
  • Detects: failure on this specific explicit path only
  • Triggers: candidate path failover within SR-Policy
  • Integration: works with SR-Policy preference/priority mechanism

Sub-50ms detection: With BFD timers at 3×10ms = 30ms detection, TI-LFA pre-computed backup paths activate in under 50ms total. The sequence is: BFD detects failure → signals local FRR module → TI-LFA backup activated → traffic rerouted. This happens before IGP reconverges.

Troubleshooting Common Issues

Systematic diagnosis of the most frequently encountered SR-MPLS operational problems. Organized by severity and frequency.

SRGB Mismatch Between Routers
Symptom: Ping to remote Prefix-SID succeeds locally but traceroute shows wrong labels or drops mid-path.
Cause: Routers have different SRGB ranges configured. Router A computes label = 16004, Router B computes label = 17004 for the same SID index.
Diagnosis: show segment-routing global-block on all routers — verify identical base and range. Check ISIS TLV 242 SR-Capabilities Sub-TLV for SRGB advertisements from each router.
Fix: Align SRGB configuration across all routers. Requires coordinated maintenance window to change SRGB (labels will temporarily change, causing brief disruption).
Label Collision (SRGB Overlaps Dynamic/Static Range)
Symptom: Some SR labels are missing from LFIB, or traffic loops/drops intermittently for specific prefixes.
Cause: SRGB range overlaps with the dynamic label range used by LDP, RSVP-TE, or BGP. Labels in the overlap zone are allocated to both SR and another protocol.
Diagnosis: show mpls label range — verify static, dynamic, and SRGB ranges are non-overlapping. Check for any static label assignments in SRGB range.
Fix: Reconfigure ranges to ensure clean separation. Recommended: static 0-15, SRLB 15000-15999, SRGB 16000-23999, dynamic 24000+.
Adj-SID Not Programmed
Symptom: SR-TE policy using Adj-SIDs is not active; show segment-routing traffic-eng policy shows "down" or "no valid segment list."
Cause: The IGP adjacency the Adj-SID refers to is not up, or the SRLB range is exhausted so no label was allocated for the adjacency.
Diagnosis: show isis adjacency — verify adjacency state. show segment-routing local-block — verify SRLB has free labels. show isis segment-routing adjacency-sid — check Adj-SID allocation.
Fix: Restore the IGP adjacency, or expand the SRLB range if exhausted.
Prefix-SID Not in LFIB
Symptom: show mpls forwarding-table does not show expected label (e.g., 16004 for PE2). Pings to remote Prefix-SID fail.
Cause: ISIS is not advertising the Prefix-SID Sub-TLV for that prefix, SID index conflict (two routers using same index), or the route itself is not in the IGP topology.
Diagnosis: show isis database verbose — inspect the ISIS LSP for the originating router and verify TLV 135/235 contains the Prefix-SID Sub-TLV. Check for duplicate SID index alerts in logs. show isis segment-routing prefix-sid-map.
Fix: Correct the SID index configuration to resolve conflicts; ensure ISIS is advertising the prefix with the N-flag set.
SR-TE Policy Not Active
Symptom: SR-Policy configured but not installed in FIB; show segment-routing traffic-eng policy shows no active candidate path.
Cause: No candidate path with sufficient preference is valid. Common reasons: Binding SID collision, segment list contains unreachable SID, PCE is unreachable (for PCE-initiated policies), or explicit path's intermediate nodes are unreachable.
Diagnosis: Check candidate path state and down reason. Verify each SID in the segment list is resolvable in the local RIB. Verify PCE connectivity via PCEP session state.
Fix: Resolve unreachable SIDs, fix BSID collision, or restore PCE connectivity.
TI-LFA Backup Not Installed
Symptom: show isis fast-reroute shows some prefixes without backup path. Failures on those links/nodes cause traffic loss for full IGP convergence time (~1-2 seconds).
Cause: TI-LFA cannot find an alternate path satisfying the loop-free condition for that specific prefix. This happens in certain topologies (e.g., linear topologies, some ECMP cases) or when the backup would require more than the configured maximum repair segments.
Diagnosis: show isis fast-reroute detail — shows why backup was not computed. Topology inspection: are there alternate paths? Are P-space and Q-space disjoint (no PQ node reachable within configured repair segment limit)?
Fix: Add redundant links to provide topological diversity, increase max repair segments (up to 3 handles virtually all topologies), or use Remote LFA as an alternative where TI-LFA coverage is incomplete.

SR-TE & SR-Policy Terms

Term Definition
Headend The router that originates (imposes) the SR label stack. For L3VPN, this is the ingress PE. For SR-TE, this is the router where the SR-Policy is configured and where label imposition happens.
Color A 32-bit extended community attached to BGP routes. Signals the intended transport path intent (e.g., 100 = low-latency, 200 = high-bandwidth). Headend PEs match color to select the appropriate SR-Policy for traffic steering.
Endpoint The tail-end router of an SR-Policy — typically the remote PE's loopback IP address. Combined with Color forms the unique SR-Policy identifier: (headend, color, endpoint).
Candidate Path A specific path within an SR-Policy, defined by a segment list (explicit) or computed by CSPF/PCE (dynamic). Multiple candidate paths can exist per policy; the one with the highest preference that is operationally up is active.
Segment List The ordered list of SIDs that defines the explicit path for an SR-Policy candidate path. Example: [16002, 16003, 16004] = force traffic through P1 → P2 → PE2 regardless of IGP shortest path.
Weight Relative weight assigned to a segment list within an SR-Policy candidate path. A candidate path can contain multiple segment lists with different weights for weighted ECMP. Traffic is distributed proportionally (e.g., weight 3 and weight 1 = 75%/25% split).
BSID Binding SID — a locally allocated label that is an alias for an SR-Policy. External traffic pushed to the BSID label is steered into the policy's active segment list. Simplifies inter-domain SR-TE (border router just knows the BSID, not the full internal segment list).
ODN On-Demand Next-Hop — automation mechanism where SR-Policies are dynamically created and destroyed in response to BGP route advertisements carrying Color communities. Eliminates manual SR-Policy provisioning at scale.
PCE Path Computation Element — a centralized controller (server) with global network topology visibility via BGP-LS. Computes SR-Policy paths satisfying complex constraints (bandwidth, latency, diversity, affinity) that individual routers cannot compute locally.
PCEP Path Computation Element Protocol (RFC 5440 + RFC 8231/8281 extensions) — the protocol between a PCC (router) and PCE. Used to request path computation, receive computed paths, and for PCE to push (PCInitiate) SR-Policies to routers.
CSPF Constrained Shortest Path First — Dijkstra SPF with additional constraints applied (metric type, affinity/color, max hop count, bandwidth). Runs either locally on the headend router or on the PCE. Produces an explicit segment list satisfying all constraints.
Flex-Algo Flexible Algorithm — allows the IGP to compute multiple independent SPF trees simultaneously, each using different metric types or link constraints, without requiring SR-TE. Each algorithm produces its own Prefix-SID label space.
FAD Flex-Algo Definition — the configuration block (advertised in ISIS TLV 242) that defines a Flex-Algo: algorithm ID, metric type, calculation type, and link constraints (include-any, include-all, exclude-any affinity filters). Originated by one or more routers in the domain.

Key Formulas & Rules

Label Computation
Label = SRGB_base + SID_index. With SRGB 16000-23999: PE2 (index 4) → label 16004. This calculation is performed independently by every router in the domain using the SRGB advertised by the originating router.
Uniform SRGB Property
When all routers share the same SRGB base, label values are globally consistent. Label 16004 means "PE2" on every router in the domain. Traceroute shows the same label at every hop. This is a design goal, not a requirement.
PHP Rule
The penultimate hop (last router before egress) pops the outermost transport label and forwards the remaining stack natively to the egress. Enabled unless the egress sets the P-flag (No-PHP) or E-flag (Explicit-Null) in its Prefix-SID advertisement.
BSID Expansion
When a router receives a packet with label = BSID value, it replaces the BSID label with the full segment list of the associated SR-Policy. The packet continues with the expanded label stack. BSID enables single-label steering into arbitrarily complex paths.
ODN Trigger Rule
ODN creates an SR-Policy automatically when a BGP route is received with a Color Extended Community and the headend has no existing SR-Policy matching (headend, color, endpoint). The policy is torn down when no BGP routes reference it.
TI-LFA Coverage
TI-LFA provides 100% topological coverage (given sufficient path diversity) using at most 3 repair segments in the backup label stack. Real-world networks with standard topology achieve near-100% coverage with ≤2 repair segments in most cases.
MSD Constraint
Maximum SID Depth (MSD) is the hardware limit on label stack depth. Advertised per-node and per-link in ISIS. SR-TE path computation must ensure the total segment list depth ≤ MSD of every transit router. Typical values: 10-16 labels.
Adj-SID Uniqueness
Adj-SIDs are locally unique per router, not globally unique. Two different routers may allocate the same Adj-SID label value for different adjacencies. When building SR-TE segment lists, always pair Adj-SIDs with the preceding Prefix-SID of the allocating router.

CLI Patterns (Vendor-Neutral Pseudo-CLI)

Common verification commands across Cisco IOS-XR / IOS-XE, Junos, and Nokia SR OS — patterns are representative; exact syntax varies by platform and version.

SR Global State

show segment-routing global-block
! Shows SRGB: base label, range, state (allocated/in-use)

show segment-routing local-block
! Shows SRLB: base label, range, used/free count

show segment-routing mapping-server
! Shows MS-advertised SID bindings (LDP coexistence)

show segment-routing srv6 locator
! SRv6 locator info (if running SR-MPLS+SRv6 dual)

ISIS SR State

show isis segment-routing prefix-sid-map
! Local and received Prefix-SIDs: prefix, index, label, flags

show isis segment-routing adjacency-sid
! All Adj-SIDs: local label, neighbor, interface, flags

show isis database verbose
! Full LSP with TLV 135/235/242 sub-TLV detail

show isis fast-reroute detail
! TI-LFA backup paths per prefix, repair segments, coverage

MPLS Forwarding

show mpls forwarding-table
! All LFIB entries: in-label, out-label, next-hop, bytes

show mpls forwarding-table labels 16000 16999
! Filter to SRGB range only — SR-specific entries

show mpls forwarding-table labels 15000 15999
! Filter to SRLB range — Adj-SID entries

show mpls label range
! Configured label ranges: static, dynamic, SRGB, SRLB

SR-TE Policies

show segment-routing traffic-eng policy
! All SR-Policies: color, endpoint, BSID, active candidate path

show segment-routing traffic-eng policy name <name> detail
! Full policy detail: all candidate paths, segment lists, state

show segment-routing traffic-eng forwarding policy
! Forwarding state: BSID, label stack, next-hop, interface

show segment-routing traffic-eng pcc
! PCEP session state: PCE address, delegation, report counts

OAM Commands

ping mpls segment-routing prefix 10.0.0.4/32
! MPLS ping to Prefix-SID — verifies LSP end-to-end

traceroute mpls segment-routing prefix 10.0.0.4/32

traceroute mpls nil-fec labels <stack>
! Nil-FEC traceroute — validates explicit SR-TE segment list

show bfd session
! BFD sessions: state, interval, detect-mult, interface
