SRv6 Network Programming Model

Instructions encoded in IPv6 addresses — the programmable data plane

Prerequisite: This guide assumes familiarity with basic SRv6 concepts from the SRv6 Complete Guide — including SID structure, SRH mechanics, and uSID compression. Here we go deeper into the network programming framework defined by RFC 8986.

What Is SRv6 Network Programming?

SRv6 Network Programming is a paradigm where the data plane itself becomes programmable. Each SRv6 SID encodes not just a destination, but an instruction — a specific behavior the node must execute when it processes the packet. The ordered list of SIDs in the SRH forms a program that the network executes hop by hop.

Traditional Forwarding

Routers make independent decisions at each hop based on destination lookup. The source has no control over the path or per-hop actions.

SRv6 Network Programming

The source encodes a sequence of instructions (SIDs) in the packet. Each node reads its instruction, executes the behavior, and advances to the next. The network executes the source's program.

Think of it this way: An SR-MPLS label stack says "go here, then here, then here." An SRv6 SID list says "go here and do this, then go there and do that." Each SID carries both routing (via the locator) and an action (via the function).

Transit Behaviors

How packets traverse SRv6 nodes that are NOT the current SID destination

What "Transit" Means in SRv6

A transit node is any router that forwards an SRv6 packet without being the destination indicated in the IPv6 Destination Address. The node simply performs standard IPv6 longest-prefix-match forwarding on the DA — it does not inspect the SRH, does not decrement Segments Left, and does not modify the DA.

Key insight: Transit nodes do NOT need to understand SRv6 at all. They only need to be IPv6-capable. This is a fundamental advantage over SR-MPLS, where every transit node must understand MPLS label operations.

When Does Transit Apply?

A node operates in transit mode when both of these conditions are true:

  • The IPv6 DA does not match any locally-programmed SID (My Local SID Table)
  • The node has a route toward the DA in its IPv6 FIB

The node simply forwards the packet toward the DA — SRH and all — as if it were any ordinary IPv6 packet.

Transit is the default behavior. Only when DA matches a local SID does the node perform SRv6-specific processing.

Transit Processing Flowchart

When a packet arrives at any SRv6-capable node, the decision logic is straightforward:

1
Packet arrives — inspect IPv6 DA

The node checks the Destination Address against its My Local SID Table (the set of all SIDs this node owns).

2
DA matches a local SID?

Yes: Perform the SID's programmed behavior (uN, uA, uDT4, etc.). This is endpoint processing, not transit.

No: Proceed to transit processing (step 3).

3
Transit: forward via IPv6 FIB

Perform a standard IPv6 longest-prefix-match lookup on the DA. Forward the packet — including the SRH — toward the next hop. Do not modify the SRH or DA.

4
No route for DA?

If the IPv6 FIB has no matching route for the DA, the packet is dropped and an ICMPv6 Destination Unreachable is sent back. This is standard IPv6 behavior — nothing SRv6-specific.

Performance note: Transit processing is as fast as regular IPv6 forwarding. There is zero overhead for SRH presence — the extension header is carried transparently. In hardware-based routers, transit SRv6 packets are forwarded at line rate in the same pipeline as any IPv6 packet.
uN — Micro-SID Node Behavior

The shift-and-forward operation that makes uSID work at every waypoint

What uN Does

uN is the micro-SID equivalent of the End (Endpoint) behavior from RFC 8986. When a packet's DA matches a node's locator prefix, the uN behavior consumes the current 16-bit micro-SID and advances to the next one — all by manipulating the DA in place.

Core principle: uN replaces both End and End.X (when ECMP is acceptable) behaviors in compressed uSID deployments. A single uN operation is equivalent to an MPLS label swap — it advances the forwarding state by one hop.

The incoming packet has a DA where the first 48 bits (block + function) match this node's uN SID. For example, packet arrives at P1 with DA = fcbb:bb01:0200:0400:0300:0048::. The prefix fcbb:bb01:0200::/48 matches P1's locator.

2
Read the NEXT 16-bit uSID

P1 looks at bits 48-63 of the DA (the next 16-bit slot after its own function). In this case, that is 0400 — which is PE2's node identifier.

3
Shift DA left by 16 bits

Remove the consumed uSID (0200) by left-shifting. The DA becomes fcbb:bb01:0400:0300:0048::. The block prefix is preserved; everything after it shifts left; a zero fills the vacated rightmost slot.

4
Check: is the new active uSID zero?

If the next uSID (now at bits 32-47) is 0000, the container is exhausted — decrement Segments Left in the SRH and copy SRH[SL] to DA. If it is non-zero (our case: 0400), continue with the shifted DA.

5
Forward based on new DA

P1 performs an IPv6 FIB lookup on the new DA fcbb:bb01:0400:0300:0048::. The longest match is PE2's locator fcbb:bb01:0400::/48. P1 forwards toward PE2 via IGP shortest path (ECMP if available).

uA — Micro-SID Adjacency Behavior

Pin traffic to a specific link — the SRv6 uSID equivalent of Adjacency SID

What uA Does

uA is the micro-SID equivalent of End.X (Endpoint with cross-connect to a specific adjacency). While uN forwards via IGP shortest path (allowing ECMP), uA pins traffic to a specific outgoing interface.

When You Need uA Instead of uN

  • Traffic engineering: Force traffic through a specific link for path diversity
  • ECMP avoidance: Ensure deterministic path for delay-sensitive traffic
  • Link-specific steering: Use a dedicated optical wavelength or low-latency path
  • Failure isolation testing: Verify per-link behavior during maintenance
Important: uA bypasses IGP ECMP — use it only when you need explicit path control. For normal traffic, uN with IGP ECMP provides better load distribution and resilience.

uA Operation — Step by Step

uA processing is nearly identical to uN, with one critical difference at the forwarding step: uA sends the packet out a specific interface rather than performing an IGP FIB lookup.

1
Packet arrives — DA matches node's locator + uA function

The DA's block+function matches the node's locator, and the uA function value (e.g., 0001) is looked up in the Local SID Table. This maps to a specific adjacency.

2
Identify the target adjacency

The uA function encodes which interface/neighbor to use. For example, PE1's uA function 0001 might map to the PE1→P2 (south) link, while 0002 maps to PE1→P1 (north).

3
Shift DA left by 16 bits (same as uN)

The consumed uA function is shifted out, exactly like uN. The DA advances to the next packed uSID.

4
Forward out the specific interface (NOT IGP lookup)

Unlike uN which does an FIB lookup, uA sends the packet directly out the adjacency identified in step 2. This is the key difference — no routing table consultation.

End and End.X — Classic Endpoint Behaviors

The original SRv6 transit functions before uSID compression — RFC 8986 baseline behaviors

End — Basic Endpoint Function

The End function is the most fundamental SRv6 behavior. When a packet arrives with DA matching a node's End SID, the node acts as a transit waypoint — it advances the SRH pointer and forwards the packet toward the next segment.

End Processing — Step by Step

End behavior (RFC 8986 Section 4.1):
1. Verify Segments Left (SL) > 0 — if SL = 0, drop the packet
2. Decrement SL by 1
3. Copy SRH[SL] (the next segment) into the IPv6 Destination Address
4. Forward via longest-prefix-match on the new DA

The End function never removes the SRH. The full segment list stays intact throughout the path. Only the DA and SL change at each hop.

End.X — Endpoint with Cross-Connect

End.X is identical to End in steps 1-3, but in step 4 it forwards out a specific adjacency instead of doing a routing table lookup. This pins traffic to a particular link.

End.X Processing — Step by Step

End.X behavior (RFC 8986 Section 4.2):
1. Verify SL > 0
2. Decrement SL by 1
3. Copy SRH[SL] into IPv6 DA
4. Forward out the specific layer-3 adjacency bound to this End.X SID (bypass IGP lookup)

When to Use End.X

  • Strict path steering: Force traffic onto a specific link (e.g., for traffic engineering around a congested path)
  • TI-LFA backup paths: Repair segments use End.X to steer traffic away from the failed link
  • Topology probing: OAM packets targeting specific adjacencies for link-level testing

The SRv6 service endpoints that strip the outer IPv6 header and deliver customer traffic

Decap Behavior Overview

When the active SID is a decapsulation function (End.DT* or End.DX*), the node performs three actions: remove the SRH (if present), remove the outer IPv6 header, and then either look up the inner packet in a VRF (DT) or forward directly to a next-hop (DX).

End.DT4 — VRF Decapsulation (L3VPN)

End.DT4 is the SRv6 replacement for the MPLS per-VRF VPN label. When a packet arrives with DA matching the End.DT4 SID, the node strips the outer IPv6 encapsulation and performs an IPv4 route lookup in the bound VRF.

End.DT4 processing (RFC 8986 Section 4.6):
1. Verify DA matches a local End.DT4 SID
2. Remove the SRH (if present and SL = 0)
3. Remove the outer IPv6 header entirely
4. Extract the inner IPv4 packet
5. Perform an IPv4 FIB lookup in the VRF bound to this SID
6. Forward the inner IPv4 packet based on the VRF lookup result

End.DT6 and End.DT46

End.DT6 is identical but for IPv6 inner payloads — the VRF lookup is IPv6. End.DT46 handles both IPv4 and IPv6 inner packets in a single dual-stack VRF, inspecting the inner header's version field to determine the address family.

End.DX4/DX6 — Direct Cross-Connect

End.DX4 and End.DX6 skip the VRF lookup entirely. Instead, after decapsulation, the inner packet is forwarded directly to a pre-configured next-hop on a specific interface. This is a Layer 3 cross-connect — equivalent to an MPLS per-CE VPN label.

1. Remove SRH (if present, SL = 0)
2. Remove outer IPv6 header
3. Forward inner IPv4 packet directly to the bound {interface, next-hop} — NO VRF lookup

End.DX2 — L2VPN Pseudowire

End.DX2 handles Layer 2 payloads. After removing the outer IPv6+SRH, the inner Ethernet frame is forwarded to a specific attachment circuit (AC). This is the SRv6 replacement for MPLS pseudowire (PW) labels.

Key Design Principles

uDT/uDX Position Rule

uDT/uDX must always be the last function in the uSID chain. After decapsulation, there are no more segments to process — the inner packet is delivered to the service.

Combined Transport + Service

A single DA like fcbb:bb02:0100:0300:0048:: encodes two transit hops (PE3, PE4) and the VPN decap action (uDT4) — all in 128 bits. In SR-MPLS, this would need 3 separate labels.

End.AM and End.AD — Service Proxy Functions

Inserting SR-unaware service appliances into an SRv6 path — masquerading and dynamic proxy

The Problem: SR-Unaware Service Functions

Many network service appliances — firewalls, DPI engines, IDS/IPS, load balancers, NAT devices — cannot process SRv6 headers. When traffic must traverse these appliances as part of an SRv6 service chain, the SRv6 node adjacent to the appliance must hide or remove the SRH before sending traffic to the appliance, then restore it when the traffic returns.

RFC 8986 defines two proxy mechanisms for this purpose:

End.AM — Static Masquerading

Hides the SRH by modifying the IPv6 header in-place. The service appliance sees a clean IPv6 packet. On return, the SRH is restored.

End.AD — Dynamic Proxy

Fully decapsulates the outer IPv6+SRH and sends only the inner packet to the service. On return, re-encapsulates with the cached SRH state.

End.AM works at the IPv6 layer. It modifies the outer IPv6 header to hide the SRH from the service appliance, while caching the SRH locally. The service appliance sees and processes a normal IPv6 packet.

End.B6 — Binding SID Endpoint

Policy abstraction and cross-domain stitching — hiding SID lists behind a single identifier

What End.B6 Does

End.B6 binds a local SID to a pre-configured SRv6 policy (a SID list). When a packet arrives with DA matching the B6 SID, the node replaces or extends the segment routing path by inserting the policy's SID list. This is the SRv6 equivalent of the SR-MPLS Binding SID.

End.B6 is the foundation of cross-domain SRv6 — it provides domain isolation and policy abstraction. Domain 1 doesn't need to know any SIDs from Domain 2. It just steers traffic toward the border node's B6 SID, and the border node "stitches" the path into Domain 2 using its local policy. This is critical for multi-domain, multi-vendor, and multi-AS SRv6 networks.

End.B6.Encaps vs End.B6.Insert

Two variants exist, differing in how the new SID list is applied:

End.B6.Encaps

  • Pushes a new outer IPv6 header with a new SRH containing the policy's SID list
  • Original IPv6+SRH becomes the inner packet
  • Results in a double-encapsulated packet
  • Cleanest isolation — Domain 2 never sees Domain 1 headers
  • Most common in production

End.B6.Insert

  • Inserts a new SRH into the existing IPv6 header
  • No new outer header — extends the current segment list
  • More efficient (no double encap overhead)
  • But Domain 2 SIDs become visible in the existing SRH
  • Used when MTU is a concern

Use Cases for End.B6

1. Cross-Domain Stitching (Primary Use Case)

As shown above, End.B6 enables multi-domain SRv6 where each domain maintains its own SID block and internal topology. Border routers use B6 to stitch paths across domain boundaries without leaking internal SIDs.

2. Policy Abstraction

A complex multi-hop SRv6 policy (e.g., 6+ SIDs for a specific TE path) can be abstracted behind a single B6 SID. Ingress PEs only need to reference the B6 SID rather than the full explicit SID list. When the underlying path changes, only the B6 policy at the headend node needs to be updated.

3. Scalability

Without B6, cross-domain paths would require the ingress PE to know SIDs from every domain. With B6, the ingress PE only needs to know the border router's B6 SID and its own domain's SIDs. This reduces signaling complexity and state.

H.Insert — Headend SRH Insertion

Insert an SRH directly into an existing IPv6 packet — no outer encapsulation, no extra header

What H.Insert Does

Unlike H.Encaps, H.Insert does not create a new outer IPv6 header. Instead, it modifies the original IPv6 header in-place:

  • The original Destination Address is saved into a new SRH
  • The DA field is overwritten with the first SID in the segment list
  • The SRH is inserted between the IPv6 header and the upper-layer payload
  • At the final segment endpoint, the original DA is restored from the SRH
H.Insert modifies the original packet — this can break protocols that checksum the IPv6 header (e.g., IPsec AH mode). Use with care and only for native IPv6 TE steering.

Key Property

No outer encapsulation overhead

Only the SRH bytes are added. No extra 40-byte IPv6 header. Better MTU utilization for native IPv6 traffic.

Original packet is modified

The DA changes, NH field changes, and SRH is injected. Breaks integrity checks on the IPv6 header. RFC 8754 advises caution.

On-Demand Nexthop (ODN)

Automated SR-Policy creation triggered by BGP color communities — the dominant SRv6 TE deployment model

The Problem ODN Solves

Without ODN, an operator must pre-provision an SR-Policy for every {headend, color, endpoint} combination. In a network with 100 PEs and 5 SLA classes (colors), that's potentially 100 x 5 x 99 = 49,500 policies to configure. ODN eliminates this — you define a template per color and the network auto-creates policies on demand.

Without ODN (Static Policies)

Pre-provision SR-Policy for each destination PE, per color. Config grows linearly with PE count. Manual updates when topology changes.

With ODN (Template-Driven)

Define one ODN template per color. Policies auto-created when BGP routes arrive with matching color+endpoint. Zero per-destination config. Scales to thousands of PEs.

How ODN Works — Step by Step

Walkthrough using our topology: PE4 advertises a VPN prefix with color 100 (low-latency), PE1 auto-creates a policy:

1
PE4 attaches Color Extended Community to VPN route

PE4 (fcbb:bb02:0300) advertises VPN prefix 10.4.0.0/24 via BGP with Color = 100 (low-latency intent) and next-hop = PE4's loopback. The color community is attached via route-policy at the egress PE.

2
PE1 receives BGP route with color + endpoint

PE1 receives the VPN route. BGP next-hop resolution sees: endpoint = PE4, color = 100. PE1 looks for an existing SR-Policy matching {PE1, color 100, PE4}.

3
No matching policy exists — ODN template triggers

PE1 has an ODN template for color 100 that specifies: dynamic path computation, metric-type = min-delay, affinity include-any = north-core. This template is instantiated with endpoint = PE4.

The headend (or PCE) computes a path satisfying the template constraints. Result: SR-Policy {PE1, color 100, PE4} with segment-list fcbb:bb01:0200:0400:0070:: (P1 → PE2 → PE2's BSID for Domain 2). The policy is installed in the RIB/FIB.

ODN eliminates per-destination policy provisioning — one template covers all destinations with the same SLA. When PE5, PE6, PE7 come online and advertise routes with color 100, PE1 auto-creates policies for each of them using the same template. No operator intervention needed.

uSID Hardware Requirements

The uSID shift operation requires specific hardware support beyond standard IPv6 forwarding:

  • DA rewrite with shift: The ASIC must read the 128-bit DA, identify the active 16-bit uSID, and rewrite the DA with remaining uSIDs shifted left. This is a memory shift operation in the forwarding pipeline.
  • Block detection: The ASIC must recognize when the DA equals the block prefix (all uSIDs consumed) and trigger container advance or decap.
  • Multi-function processing: At a node like PE2 in the B6 example, the ASIC processes its uN (shift) and then immediately processes the next uSID (uB6) — multiple micro-operations in one pass.
uSID's major hardware advantage: locator-based FIB entries scale like IPv6 routes, not like per-SID TCAM entries. A network with 500 nodes and 10 VRFs per node needs only ~500 FIB entries for SRv6 reachability (one /48 per node). With classic SRv6, you might need 5000+ entries (one per unique SID). This is the same scaling model as IPv6 routing itself.

Key Takeaways

Seamless coexistence: SRv6 and SR-MPLS domains operate independently. PE1 (SRv6) encapsulates as usual — it does not know the destination is SR-MPLS. PE-MPLS processes standard MPLS labels — it does not know the traffic originated from SRv6. The gateway is invisible to both endpoints.
Migration enabler: This is the key mechanism for gradual SRv6 adoption. You can deploy SRv6 in new parts of the network while legacy SR-MPLS domains continue operating unchanged. The gateway function at border routers handles the translation. Over time, as SR-MPLS domains are upgraded to SRv6, the gateway is simply removed — no service disruption.
Bidirectional: The reverse path (PE-MPLS → PE1) works symmetrically. PE2 would have an MPLS-to-SRv6 gateway function that strips MPLS labels and pushes an SRv6 outer IPv6 header with PE1's uDT4 SID. The translation is stateless and symmetric.
Performance consideration: The gateway translation at PE2 involves header manipulation (strip 40B IPv6, push 8B MPLS). Modern routers perform this in hardware at line rate. The per-packet cost is comparable to a standard label swap operation in SR-MPLS.
OAM and Troubleshooting

Operational tools for monitoring, validating, and debugging SRv6 networks

SRv6 OAM leverages IPv6 infrastructure — standard tools like ping and traceroute work for basic reachability, while SRv6-specific extensions enable path validation along segment lists.

SRv6 OAM Challenges

While SRv6 inherits IPv6 OAM capabilities, segment routing introduces unique challenges that require purpose-built tools and careful operational workflows.

Traditional IPv6 OAM

Standard ping/traceroute follows the IGP shortest path. Cannot validate SR policy paths. ICMP responses may take a different return path, confusing results. No visibility into SRH processing correctness.

SRv6-Specific OAM

SRv6 ping targets specific SID endpoints to test SID table correctness. SRv6 traceroute walks along a segment list. Each responding node confirms SRH processing. End.OP SID enables dedicated OAM handling.

Why Standard Tools Fall Short

  • Path mismatch: ping fcbb:bb01:0400:: confirms PE2 is reachable via IGP, but does NOT validate that traffic steered through an SR policy takes the expected path
  • Silent drops: SRv6 SID misses result in silent drops by default — no ICMP unreachable is generated when a SID is not in the local SID table
  • Return path asymmetry: ICMP replies do not follow the SRv6 policy path in reverse, making round-trip measurements misleading for TE paths
  • Transit opacity: Transit nodes do not inspect SRH — they cannot report SRv6-specific errors
Critical: SRv6 SID misses are silent drops by default — always verify SID tables first when debugging connectivity failures.

Troubleshooting Workflow

When SRv6 traffic is not forwarding correctly, follow this systematic bottom-up approach. Each step validates a specific layer before moving to the next.

1
Verify SID Table — Are SIDs installed?

Check the local SID table on the target node. Every expected SID must appear with the correct behavior and VRF/interface binding.

show segment-routing srv6 sid
show segment-routing srv6 locator [name] detail

What to check: SID is present, behavior matches expectation (uN, uDT4, uA, etc.), associated VRF or interface is correct, SID is not stale or withdrawn.

2
Verify Locator Reachability — Is IGP advertising?

The locator prefix must be in the IGP (IS-IS or OSPF) and reachable from the ingress node. Without IGP reachability, SRv6 encapsulated packets cannot reach the endpoint.

ping fcbb:bb01:0400::   # Ping the locator prefix
show isis route ipv6 fcbb:bb01:0400::/48
show route ipv6 fcbb:bb01:0400::/48

What to check: Locator prefix is in IS-IS LSDB, route is in IPv6 RIB/FIB, next-hop and outgoing interface are correct.

3
Verify SRH Processing — Do transit nodes forward correctly?

Use SRv6 traceroute with an explicit segment list to confirm each node along the path processes the SRH correctly and forwards to the next SID.

traceroute srv6 segment-list fcbb:bb01:0200:: fcbb:bb01:0400:: fcbb:bb02:0300::
# Or platform-specific equivalent

What to check: Each expected hop appears in the trace output, no unexpected "* * *" gaps, hop count matches expected topology.

4
Verify VPN Service — Does end-to-end service work?

Finally, test the end-to-end service by pinging through the VRF. This validates the full chain: headend encapsulation, transit forwarding, endpoint decapsulation, and VRF routing.

ping vrf CUSTOMER-A 10.0.0.1 source 10.0.0.2
show bgp vpnv4 unicast vrf CUSTOMER-A
show bgp l2vpn evpn

What to check: VRF routes are present with SRv6 SID as next-hop, BGP session is up, color/community attributes are correct for TE steering.

OAM Best Practices

Proactive SID Monitoring
Periodically poll show segment-routing srv6 sid on all PEs. Alert on SID count changes — a disappearing SID often precedes a service outage.
Locator Prefix BFD
Use BFD over SRv6 to detect fast path failures. BFD packets can be steered through the same SR policy as service traffic for accurate liveness detection.
MTU Budgeting
Plan MTU with SRv6 overhead: 40B (outer IPv6) + 8B (SRH base) + 16B per SID in SRH. With uSID and no SRH, overhead is just 40B. Ensure path MTU accommodates worst-case encapsulation.
End.OP Allocation
Allocate an End.OP SID on every PE and P node. This provides a reliable OAM target even when service SIDs may not generate ICMP responses.
Drop Counter Baselines
Baseline SRv6-specific drop counters (SID miss, SL exceeded, MTU drops). Alert on deviations. Silent SRv6 drops are the most common troubleshooting challenge.

Migration Considerations

Brownfield MPLS Network
SR-MPLS is the natural first step — same data plane, just replace LDP with SR. SRv6 migration can follow incrementally using interworking gateways (End.B6 binding SIDs or SRv6-to-MPLS stitching at domain boundaries).
Greenfield IPv6 Network
SRv6 with uSID is the natural choice — no MPLS dependency, globally unique SIDs, richer programming model. Avoids deploying MPLS infrastructure that will eventually be replaced.
Hybrid/Multi-Domain
Common pattern: SR-MPLS in legacy WAN domains, SRv6 in new metro/DC domains. Interworking at domain boundaries using BGP with SRv6 SID transposition. Both share the same SR policy framework (RFC 9256).
Hardware Refresh Cycle
Align SRv6 migration with natural hardware refresh. Newer merchant silicon (Memory, memory, memory-V2, memory, memory-2) has native SRv6/uSID support. Avoid forklift upgrades — migrate domain by domain as hardware ages out.

Existing MPLS network, no immediate IPv6 plan SR-MPLS Lowest disruption. Replace LDP with SR-MPLS. Migrate to SRv6 later.
New build, IPv6-ready hardware SRv6 uSID Future-proof. No MPLS overhead. Global SID uniqueness from day one.
Multi-domain with different hardware generations Hybrid SR-MPLS in legacy domains, SRv6 in new. Interwork at boundaries.
Strong SFC requirements SRv6 Native proxy behaviors (End.AS/AD/AM). No NSH dependency.
Ultra-high-scale TCAM-constrained core SR-MPLS Compact 20-bit labels use less TCAM than IPv6 /48 routes.
Network slicing / 5G transport SRv6 uSID Richer SID semantics for slice identification. Aligned with 3GPP direction.
Key takeaway: The SR policy framework (RFC 9256) is the same for both SR-MPLS and SRv6. Control-plane investment (BGP, IS-IS SR extensions, PCE, SR policy) is preserved regardless of data-plane choice. Migration between the two is a data-plane evolution, not a control-plane rewrite.
RFC and Standards Map

Complete reference of SRv6-related RFCs, drafts, and standards

Standards status as of 2025: The core SRv6 architecture is fully standardized (RFC 8402, 8754, 8986). Key service and policy RFCs are published. Compressed SRv6 (uSID, G-SID) and some advanced features remain in draft stage with active implementation.

Core SRv6 Architecture

The foundational RFCs that define Segment Routing and the SRv6 data plane.

RFC / DraftTitleCategoryStatusKey Relevance
RFC 8402 Segment Routing Architecture Core Published Defines the SR architecture for both MPLS and IPv6 data planes. The foundational document for all segment routing work.
RFC 8754 IPv6 Segment Routing Header (SRH) Core Published Defines the SRH extension header format: segment list encoding, Segments Left processing, SRH TLVs.
RFC 8986 SRv6 Network Programming Core Published The behavior catalog: defines End, End.X, End.DT4, H.Encaps, proxy behaviors, and the full SRv6 instruction set.
RFC 9256 Segment Routing Policy Architecture Core Published SR Policy framework: candidate paths, segment lists, binding SIDs, weighted ECMP, preference. Applies to both SR-MPLS and SRv6.
RFC 9352 IS-IS Extensions for SRv6 Core Published IS-IS TLVs for advertising SRv6 locators, SIDs, and capabilities. Essential for IGP-based SRv6 deployments.

Compressed SRv6 and OAM

Standards for SRv6 compression schemes (uSID, G-SID) and operational management.

RFC / DraftTitleCategoryStatusKey Relevance
draft-filsfils-spring-srv6-net-pgm-extension SRv6 Network Programming: uSID Compressed Draft Defines micro-SID (uSID): 16-bit compressed SIDs packed into 128-bit containers. Shift-and-forward mechanics. The dominant compression approach in deployment.
draft-cl-spring-generalized-srv6-for-cmpr Generalized SRv6 Network Programming for Compression (G-SID) Compressed Draft Alternative compression using Generalized SIDs (G-SID). Flexible SID lengths. Proposed by China-origin operators and vendors.
draft-ietf-spring-srv6-srh-compression Compressed SRv6 Segment List Encoding Compressed Draft IETF unified framework for SRv6 compression, attempting to harmonize uSID and G-SID approaches.
RFC 9259 Operations, Administration, and Maintenance (OAM) in SRv6 OAM Published SRv6 OAM mechanisms: End.OP behavior, O-flag in SRH, SRv6 ping/traceroute procedures, failure detection.
Compression landscape: uSID is the most widely deployed compression scheme, with production implementations from major vendors. G-SID offers an alternative approach. The IETF SPRING WG is working on a unified compression framework. Both are backward-compatible with base SRv6.
